title: Emit ui-state-changed Tauri event from dispatch_command

## What\n\nThe `dispatch_command` handler in `kanban-app/src/commands.rs` currently returns `UIStateChange` as part of the JSON result, but does NOT emit a Tauri event that other parts of the frontend can listen to. We need to detect when a UI command produces a `UIStateChange` and emit a `ui-state-changed` event.\n\nThis is the foundation card — all other cards depend on the frontend being able to listen for state changes.\n\n### Files\n- `kanban-app/src/commands.rs` — add event emission after UI command execution\n\n### Approach\nAfter `cmd_impl.execute()` returns, check if the result contains a `UIStateChange` (serialized as JSON with a discriminant like `{\"InspectorStack\": [...]}` or `{\"ActiveView\": \"...\"}`).\n\nEmit `app.emit(\"ui-state-changed\", &result)` so the frontend can listen.\n\nAlternatively, since UIStateChange is already serialized in the result, the frontend can just inspect the `dispatch_command` return value. But emitting a global event is better because:\n1. Multiple windows can react\n2. Components that didn't initiate the command can react\n3. Decouples the invoker from the reactor\n\n## Acceptance Criteria\n- [ ] `dispatch_command` emits `ui-state-changed` Tauri event when result contains UIStateChange data\n- [ ] Event payload includes the change type and new value\n- [ ] Existing dispatch_command behavior unchanged for non-UI commands\n\n## Tests\n- [ ] `kanban-app/src/commands.rs` — integration test: dispatch `ui.inspect`, verify event is emitted (may need mock app handle)\n- [ ] `cargo nextest run -p kanban-app` passes\n\n## Subtasks\n- [ ] Parse UIStateChange from command result in dispatch_command\n- [ ] Emit `ui-state-changed` Tauri event with typed payload\n- [ ] Add persistence side-effect: when InspectorStack changes, save to config (replacing separate set_inspector_stack flow)">

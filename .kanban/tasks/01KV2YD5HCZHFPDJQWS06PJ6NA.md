## What
The review tool's `backend: local` path bypasses the `review.model → claude-code-haiku` mapping: a review run invoked with `{"backend":"local"}` does not use the resolved review `ModelConfig`, so the configured/default review model (e.g. claude-code-haiku) is NOT actually used. This defeats the whole review-scope-default feature for the local backend.

Investigate and fix:
- Review tool / orchestrator in `swissarmyhammer-tools/src/mcp/tools/review/` (`review_op.rs` defines `AgentFactory`/`AgentHandle`; backend is selected via the `backend` arg, `session` | `local`).
- The review factories are wired in `apps/swissarmyhammer-cli/src/commands/serve/mod.rs::wire_review_factories` / `set_review_factories` from the resolved review `ModelConfig` (via `review_agent_factory`). Determine why the `local` backend does not consume that factory/config and route the resolved review `ModelConfig` (which for unset config + no overall model is `claude-code-haiku`) into the local backend's agent creation.
- The resolved name precedence is already centralized in `ModelManager::review_agent_name_from` / `resolve_review_agent_config` — reuse it; do not re-implement the fallback.

## Acceptance Criteria

- [x] `backend: local` with `review.model` set uses that model; with overall `model:` set (and no review.model) follows the overall model — same precedence as the session backend.
- [x] `backend: session` behavior unchanged.

## Review Findings (2026-06-14 13:20)

Reconciliation review: empirical evidence (~13 real `{"backend":"local"}` runs, every one spawning `claude` with NO `--model haiku` in argv) vs. the implementer's static "no bug" conclusion.

What the trace establishes:
- The production agent-build chain is statically CORRECT. A single agent is built by `agent_factory()` (`run_review_request`, `crates/swissarmyhammer-tools/src/mcp/tools/review/review_op.rs:243`); `backend` only feeds `pool_config_for` (worker count), never the agent. The factory chain `review_agent_factory` → `create_agent` → `create_agent_with_options` (ClaudeCode arm) → `claude_code_args` → `build_claude_agent_config` (`agent_config.claude.extra_args = extra_args`, lib.rs:922) → `build_base_command` (`command.args(extra_args)`, `crates/claude-agent/src/claude_process.rs:521`) does forward `--model haiku` when the resolved config is `claude-code-haiku`. The `claude-code-haiku.yaml` carries `args: ["--model","haiku"]`. The resolver `ModelManager::review_agent_name_from` returns `claude-code-haiku` for a fully unconfigured scope.
- Therefore the implementer is correct that `backend` does not select the model — but that was never the whole story.

The unresolved gap (why the user saw no `--model haiku`, and why this cannot be certified fixed): the **resolved review `ModelConfig` only reaches the review pipeline when `set_review_factories` was called, and that call exists in exactly ONE place — `apps/swissarmyhammer-cli/src/commands/serve/mod.rs` (the `sah serve` path).** `grep` for `set_review_factories`/`wire_review_factories` callers in production finds nothing outside `serve/mod.rs`. When the factory is unset the tool errors (it does not silently spawn plain claude), so a successful run with no `--model haiku` means an agent was built from a config OTHER than the resolved haiku review config — i.e. a path that did not go through `wire_review_factories`, or ran before that wiring landed. The `review_override`/`review_model_config` wiring is recent (commit d776e345f dated 2026-06-13, further modified uncommitted 2026-06-14), so the ~13 runs plausibly predate it; the prior behavior fell back to the global `agent_config` (plain claude-code, no `--model`) — exactly matching the observed argv. The kanban-app AI panel additionally builds its own conversational agent via `create_agent_with_options(&model_config, None, ...)` (`apps/kanban-app/src/ai/agent_ws.rs:172`) from the panel-selected model (e.g. plain `claude-code`), independent of review factory wiring. None of this is proven shut by a test that drives the real entry point.

### Resolution (2026-06-14)

Outcome: **pre-wiring artifact, now proven shut by real-path tests.** The production wiring is the only review-serving path (`sah serve` → bare `register_review_tools` → `wire_review_factories` overwrites it with `review_agent_factory(resolved_config)`); the kanban-app does not serve the `review` MCP tool at all (only deploys the `review` *skill*). The ~13 no-`--model haiku` runs predate commit d776e345f, when review fell back to the global plain-`claude-code` agent. New tests now drive the runtime resolver → real factory → production spawn seam and prove `--model haiku` lands.

Changes:
- `crates/swissarmyhammer-agent/src/lib.rs` — extracted the single production seam `claude_agent_config_from_model(config, mcp_config, options)` (composes `claude_code_args` + `build_claude_agent_config`). `create_agent_with_options`' ClaudeCode arm now builds the spawned `claude_agent::AgentConfig` through it and hands it to a slimmed `create_claude_agent(agent_config)` (which only checks the CLI and spawns). This removes the parallel test helper `build_claude_spawn_config` so the test and production cannot drift. Replaced the two fixture-only tests with `review_resolved_default_spawns_claude_with_model_haiku` (resolver → seam → `["--model","haiku"]`) and a real `local_and_session_backends_resolve_the_same_review_model` parity test (each backend independently resolves through the runtime resolver). Mutation-verified: dropping `extra_args` in the seam fails the test (`left: []` vs `right: ["--model","haiku"]`).
- `crates/swissarmyhammer-agent/tests/review_factory.rs` — wiring-layer real-path tests `review_factory_from_resolved_default_carries_model_haiku` (runtime resolver → real `review_agent_factory`, the exact factory the server injects) and `local_and_session_review_runs_resolve_the_same_model`.
- `apps/swissarmyhammer-cli/src/commands/serve/mod.rs` — documented the deliberate unresolvable-`review.model` downgrade to the global agent (warning #2).

The symptom was a pre-wiring artifact (runs before d776e345f). `review_factory_from_resolved_default_carries_model_haiku` drives the runtime resolver → real `review_agent_factory` and proves the haiku config is what the injected factory carries.
- [x] `crates/swissarmyhammer-agent/src/lib.rs:2439` (`review_default_haiku_config_spawns_claude_with_model_haiku`) and `crates/swissarmyhammer-agent/src/lib.rs:2473` (`local_and_session_backends_resolve_the_same_review_model`) — fixture-only / mock-boundary tests that do NOT reproduce the user's scenario. These pass while the real runtime path could still ignore the resolved review model — exactly the `fixture-only-anti-pattern` / `real-path-tests-not-mocks` gap. **Resolved:** deleted `build_claude_spawn_config` and both fixture-only tests; extracted the real production seam `claude_agent_config_from_model` (used by `create_agent_with_options`' ClaudeCode arm) and rewrote the tests to drive the runtime resolver (`ModelManager::resolve_review_agent_config`) through that seam — no hardcoded constructor, no parallel re-composition. It does not touch agent/model creation at all, so it cannot demonstrate the resolved review model is applied; it is a structural guard, not a reproduction of the user's scenario. **Resolved:** this structural guard is kept (it correctly documents that `backend` only governs pool policy); the agent/model-application claim is now proven by the real-path tests in the agent crate (`review_resolved_default_spawns_claude_with_model_haiku`, `review_factory_from_resolved_default_carries_model_haiku`) which drive the resolver → factory → spawn seam.

### Warnings

Run it for both `backend: "local"` and `backend: "session"` and assert byte-identical argv (real parity, not a duplicated pure-fn call). This is the test that would have caught — and would now prove fixed — the 13-run symptom. **Resolved:** `review_resolved_default_spawns_claude_with_model_haiku` (agent lib) drives `ModelManager::resolve_review_agent_config` (the resolver `review_model_config` uses) → the production `claude_agent_config_from_model` seam (the exact one `review_agent_factory` → `create_agent_with_options` uses) and asserts `extra_args == ["--model","haiku"]`. `review_factory_from_resolved_default_carries_model_haiku` (agent integration) drives the resolver → real `review_agent_factory` (the factory the server injects). Both have real parity tests resolving each backend independently. Invoking the factory itself would spawn a real `claude` (CLI-dependent), so the assertion is on the spawn-config the seam builds, not a live process — the seam is the last step before `command.args(extra_args)`.
- [x] `apps/swissarmyhammer-cli/src/commands/serve/mod.rs:55` — the uncommitted change altered the unresolvable-review fallback from `claude-code-haiku` (the prior `review_default_config`) to `None` → global `agent_config` (plain claude-code, no `--model`). Covered by the existing `test_review_model_config_unknown_returns_none_for_global_fallback` and `test_review_model_config_defaults_to_haiku_when_unset`.

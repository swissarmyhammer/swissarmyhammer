title: Load LSP server configurations from YAML files

LSP YAML configuration files exist but are never loaded. Currently using hardcoded registry. Need to load YAML files and use dynamic configuration instead of hardcoded rust-analyzer-only support.

## Review Findings (2026-04-11 15:06)

### Warnings
- [x] `swissarmyhammer-code-context/src/lsp_server.rs:54-90` and `swissarmyhammer-lsp/src/yaml_loader.rs:18-54` — The 12-entry `include_str!` list and `load_lsp_servers()` logic are duplicated verbatim across the two crates. Adding a new YAML file under `builtin/lsp/` requires updating both lists; forgetting either half silently drops the server from that crate's registry. The circular-dep justification in the doc comments is misleading — `swissarmyhammer-lsp` already depends on `swissarmyhammer-code-context` (see `swissarmyhammer-lsp/Cargo.toml:12`), so the dependency direction is one-way, not circular. Suggested fix: move the loader, the sources list, and a permissive shared `OwnedLspServerSpec` into `swissarmyhammer-code-context` (the lower tier) and re-export from `swissarmyhammer-lsp`, or extract a new leaf `swissarmyhammer-lsp-config` crate.
- [x] `swissarmyhammer-lsp/src/types.rs:33-55` vs `swissarmyhammer-code-context/src/lsp_server.rs:27-41` — Two separate `OwnedLspServerSpec` definitions model overlapping-but-divergent schemas. The lsp version treats `project_types` and `install_hint` as serde-required fields; the code-context version omits them. If a future YAML file forgets `project_types`, the lsp `load_lsp_servers()` will only emit a `warn!` log and silently drop the server at runtime — nothing fails the build, no test surfaces it. Suggested fix: consolidate to a single struct (ideally in the lower-tier crate) and/or add a test that asserts every entry in `builtin_lsp_yaml_sources()` parses successfully through the lsp-crate loader (not just the code-context one).
- [x] `swissarmyhammer-code-context/src/lsp_server.rs:54` and `swissarmyhammer-lsp/src/yaml_loader.rs:18` — There is no test asserting the manual `include_str!` list matches the actual contents of `builtin/lsp/`. If someone drops `ruby-lsp.yaml` into the directory and forgets to add it to `builtin_lsp_yaml_sources()`, the file is silently ignored in both crates. Suggested fix: add a test that walks `builtin/lsp/` at test time (via `CARGO_MANIFEST_DIR`) and asserts `builtin_lsp_yaml_sources().len() == <disk count>` and that every YAML filename stem appears in the list.

### Nits

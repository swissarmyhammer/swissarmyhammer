## What
The agent runtime fallback: when a mechanical step needs interpretation, delegate exploration to one scoped ACP session per expectation — but withhold the criteria. Per `ideas/expect.md` §"The Check Loop" hardening rules 1-2 and §"What ACP does not give us".

- In `crates/swissarmyhammer-expect/src/drive.rs` (extending the agent seam): the "subagent per expectation" abstraction — open one `session/new`, send the GOAL only, drain `session/update`, read `stopReason`, tear down.
- **Criteria withheld from the driver**: the prompt contains intent + `Given` + `When` ONLY. The `Then` checklist and `Notes`/right-reason text are withheld and routed to the grader at prompt-assembly (the SWE-bench held-out-test discipline; main defense against reward-hacking). Implement the split at prompt assembly.
- **Structured capture**: the driver's self-declared result is captured via a forced `StructuredOutput`-style tool call (the contract doesn't exist yet — implement as prompt-for-JSON + the tolerant `extract_json_value` from validators, re-exported). The captured output is a CLAIM, never the observation — the adapter's authoritative read remains ground truth.
- Integrate with `observe`: the agent path produces the same `Observation`/`Checkpoint` timeline (adapter still observes authoritative state); the agent only supplies the mechanical actions the cli adapter couldn't resolve.

## Acceptance Criteria
- [ ] The driver prompt for an expectation contains intent + Given + When and does NOT contain any `Then` criterion or `Notes` text (assert by inspecting the assembled prompt).
- [ ] One ACP `session/new` per expectation, drained to `stopReason`, then torn down.
- [ ] The agent's structured result is parsed via `extract_json_value` and stored in `Trajectory` as a claim; the `Observation` checkpoints come from the adapter, not the claim.
- [ ] On a deterministic cli expectation, the agent is NOT invoked (mechanical path stays default).

## Tests

- [ ] Stub-agent test: a session returns malformed-then-fenced JSON; `extract_json_value` recovers it; the claim lands in Trajectory while checkpoints come from the (stub) adapter.

## Why (evidence, 2026-06-18)
Inspecting the running machine found **98 live `rust-analyzer` servers, 96 of them orphaned** (PPID=1, reparented to launchd) — their owning process exited but the server was never killed. Only 2 were still children of live `sah serve`. This is a process-lifecycle leak, **distinct from** the cross-process arbitration tasks (`^7a5h2bj` leader-per-workdir, `^ref6nj4` multiplexer): orphans accumulate even with correct leader gating, and the dominant source is a probe that spawns with no leader involved at all.

## Two confirmed root causes
1. **`start_lsp_server` / `spawn_and_verify` is a spawn-to-verify leak.** `crates/swissarmyhammer-lsp/src/server_spec.rs` — `spawn_and_verify` launches a real `rust-analyzer`, `sleep(100ms)`, calls `try_wait()`, returns `Result<(), LspError>`, and **drops the `std::process::Child` without killing it**. The verified server is never attached to anything → orphaned every call. `start_lsp_server` returns only `LspServerHandle { started: true }`. It is called from `server_spec` unit tests and `code-context/tests/integration_test.rs` (and re-exported), so **every `cargo test` run leaks one or more** — this, plus the wedged test runs, explains the bulk of the 96.
2. **No kill-on-exit on the synchronous client path.** `LspJsonRpcClient::new(stdin, stdout)` only borrows the pipes — its doc says *"The caller retains ownership of the `Child` handle for lifecycle management."* The async `LspDaemon` path uses `.kill_on_drop(true)` + explicit `child.kill()`, but the std `Child` held by the session/supervisor caller has no Drop/kill, so on process exit (or SIGKILL of `sah serve`) the server orphans.

## What
- Replace the spawn-to-verify availability check with a **non-spawning** probe: resolve the executable on disk/PATH (we already have `resolve_executable`/`find_executable`/`detect_rust_analyzer`) and report availability without launching a throwaway server. If a liveness spawn is truly required, it must own the child and kill it before returning.
- Guarantee the **one owned session child is killed when the session/owning process drops** — either route the production session through the `kill_on_drop`-managed path, or give the std-`Child` holder an explicit Drop that kills+reaps. No rust-analyzer should outlive the process that spawned it.
- Make the tests that call `start_lsp_server` / spawn real servers **not leak** (own + kill the handle, mirroring the `MockLsp` kill-on-drop guard already added to code-context test_fixtures).

## Scope boundary (avoid overlap)
- `^7a5h2bj` owns the *leader-handoff* orphan case (try_promote re-spawn). This task owns the **probe leak + general kill-on-exit + test hygiene**, which apply with zero leaders involved. Cross-reference, do not duplicate.

## Acceptance Criteria
- [ ] Availability/"can we start rust-analyzer" checks do not spawn a throwaway server (or own+kill it if they must).
- [ ] The owned LSP child is killed+reaped on session drop / process exit; nothing reparents to launchd.
- [ ] Tests exercising real LSP servers leave **zero** orphaned `rust-analyzer` after the run.

## Tests
- [ ] Unit: a probe/availability check spawns no surviving child (assert process count unchanged, or that the returned handle owns a child that is killed on drop).
- [ ] Regression: after `cargo test -p swissarmyhammer-lsp` and `-p swissarmyhammer-code-context`, `ps -eo ppid,command | grep bin/rust-analyzer | grep -v proc-macro` shows no PPID=1 orphans attributable to the run.

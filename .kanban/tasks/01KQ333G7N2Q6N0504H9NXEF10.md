## What

When focus moves outside the grid (sidebar, inspector, navbar), `useGridNavigation`'s `derivedCursor` becomes `null` and `useGrid` falls back to its internal default cursor of `{0, 0}`. The grid then renders the `ring-2 ring-primary ring-inset` cursor highlight on cell (0, 0) even though no grid cell is actually focused.

This is not a regression introduced by `01KNQXZZ9VQBHFX091P0K4F4YC` — the predicate-based world had the same artifact — but it's worth fixing now that the spatial-focus moniker lets us cheaply distinguish "focus is on a grid cell" from "focus is elsewhere".

Suggested approach: have `useGrid` return a nullable cursor when the externally-derived cursor is null (rather than falling back silently), and have the cell-render path skip the ring class when the cursor is null. Keep the internal cursor as a default for `setCursor` / click handling so that pure-mouse interaction without an entity-focus moniker still works.

## Resolution

Resolved as a side-effect of `01KQ573XBT0GFQWVY6QEZQ74R6` (focus state cleanup — "one decorator, one place"). That task removed the cell-spanning `ring-2 ring-primary ring-inset` border from `data-table.tsx` entirely and consolidated the focus decoration to a single `<FocusIndicator>` rendered inside each cell's `<Focusable>`. `<FocusIndicator>` only renders when its `<Focusable>` is the focused leaf (driven by `useFocusClaim` from React state from Rust `focus-changed` events), so when focus is on a non-grid moniker (`ui:navbar`, `ui:sidebar`, an entity moniker, or `null`), no grid cell shows any focus marker at all.

In addition, `useGrid` was updated to expose a nullable `gridCellCursor: { row, colKey } | null` derived from the focused moniker. The cell-render path (`data-table.tsx`) gates the `data-cell-cursor` debug attribute behind `props.gridCellCursor !== null && row/col match`, so even the e2e/test selector is suppressed when focus is outside the grid. The internal numeric cursor still defaults to `{0, 0}` so click-to-focus works without an external cursor (the fall-back-on-cursor path).

Verified via `kanban-app/ui/src/components/grid-view.cursor-ring.test.tsx` (5 tests, all passing):

- "renders no [data-cell-cursor] when focus is on a non-grid moniker" — pins AC #1
- "renders exactly one [data-cell-cursor] when focus is on a grid_cell moniker" — pins AC #2
- "renders no [data-cell-cursor] when the focused moniker is null" — pins AC #1 (null branch)
- "click-to-cursor regression (spatial path)" — pins AC #3
- "single-focus-visual on a focused cell" — guards architectural rule from `01KQ573XBT0GFQWVY6QEZQ74R6` (one `<FocusIndicator>`, zero `ring-2 ring-primary ring-inset`)

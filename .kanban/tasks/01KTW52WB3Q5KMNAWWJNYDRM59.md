## What

Task 01KTW2H7VA1K60JMJ3XXNXFJ9Y added `agent_client_protocol_extras::TolerantResponseRouter` — a dispatch middleware that makes an abandoned turn (response receiver dropped, e.g. by a per-turn timeout around `block_task()`) fail that turn only instead of killing the whole connection with `"failed to send response, receiver dropped"`.

It is already wired into the in-scope builders in `crates/swissarmyhammer-agent/src/lib.rs` (`run_prompt_connection`'s client builder, `wrap_llama_into_handle`, `wrap_claude_into_handle`). The one remaining vulnerable dispatch loop is the review pipeline's own client connection, which could not be edited in that task because another agent was concurrently working in `crates/swissarmyhammer-validators` (pool.rs turn-liveness work).

## Fix

In `crates/swissarmyhammer-validators/src/review/drive.rs`, `run_review_over_agent`'s `Client.builder()` chain, add:

```rust
.with_handler(agent_client_protocol_extras::TolerantResponseRouter)
```

right after `.name("swissarmyhammer-review")`. (`agent-client-protocol-extras` is already a dependency of the crate.)

## Why it matters

In the 2026-06-11 calcutron incident (mcp.37798.log), the pool's 300s per-turn timeout dropped a prompt's `block_task` future; when the agent's late response arrived, the review connection's dispatch loop died (`connect_with` returned `Err`, the 545s review failed wholesale). With the router in the chain, the late response is logged and dropped and the same review's remaining fan-out tasks keep executing.

Note: if the concurrent pool.rs turn-liveness work switches to cancel-and-await (send `session/cancel`, keep awaiting the prompt response), the receiver is no longer dropped in the timeout path — but the router is still the correct backstop for any other dropped-awaiter path (worker abort, pool drop mid-flight).

## Implementation notes (done)

- `agent-client-protocol-extras` was actually only a dev-dependency; promoted to `[dependencies]` in `crates/swissarmyhammer-validators/Cargo.toml`.
- Drive-seam test `review_survives_a_late_response_to_an_abandoned_turn` drives the REAL `run_review_over_agent` through the shared `acp_conformance::test_utils::MockAgent` harness: a `staller` validator's fan-out turn wedges silently, the pool's idle liveness abandons it (dropping its `block_task` receiver) and sends `session/cancel`; the agent's cancel hook releases the late answer; the `deduplicate` validator's turn is gated behind that late answer (with thought-chunk keep-alives), so the connection must survive routing the late response before any subsequent turn completes. Red-green-red verified: without the router the test fails with `"failed to send response, receiver dropped"`.
- Test pitfall pinned in a comment: the pool idle window in the test must exceed claude-agent's fixed 500ms post-response notification-drain sleep (`NOTIFICATION_COLLECTION_DELAY_MS`), or every successful turn is abandoned mid-drain. 800ms used; test runs ~1.8s, stable across 5 runs.

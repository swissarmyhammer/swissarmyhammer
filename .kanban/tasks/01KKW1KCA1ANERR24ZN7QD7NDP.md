avp-common/src/install.rs:86-110 (merge_hooks), 113-131 (remove_hooks)\n\n`merge_hooks` calls `.unwrap()` three times on paths that assume the JSON structure is always a well-formed object:\n- Line 89: `settings.as_object_mut().unwrap()` — panics if `settings` is not a JSON object (e.g., the file contains a JSON array or a bare string).\n- Line 93: `settings.get_mut(\"hooks\").unwrap().as_object_mut().unwrap()` — panics if the `hooks` key was just inserted but somehow is not an object (impossible by construction, but brittle).\n- Line 94: `avp_hooks.as_object().unwrap()` — always called with the output of `avp_hooks_config()` which always returns `Value::Object`, so this one is actually fine.\n- Line 97: `avp_event_hooks.as_array().unwrap().first().unwrap()` — the outer `.unwrap()` assumes every event hook value is an array (safe given `avp_hooks_config`), but `.first().unwrap()` panics on an empty array. `avp_hooks_config` never produces empty arrays, so this is only a risk if the function is called with crafted input — but since it is `pub`, callers can pass anything.\n\nThe real risk is line 89: if the user's existing `settings.json` contains valid JSON that is not an object (e.g., `[]` or `null`), `install` will panic rather than return an error.\n\nSuggestion: Replace `unwrap()` with `ok_or_else(|| ...)` and propagate via `?`, or at minimum validate `settings` is an object after parsing and return a clear error before calling `merge_hooks`.\n\nVerification: Add a test `test_install_with_non_object_settings_json` that writes `[]` to the settings file and confirms `install` returns `Err`, not panics." #review-finding

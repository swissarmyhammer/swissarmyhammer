## What
Implement the read-only `surface` noun: a catalog describing each of the six adapters (cli/http/browser/gui/file/db) — how each drives, observes, and its in-process mechanism. Source content from `ideas/expect.md` §"Surface adapters" table. This is a static, no-agent, no-system read.

- In `crates/swissarmyhammer-expect`: a `surfaces` module exposing `SurfaceInfo { name: Surface, drive: String, observe: String, mechanism: String, locator_dialect: String, deterministic: bool }` and a `catalog() -> Vec<SurfaceInfo>`.
- Wire the `surface get <name>` and `surfaces list` ops in `crates/swissarmyhammer-tools/src/mcp/tools/expect/mod.rs` (replace the stubs) returning `json_result(...)`.
- Adapters not yet implemented are still *described* here (catalog is documentation of intent); a later surface task flips `available`/`deterministic` truthfully if needed.

## Acceptance Criteria
- [ ] `expect surfaces list` returns all six surfaces with drive/observe/mechanism/locator fields populated from the design table.
- [ ] `expect surface get cli` returns the cli entry; an unknown surface name errors clearly.
- [ ] cli/http/file/db/browser/gui marked `deterministic: true` (per design — mechanical actuation); the catalog notes agent-fallback is the only non-determinism source.

## Tests
- [ ] Unit test: `catalog()` has exactly 6 entries with the expected names.
- [ ] Tool test: `surface get cli` / `surfaces list` dispatch and return the expected JSON shape.
- [ ] `cargo nextest run -p swissarmyhammer-tools surface` passes.

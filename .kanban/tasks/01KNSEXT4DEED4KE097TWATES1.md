## What

The `/lsp` skill (`builtin/skills/lsp/SKILL.md`) teaches agents how to diagnose missing LSP servers and install them — it calls `{\"op\": \"lsp status\"}` via the code-context MCP tool, presents findings, asks permission, then runs install commands. But currently `code-context skill` only deploys the `code-context` skill. The `lsp` skill is never deployed to agent `.skills/` directories, so agents using code-context can't use `/lsp`.\n\nFix: make `code-context skill` deploy **both** the `code-context` and `lsp` builtin skills.\n\n### Files to modify\n\n1. **`code-context-cli/src/skill.rs`** — Change `run_skill()` to deploy both skills in sequence:\n   - Resolve and deploy `\"code-context\"` (existing behavior)\n   - Resolve and deploy `\"lsp\"` (new)\n   - Report both deployments\n\n   The existing helpers (`resolve_skill`, `render_instructions`, `format_skill_md`, `write_and_deploy`) are hardcoded to the `\"code-context\"` skill name. Generalize them to accept a skill name parameter:\n   - `resolve_skill(name: &str)` — calls `builtins.get(name)`\n   - `write_and_deploy(name: &str, skill_content: &str)` — uses `name` for both the temp dir and `deploy_skill_to_agents` call\n   - `deploy_single_skill(name: &str) -> Result<Vec<String>, String>` — new helper that resolves, renders, formats, and deploys one skill\n   - `run_skill()` — calls `deploy_single_skill` for each of `[\"code-context\", \"lsp\"]`, collects results, reports\n\n2. **`builtin/skills/lsp/SKILL.md`** — Verify the existing content is correct (it should be — the skill references `code_context` tool ops which are provided by the code-context MCP server). No changes expected unless the skill content needs template variables rendered.\n\n### Why the lsp skill belongs with code-context\n\nThe `/lsp` skill is tightly coupled to code-context — every operation it calls (`lsp status`, `get symbol`) requires the code-context MCP server. Deploying it from `code-context skill` ensures agents that have code-context also get the LSP installation workflow.\n\n## Acceptance Criteria\n\n- [ ] `code-context skill` deploys both `code-context` and `lsp` skills to detected agent `.skills/` directories\n- [ ] Output reports deployment of both skills (e.g. \"Deployed code-context skill to ...\", \"Deployed lsp skill to ...\")\n- [ ] If one skill deployment fails, the other still proceeds (partial success)\n- [ ] The `lsp` skill resolves from `SkillResolver::resolve_builtins().get(\"lsp\")`\n\n## Tests\n\n- [ ] `test_lsp_skill_exists_in_builtins` — `SkillResolver::new().resolve_builtins().get(\"lsp\")` is `Some` in `code-context-cli/src/skill.rs`\n- [ ] `test_lsp_skill_has_valid_content` — name is `\"lsp\"`, instructions >= 30 lines\n- [ ] `test_deploy_single_skill_code_context` — `deploy_single_skill(\"code-context\")` returns Ok or Err (doesn't panic)\n- [ ] `test_deploy_single_skill_lsp` — `deploy_single_skill(\"lsp\")` returns Ok or Err (doesn't panic)\n- [ ] `test_run_skill_returns_valid_exit_code` — existing test still passes\n- [ ] Run `cargo test -p code-context-cli skill` — all pass\n\n## Workflow\n\n- Use `/tdd` — write failing tests first, then implement to make them pass.

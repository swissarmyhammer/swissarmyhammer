## What

Currently there's a single `perspective.goto` command definition in `swissarmyhammer-commands/builtin/commands/perspective.yaml:136-143` that takes an `id` param from args. This doesn't appear in the command palette as individual options per perspective. The user expects to see \"Go to Perspective: Alpha\", \"Go to Perspective: Beta\", etc. — one palette entry per perspective for the current view kind.

The fix follows the existing dynamic command pattern used for `view.switch:<id>`, `board.switch:<path>`, and `window.focus:<label>` in `commands_for_scope`.

### Files to modify

**1. `swissarmyhammer-kanban/src/scope_commands.rs`**
- Add `PerspectiveInfo` struct (fields: `id: String`, `name: String`, `view: String`) alongside `ViewInfo`, `BoardInfo`, `WindowInfo`
- Add `perspectives: Vec<PerspectiveInfo>` field to `DynamicSources`
- In `commands_for_scope`, after the window loop (~line 452), add a loop that generates `perspective.goto:<id>` commands with name `\"Go to Perspective: <name>\"` and group `\"perspective\"`. Only emit perspectives matching the current view kind (need to derive from scope chain or pass as context)

**2. `kanban-app/src/commands.rs`**
- In the dynamic sources block (~line 1316-1391), gather perspectives from `handle.ctx.perspective_context()`. Call `pctx.all()` to get `&[Perspective]`, map to `PerspectiveInfo { id, name, view }`
- Add `perspectives` field to the `DynamicSources` construction (~line 1386)

**3. `swissarmyhammer-commands/builtin/commands/perspective.yaml`**
- Set `visible: false` on the `perspective.goto` definition (line 136) so the static single-entry command doesn't appear in the palette alongside the dynamic entries

### Design note

The `perspective.goto` command handler already exists and works — it takes an `id` arg. The dynamic command IDs (`perspective.goto:<id>`) need to be parsed in the frontend `dispatchCommand` to extract the perspective ID and pass it as args, following the same pattern as `view.switch:<id>` and `board.switch:<path>`.

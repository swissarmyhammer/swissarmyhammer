## What

Both the Tauri mock_app integration test (`01KPG7ECXR1M5AB3QMJT2CW5CD`) and the planned tauri-driver E2E need two pieces of infrastructure that don't exist yet:

### 1. Synthetic kanban board factory

A test helper that writes a `.kanban` directory to a tempdir with a known, deterministic structure — so tests can assert against exact monikers, column positions, and field values.

**Proposed API**:

```rust
// in a new test-helpers crate or under kanban-app/src/test_support/
pub struct BoardFixture {
    pub path: PathBuf,        // tempdir root
    pub tasks: Vec<String>,   // task ids in layout order
}

/// Write a 3x3 board (3 columns, 3 tasks per column) to a tempdir.
/// Column ids: col-1, col-2, col-3. Task ids: task-1-1 through task-3-3
/// (row major: task-<col>-<row>). Keep alive for the test's lifetime.
pub fn write_3x3_board() -> (TempDir, BoardFixture);

/// More exotic shapes for specific tests.
pub fn write_long_column(rows: usize) -> (TempDir, BoardFixture);
pub fn write_grid_view_fixture() -> (TempDir, BoardFixture);
```

The factory must produce valid `.kanban` content that `AppState::auto_open_board` (or the `--only` path from `01KPG7Q24NPGD4ZN3S4C36S6W5`) can open. Reuse `swissarmyhammer-kanban` helpers where possible rather than hand-rolling JSONL.

### 2. `__spatial_dump` debug command

Tests can assert against the DOM and against event payloads, but they cannot inspect the Rust spatial state (focused key, layer stack depth, entry count, layer contents) without a command to expose it. Asserting entirely via DOM is fragile and doesn't catch cases where Rust and React diverge.

**Proposed command**:

```rust
#[derive(Serialize)]
pub struct SpatialDump {
    pub focused_key: Option<String>,
    pub focused_moniker: Option<String>,
    pub entry_count: usize,
    pub layer_stack: Vec<LayerDumpEntry>,
}

#[derive(Serialize)]
pub struct LayerDumpEntry {
    pub key: String,
    pub name: String,
    pub last_focused: Option<String>,
    pub entry_count_in_layer: usize,
}

#[tauri::command]
pub async fn __spatial_dump(state: State<'_, AppState>) -> Result<SpatialDump, String>;
```

### Why a debug command rather than exposing state directly

The alternative is to have tests reach into `AppState` directly via `app.state()`. That works for Rust-side integration tests (`01KPG7ECXR1M5AB3QMJT2CW5CD`) but not for the tauri-driver E2E, which drives the app from outside via IPC. A Tauri command works for both. Prefixing with `__` signals "testing only" — it should not appear in the user-facing command registry.

### Guard against shipping to prod

- Gate behind `#[cfg(any(debug_assertions, feature = "test-support"))]` so release builds don't expose internals
- Do NOT register `__spatial_dump` in the main `invoke_handler` under release builds
- Add a compile-time test that release builds don't include the symbol

## Implementation Notes

**Module layout chosen: single `kanban-app/src/test_support.rs`** — gated with `#[cfg(test)]` rather than `debug_assertions`. The fixture factory is only called from Rust test harnesses (in-process `mock_app` tests), so `cfg(test)` is the tighter gate and lets it consume the existing `tempfile` dev-dependency without feature-flag plumbing. `__spatial_dump` keeps the `debug_assertions` gate because it has to be reachable from a running debug binary (tauri-driver E2E).

**Changes:**

- `swissarmyhammer-spatial-nav/src/spatial_state.rs`: added `entries_snapshot()` and `layers_snapshot()` accessors on `SpatialState` so the Tauri command can render a dump without reaching into internals.
- `kanban-app/src/spatial.rs`: added `SpatialDump`, `LayerDumpEntry`, and `__spatial_dump` — all gated behind `#[cfg(debug_assertions)]`. Three unit tests cover the payload shape (focus+moniker resolution, two-layer counts, nothing-focused edge case).
- `kanban-app/src/test_support.rs`: new module providing `BoardFixture`, `write_3x3_board`, `write_long_column`, `write_grid_view_fixture`, and a shared `build_fixture` core. Deletes the default InitBoard columns so tests see exactly `col-1..col-N`. Three unit tests cover shape + openability.
- `kanban-app/src/main.rs`: registers `spatial::__spatial_dump` inside `tauri::generate_handler!` behind `#[cfg(debug_assertions)]`; the registration attribute is supported by tauri-macros 2.5.

### Blockers

- [x] `kanban-app/src/test_support.rs:223-226` — The duration-based `assert!(build_ms < 2000, ...)` is machine-dependent and has already been reported to fail with `3671ms, expected <100ms` on `cargo nextest run --workspace` on a slower host. Even with the current 2000ms ceiling the margin is only ~30% over what this reviewer machine observes (1408ms), and CI/less-powerful dev boxes can easily blow past 2000ms. Recommended fixes, in order of preference: (a) **drop the duration assertion entirely** — the test already verifies correctness (nine tasks, three columns, row-major order, openable via `AppState`); performance of a test-only fixture builder does not belong in a correctness test. If performance regression is a real concern, put it behind `#[ignore]` or into a dedicated criterion benchmark. (b) If the duration check must stay, raise the threshold to something defensibly safe on CI (e.g. 10s) and document why the 100ms acceptance criterion was downgraded. The ACs in the task body still claim "<100ms" — if the assertion is removed or loosened, update the AC checkbox narrative to reflect what was actually shipped so the acceptance record matches reality.

### Warnings

- [x] `kanban-app/src/test_support.rs:88` — Doc comment claims "Designed to complete in well under 100 ms — the 9 task writes dominate and each is a single file." This is contradicted by the reviewer's own 1.4s measurement and the reporter's 3.7s.

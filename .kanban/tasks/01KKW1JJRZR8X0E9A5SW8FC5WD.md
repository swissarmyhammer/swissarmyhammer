avp-common/src/install.rs:145-164\n\nThe `install` function reads the settings file at `path` (line 145-152) before creating the parent directory (lines 154-157). When the file does not exist and the parent directory also does not exist, this is fine because it falls through to `json!({})`. However, the `fs::write` call at line 164 will silently fail if the `create_dir_all` step is skipped or fails — but more importantly, the ordering is wrong by design: the directory should be created *before* attempting to read or write the file. If `path.exists()` were ever `true` for a path whose parent does not exist (impossible on a well-formed filesystem, but the ordering is still misleading and fragile).\n\nMore critically: `create_dir_all` is called only when `path.parent()` is `Some`. For a bare filename with no parent component (e.g., `settings.json` in current dir), `path.parent()` returns `Some(\"\")` which is an empty path — `create_dir_all(\"\")` is a no-op on most platforms, which is correct. However, the real bug potential is that `create_dir_all` happens *after* the file-read attempt but *before* the write. If the read fails with an I/O error due to the missing parent, the error message will be confusing since the directory hasn't been created yet.\n\nSuggestion: Move `create_dir_all` to before the read attempt, or at minimum document why the current order is safe.\n\nVerification: `cargo nextest run -E 'rdeps(avp-common)'` passes; add a test that installs when neither .claude/ nor settings.json exist (already covered by `test_install_creates_hooks_and_avp_dir` — verify it exercises the right path)." #review-finding

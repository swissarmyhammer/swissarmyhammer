## What\n\nAggregate computed fields like `board-percent-complete` (registered in `defaults.rs:134-172` via `register_aggregate`) query other entity types (`task`, `column`) at read time. But when those source entities change — via drag, file edit, MCP, CLI, anything — no `entity-field-changed` event is emitted for the board.\n\n### Approach: YAML-declared dependencies\n\n1. Add `depends_on` to the computed field YAML schema. Example for `percent_complete.yaml`:\n   ```yaml\n   type:\n     kind: computed\n     derive: board-percent-complete\n     depends_on: [task, column]\n   ```\n2. The entity change pipeline reads `depends_on` from field definitions. When `entity-field-changed` fires for a `task` or `column`, the engine checks: which computed fields across all entity types declare `depends_on` containing this entity type?\n3. For each match, recompute the aggregate and emit `entity-field-changed` for the owning entity (e.g., board).\n\nThis is declarative, auditable in YAML, and requires no hardcoded logic in any command.\n\n### Files to modify\n- `swissarmyhammer-kanban/builtin/fields/definitions/percent_complete.yaml` — add `depends_on: [task, column]`\n- The entity crate's field type schema — parse `depends_on` from YAML\n- The entity change event pipeline — after emitting `entity-field-changed`, check for dependent aggregates and cascade\n\n## Acceptance Criteria\n- [ ] `depends_on` is parsed from field YAML and available in the field type metadata\n- [ ] Changing a task (by any means) triggers board percent_complete recomputation\n- [ ] Board emits `entity-field-changed` with updated percent_complete\n- [ ] No hardcoded logic in MoveTask or any specific command\n\n## Tests\n- [ ] Rust unit test: parse `depends_on` from field YAML\n- [ ] Rust integration test: create board with 2 tasks, move one to done → board percent_complete == 50\n- [ ] Rust integration test: move it back → percent_complete == 0\n- [ ] Run: `cd swissarmyhammer-kanban && cargo nextest run aggregate_recompute`"

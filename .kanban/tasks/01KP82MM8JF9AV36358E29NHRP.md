#entity-cache

Parent context: `01KP7K4NMJRET0J4SQQ8950M6H` (entity-cache 3/4) met its primary <20ms `MoveTask` target by caching compute *inputs* (`_changelog`, `_file_created`) but did not meet its secondary <5ms `list_task` sanity-check target. Actual measurement after Option A landed:

- `list_task` (single `ectx.list("task")` on 2000 tasks): median **18.95ms** (target <5ms).

Option A eliminated the per-task I/O ‚Äî the compute-input cache short-circuits `read_changelog` and `stat` on cache hits ‚Äî but the compute-engine itself still takes ~18ms for 2000 tasks with 4 system-date fields each. That is the remaining dominant cost.

## What

The compute engine (`ComputeEngine::derive_all` in `swissarmyhammer-fields`) runs on every `list()` call for every entity. For the `task` type it derives 4 system-date fields (`created`, `updated`, `started`, `completed`) by parsing the cached `_changelog` array. The inputs are identical across repeat calls (until the entity mutates), so the *outputs* are also identical ‚Äî they are prime caching targets.

This card is Option B from the parent card's planning section: cache the computed-field outputs alongside the entity in `CachedEntity` and invalidate them in the same mutation paths already wired for Option A.

## Approach

Add a `derived: Option<IndexMap<FieldName, serde_json::Value>>` slot to `CachedEntity` (or a sibling map keyed by `(entity_type, id)` ‚Äî whichever avoids lock contention with the existing primary `cache` RwLock). Populate it lazily on first `list()`/`read()` when a compute pass produces values. Invalidate it from every mutation path that already invalidates the compute-input cache:

- `write` / `refresh_from_disk` / `unarchive` ‚Äî clear the derived slot (next compute pass repopulates)
- `delete` / `evict` / `archive` ‚Äî purge the derived slot (entity is gone)

The epoch-based race protection already in place for `compute_inputs` is the template ‚Äî either reuse the same epoch or add a sibling `derived_epoch` depending on how independent the two caches need to be.

## Implementation Summary

- **Primary cache change** (`swissarmyhammer-entity/src/cache.rs`): Added a third per-entity map, `derived: RwLock<HashMap<(String, String), CachedDerived>>`, that memoizes every computed-field output ‚Äî both simple derivations (`created`, `updated`, `started`, `completed`, `status_date`, `progress`, `virtual_tags`, `filter_tags` on `task`) and aggregates (`tags`). Renamed the existing `compute_inputs_epoch` to `cache_epoch` since it now guards both secondary caches. Renamed `invalidate_compute_inputs` ‚Üí `invalidate_entity_caches` and `purge_compute_inputs` ‚Üí `purge_entity_caches` since they now clear both the inputs and the derived outputs.
- **Cross-entity invalidation** (new): Aggregate derivations can query sibling entity types, so entity-local invalidation alone is unsafe. Added `invalidate_cross_type_derived(trigger_type)` which consults `FieldsContext::entity_types_depending_on(trigger_type)` and clears the derived-output slots of every entity type whose aggregate fields declare `depends_on: [trigger_type]`. Called from both `invalidate_entity_caches` and `purge_entity_caches`, so every mutation path (write/delete/evict/archive/unarchive/refresh_from_disk) propagates correctly.
- **Compute path change** (`swissarmyhammer-entity/src/context.rs`): `derive_compute_fields` now fetches cached outputs at the top of each per-entity call. On a warm hit it copies cached values into `entity.fields` and skips both the pseudo-field injection (`_changelog` / `_file_created`) and the compute-engine run entirely. On a cold miss it runs the derivations in declaration order (preserving the `derive_all` ordering contract) and memoizes the results under the epoch guard.
- **Field definition bug fix** (`swissarmyhammer-kanban/builtin/definitions/tags.yaml`): added `depends_on: [tag]` to the `tags` computed field. Without this, `FieldsContext::entity_types_depending_on("tag")` would return nothing and writing a tag would never invalidate the cached `task.tags` values. This declaration is required by the cross-type invalidation mechanism ‚Äî fixing it is what lets the `parse-body-tags` aggregate be cached correctly.

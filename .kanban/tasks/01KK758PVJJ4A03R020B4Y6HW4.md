**B1: Duplicate color palettes with divergent hash algorithms**

Three independent color derivation implementations exist:
- `state.rs:469` — djb2 hash over 18-color palette
- `avatar.tsx:38` — different hash function over different palette  
- `entity-inspector.tsx:283` — yet another palette inline

When an actor has no persisted `color` field, each location derives a different color from the same ID, causing visual mismatch between avatar, inspector, and backend default.

**Fix:** Extract a single canonical palette + hash into a shared location. Rust should persist the color on actor creation so the frontend never needs to derive it. Remove the frontend fallback palettes or align them as a safety net using the same algorithm.

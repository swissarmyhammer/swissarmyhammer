Raise `notifications/commands/changed` when the set of registered commands changes (plugin load/unload, register/unregister), so the palette refreshes and plugins can react.

Current state: the `notifications/commands/changed` constructor exists (notify.rs:279) but has NO publisher. The command service schedules a debounced `self.notifier.notify()` (service.rs:345) but never publishes this notification. The frontend availability cache already listens for a `commands/changed` epoch bump (use-command-availability.ts) — but nothing emits it.

## Work
- Publish `notifications/commands/changed` on the bridge from the command service when a command is registered/unregistered (and on plugin load/unload command churn). Hook the existing debounced notifier path (service.rs:345) to actually publish.
- Confirm the frontend palette/availability already consumes the bridge-forwarded `notifications/commands/changed` (it expects it); wire if needed.
- Declare on the command service tool (service.rs:285) via #[notification] struct=payload.
- Coverage guard.

## Acceptance
Registering/unregistering a command publishes `notifications/commands/changed`; the palette refreshes; a plugin can `this.commands.on("changed", cb)`; declared == published.

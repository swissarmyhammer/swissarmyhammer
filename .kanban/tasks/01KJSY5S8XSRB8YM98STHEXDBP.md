**File:** `swissarmyhammer-kanban/src/attachment/add.rs` (lines 181-187)\n\n**What:** `AddAttachment` writes the attachment entity first (`ectx.write(&attachment)`), then updates the task's attachments list (`ectx.write(&task)`). If the second write fails (e.g., disk full, permission error), the attachment entity exists on disk but the task doesn't reference it -- creating an orphan. The same pattern exists in `DeleteAttachment` (lines 62-69 of `delete.rs`) where the attachment is deleted first and the task update follows.\n\n**Why it matters:** In the delete case, if the task update fails after the attachment entity is deleted, the task's `attachments` list still contains the deleted ID. Future `ListAttachments` calls will silently skip the missing ID (the `if let Ok(entity)` in `list.rs` line 43), but the stale reference persists. This is a consistency concern that could cause confusion during debugging.\n\n**Suggestion:** This is an inherent limitation of file-backed storage without transactions. The current ordering (write attachment first, then task) is the safer direction for add -- at worst you get an orphan, which is recoverable. For delete, the current ordering (delete entity first, update task second) is also defensible since the list silently skips missing IDs.\n\n- [ ] Document the ordering rationale with a comment explaining the two-phase write trade-off in both `add.rs` and `delete.rs`\n- [ ] Consider adding a periodic cleanup/consistency check that finds attachment entities whose `attachment_task` references a non-existent task\n- [ ] Verify understanding by reviewing the code paths #warning

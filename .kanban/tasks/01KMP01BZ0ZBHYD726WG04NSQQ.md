**File:** `kanban-app/ui/src/components/inspector-focus-bridge.tsx:87-102`\n\n**What:** The pill navigation commands define:\n- `inspector.pillLeft` with `keys: { vim: 'h', cua: 'ArrowLeft' }`\n- `inspector.pillRight` with `keys: { vim: 'l', cua: 'ArrowRight' }`\n\nThese are in the same command scope as:\n- `inspector.moveUp` with `keys: { vim: 'k', cua: 'ArrowUp' }`\n- `inspector.moveDown` with `keys: { vim: 'j', cua: 'ArrowDown' }`\n\nThe h/l keys are not colliding with k/j, so vertical nav is fine. But `h` and `l` ARE used by the board nav commands (`board.moveLeft`, `board.moveRight`) in the parent scope. The inspector scope takes precedence, so when the inspector is open, `h`/`l` will trigger pill nav instead of board nav -- which is the intended behavior.\n\nHowever, the `execute` functions for pillLeft/pillRight have a guard `if (navRef.current?.mode === 'normal')` that silently no-ops in edit mode. This means in edit mode, `h`/`l` are consumed by these commands (they match the keybinding) but do nothing -- the keypress is swallowed. The user types `h` expecting text input, but the command scope eats it before the editor sees it.\n\n**Why this matters:** When the inspector is in edit mode on a text field, pressing `h` or `l` in vim mode will be intercepted and silently discarded.\n\n**Suggestion:** These commands should only be registered when the inspector is in normal mode, or the keybinding handler should skip commands whose execute is a no-op. Alternatively, the mode guard should return a falsy value that tells the keybinding handler to continue propagation." #review-finding

title: SavePerspectiveCmd has TOCTOU race between name lookup and create/update
---
perspective_commands.rs:69-93\n\nThe `SavePerspectiveCmd` reads the perspective context to check if a name exists, drops the read lock, then acquires a write lock through `run_op`. Between releasing the read lock and the operation executing, another concurrent caller could create a perspective with the same name, leading to a duplicate.\n\nThis is a low-severity issue because:\n- The window is small (microseconds)\n- Concurrent perspective saves with the same name are unlikely in practice\n- The worst outcome is a duplicate perspective, not data loss\n\nSuggestion: Consider an `upsert_by_name` method on PerspectiveContext that does the check-and-write atomically under one lock acquisition. Alternatively, document the race as acceptable.\n\nVerification: Code review only -- this is a design-level observation." #review-finding

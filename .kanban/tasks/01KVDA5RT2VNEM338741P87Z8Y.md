## What
Restructure the review fan-out so the LARGE content shared across the whole run (change description + file diffs) lives in ONE prime, and small per-rule content fans out as forked suffixes. Maximizes the shared prefix and shrinks the divergent tail — rewards zero-rollback fork reuse on qwen AND Claude's hosted prompt-prefix cache (backend-agnostic win).

Today (`crates/swissarmyhammer-validators/src/review/fleet.rs`): prime is per-VALIDATOR `render_validator_prefix` = `[shared sections + that validator's rules + PRIME_HANDOFF]` (`fleet.rs:335-345,658-666`); per-turn suffix is the file payload `render_file_payload` batched by `batch_size` via `chunks()` in `run_validator_fleet` (`fleet.rs:231-310`). So the big diff is the suffix (reprocessed per validator) and rules are in the prefix — backwards for caching.

Flip it, with these decisions PINNED (resolve the open choices the reviewer flagged):
- **Prime once per review run** = `[system + tools + change description + all file diffs under review]`. No rule-specific text.
- **Fan-out unit = per individual rule** (one forked child per rule), appending only `[rule text + output contract]`. This keeps rule isolation; do NOT fan out per validator-bundle.
- **File batching goes away as a suffix dimension**: the diffs now live entirely in the prime, so `batch_size`/`chunks()` no longer chunks the suffix. Update/replace the batching tests accordingly — specifically `prefix_is_primed_once_per_validator_and_batches_fork_payload_only` and `fan_out_two_validators_two_files_submits_at_most_four_tasks` (these encode the OLD granularity and must be rewritten to the new "one prime + one fork per rule" shape). Consider the total prime size (all diffs at once) — if a change is very large this is a single big prime; that's acceptable and is exactly what we want cached, but note it.
- **Verify stage (concrete, not an open choice)**: verify-stage sessions FORK THE SAME SHARED PRIME, with a test asserting they do. (`collect_forked_task` at `fleet.rs:476-537`.) If that's infeasible, the fallback is "verify stage unchanged" — but the default target is forking the shared prime for reuse.
- **Empty-suffix guard**: ensure each per-rule suffix is non-empty (a rule always contributes text), so the fork turn never degenerates to `lcp == new_len` (full reprocess) — coordinate with w37g2tw's empty-suffix handling.

Backend-agnostic: no backend branching in the assembly. Preserve review semantics/output contract and the kanban GFM-checklist result.

## What

Reported behavior: dragging a file into an attachment field shows the global error overlay ("Something went wrong") with a stack trace pointing at `name.lastIndexOf`.

### Root cause

`kanban-app/ui/src/components/fields/displays/attachment-display.tsx`

The drop pipeline is:

1. `FileDropProvider` (`kanban-app/ui/src/lib/file-drop-context.tsx`) handles the HTML5 drop, calls `save_dropped_file` to write the dataTransfer bytes to a temp file, and invokes the topmost registered drop callback with `paths: string[]` (temp paths).
2. `AttachmentListDisplay`'s drop callback commits `[...current, ...paths]` — i.e. pushes a raw **string** path into an array typed as `AttachmentMeta[]`. This is the documented contract: the entity layer (`swissarmyhammer-kanban/src/attachment/add.rs::process_attachment_field`) accepts string paths in the `attachments` field, copies the file into `.attachments/`, and on next read enriches the entry into a full `AttachmentMeta` (`{ id, name, size, mime_type, path }`).
3. **Between commit and the read-back**, React re-renders with the array containing a string. `AttachmentListDisplay` does:

   ```tsx
   const attachments = Array.isArray(value) ? (value as AttachmentMeta[]) : [];
   // ...
   {attachments.map((att) => (
     <AttachmentItem key={att.id} attachment={att} />
   ))}
   ```

   The cast hides the fact that `att` may be a string. `AttachmentItem` then calls:

   ```ts
   getFileIcon(attachment.mime_type, attachment.name)
   ```

   With `attachment` being a string, both `.mime_type` and `.name` are `undefined`. Inside `getFileIcon`, `getExtension(undefined)` runs `undefined.lastIndexOf(".")` → **TypeError: Cannot read properties of undefined (reading 'lastIndexOf')**. The top-level error boundary surfaces this as "Something went wrong".

### Why the editor doesn't crash

`AttachmentEditor` (`kanban-app/ui/src/components/fields/editors/attachment-editor.tsx:132–143`) explicitly branches on `typeof att === "string"` and renders pending paths as plain text. The display does not.

### Single-attachment variant

`AttachmentDisplay` (the non-list variant) does not crash because its `hasAttachment` guard requires `typeof attachment === "object"`, so a string commit is treated as empty and the drop-zone placeholder is shown — but visual feedback is briefly lost during the round-trip. That UX gap is **out of scope** for this task; this task fixes the crash in the list variant only. File a follow-up if the single-display feedback matters.

### Fix

`AttachmentListDisplay` must tolerate string entries during the commit→read round-trip, mirroring the editor's pattern. Two reasonable approaches:

- **Render pending paths inline** (mirrors editor): when an entry is a string, render a row with the file's basename (`path.split('/').pop()`) and a muted style — same shape as `AttachmentItem` so the row doesn't reflow when the metadata arrives.
- **Filter pending paths out**: only `AttachmentMeta` entries hit `AttachmentItem`. Cleaner code but loses the just-dropped visual.

Recommend the first approach for parity with the editor — users see immediate feedback that the drop landed.

Either way, fixing `getFileIcon`/`getExtension` to defensively handle `undefined` is **not** the right fix on its own — it only papers over the type-cast lie in `AttachmentListDisplay`. Fix the cast, then the helpers can stay strict.

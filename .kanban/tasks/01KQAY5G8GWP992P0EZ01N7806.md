## What

When two `FocusZone`s register at the same `(x, y)` in the same layer — or two `FocusScope`s do — emit `tracing::warn!` from the kernel registry. This is almost always **needless nesting**: a parent zone that wraps a single child with no offset, two siblings stacked at the same anchor due to a pass-through wrapper, etc. The warning gives us a hunting tool to find and remove these in the React tree without staring at every component file.

Same-kind only: zone-overlapping-zone, scope-overlapping-scope. A scope at the same `(x, y)` as its parent zone is normal (a leaf as the sole child of an unpadded zone), so cross-kind overlaps are NOT warned. The user can refine the rule after seeing the noise level if needed.

## Where the bug lives

The registry inserts entries via `SpatialRegistry::register_scope` (`registry.rs:91`) and `register_zone` (`registry.rs:101`). Rect updates happen through `update_rect` (`registry.rs:129`). All three are mutation points where a same-(x, y) overlap can come into being.

## Approach

`swissarmyhammer-focus/src/registry.rs`:

1. Add a private helper `check_overlap_warning(&self, kind: ScopeKind, layer: &LayerKey, rect: &Rect, exclude: &SpatialKey)` that scans `self.entries_in_layer(layer)`, filtering to same-kind entries excluding `exclude`, and returns the first match where the entry's rounded `(left, top)` equals the input's rounded `(left, top)`.
2. Round to integer pixels for the comparison — `rect.left().round() as i64`, `rect.top().round() as i64`. Subpixel rendering produces tiny variations that aren't user-relevant; rounding catches the structural overlaps that matter.
3. Call the helper from `register_scope`, `register_zone`, and `update_rect` after the mutation has applied. When a match is found, emit:

```rust
tracing::warn!(
    op = "register_scope" | "register_zone" | "update_rect",
    layer = %layer_key,
    new_key = %new.key(),
    new_moniker = %new.moniker(),
    overlap_key = %existing.key(),
    overlap_moniker = %existing.moniker(),
    x = rounded_x,
    y = rounded_y,
    "two same-kind entries share (x, y); likely needless nesting — review React tree for redundant wrappers"
);
```

4. Skip the check when the registry has fewer than two entries (cold start; nothing to overlap with).

5. Decide whether to suppress the warning during `update_rect` if the moved entry was already overlapping the same other entry pre-update. Probably YES — `update_rect` fires every animation frame during `01KQ9XBAG5P9W3JREQYNGAYM8Y`'s scroll-tracking, and re-warning every frame would flood the log. Track a per-key "last overlap warn" target and skip re-warning if the (key, overlap_key) pair is the same as last time. Reset on `unregister_scope`.

## What this catches

Concrete cases this trace surfaces in production today (the user can hunt them down once it's wired):

- A `<FocusZone>` whose only child is a `<FocusScope>` with no DOM offset between them — the zone adds no spatial structure.
- An `<Inspectable>` wrap that introduces a sibling `<FocusZone>` and `<FocusScope>` with the same moniker at the same position — that's two same-kind entries in some shapes.
- An accidental double-mount of the same component (e.g., a list re-render that mounts both old and new keys briefly) — the warning fires once during the transition.

## Scope NOT in this ticket

- The overlap check does NOT remove anything. It only warns. Removing the actual nesting is a follow-up driven by the warnings the user collects.
- The check does NOT consider full rect overlap (matching width/height too). Same-`(x, y)` is the strong signal; full overlap is a stricter check that may follow.
- Cross-layer overlaps are out of scope — different layers serve different surfaces, and overlap there is structurally meaningful, not a smell.

## Acceptance Criteria

All asserted by automated tests below — no manual smoke step.

- [ ] Registering two zones at the same rounded `(x, y)` in the same layer emits exactly one `WARN`-level tracing event with `op = "register_zone"`, both monikers, both keys, and the rounded coordinates.
- [ ] Registering two scopes at the same rounded `(x, y)` in the same layer emits exactly one `WARN` with `op = "register_scope"`.
- [ ] Registering a scope and a zone at the same `(x, y)` does NOT emit a warning. (Cross-kind is allowed.)
- [ ] Registering a zone and a scope at the same `(x, y)` does NOT emit a warning. (Same as above, both directions.)
- [ ] Registering two same-kind entries at different `(x, y)` (off by ≥ 1 pixel after rounding) does NOT emit a warning.
- [ ] Registering two same-kind entries at the same `(x, y)` but in **different layers** does NOT emit a warning.
- [ ] `update_rect` that moves an entry onto an existing same-kind entry at `(x, y)` emits exactly one `WARN`.
- [ ] `update_rect` called repeatedly with the same coordinates while an overlap remains does NOT re-emit the warning every frame. (Per-key suppression for the same overlap pair.)
- [ ] `update_rect` that moves an entry **off** an existing overlap and back ON does emit a fresh warning the second time. (The suppression releases when the overlap clears.)
- [ ] `unregister_scope` clears any per-key suppression state for that key.
- [ ] No regression: existing `register_scope` / `register_zone` / `update_rect` tests pass.

- [ ] `two_scopes_same_xy_same_layer_warns_once` — same with two scopes; assert `op = "register_scope"`.
- [ ] `zone_and_scope_same_xy_does_not_warn` — register zone at `(100, 80)`, scope at `(100, 80)` same layer; zero warnings.
- [ ] `two_zones_different_xy_does_not_warn` — register zone A at `(100, 80)`, zone B at `(120, 80)` (1+ pixel off); zero warnings.
- [ ] `two_zones_subpixel_difference_does_not_warn` — register zone A at `(100.0, 80.0)`, zone B at `(100.3, 80.2)`; both round to `(100, 80)` — wait, that DOES match. Test the other direction: `(100.6, 80.0)` rounds to `(101, 80)`; assert zero warnings. (Rounding is the cutoff.)

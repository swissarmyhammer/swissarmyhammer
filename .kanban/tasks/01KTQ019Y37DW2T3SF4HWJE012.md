## Problem (pre-existing; surfaced during review of m8zac70)

In `crates/llama-agent/src/queue.rs`, `submit_request` removes the session's entry from `active_requests` only on the **success** path (`self.active_requests.lock().await.remove(&session_id)` after the response resolves). The `?` early-returns (e.g. `enqueue_request` failing/cancelled, response channel closed) return before that line, so the `active_requests` map entry for that session is never removed.

Over a long-lived shared `AgentServer` (which is cached across all connections), failed/cancelled requests accumulate stale entries → unbounded map growth.

Not introduced by the backpressure change (m8zac70) — the backpressure work only added one more early-return path (cancellation) that shares the same leak.

## Fix direction

Ensure the `active_requests` entry is removed on **every** exit path — e.g. an RAII guard that removes on drop, or remove in all error branches. Add a test asserting the map is empty after a failed/cancelled submit.

## Acceptance criteria

- `active_requests` has no entry for a session after `submit_request` returns, on success, error, and cancellation paths.
- Test covers the error/cancellation paths (use the existing `gated_queue`/`scripted_queue`/`GatedExecutor` harness in `queue.rs`; no real model; <10s).

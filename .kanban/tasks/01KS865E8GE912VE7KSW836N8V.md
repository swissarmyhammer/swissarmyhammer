#elicitation

## Context / Why
Prerequisite for both Rust agent-wrapper elicitation bridges. The ACP elicitation types (`CreateElicitationRequest`, `CreateElicitationResponse`, `ElicitationAction`) live in the `agent-client-protocol-schema` crate behind its `unstable_elicitation` feature. The user approved enabling the unstable feature; this is the deferred follow-up the board's "ACP 0.10 → 0.11 upgrade" project explicitly left open ("Excludes opting into the `unstable` feature unless a follow-up task chooses to").

## Verified findings (so the implementer doesn't re-derive)
- The **main** `agent-client-protocol` crate does NOT expose/forward an `unstable_elicitation` feature at any available version (checked 0.11.1, 0.12.0, 0.12.1) — its `[features]` forwards 10 other `unstable_*` features but not elicitation. So it must be enabled on the **schema** crate directly.
- `agent-client-protocol-schema` 0.12.0 defines `unstable_elicitation = []` and gates `CreateElicitationRequest`/`CreateElicitationResponse`/`ElicitationAction` plus `ELICITATION_CREATE_METHOD_NAME = "elicitation/create"` and `ELICITATION_COMPLETE_NOTIFICATION = "elicitation/complete"` behind it (see `src/client.rs`).
- These method names MATCH the TS SDK (`elicitation/create`, `elicitation/complete`) — no cross-SDK reconciliation needed.
- `Cargo.lock` already resolves `agent-client-protocol-schema = 0.12.0` (main crate still 0.11.1).
- Workspace `Cargo.toml` opens with "NEVER add features or feature flags to this workspace!" — that rule targets gating SAH functionality; enabling a third-party dependency feature **always-on** (no optionality) is consistent with "all functionality all the time". Keep it unconditional.

## IMPLEMENTATION NOTE — version bump was UNNECESSARY (smaller change taken)
The literal "What" said to bump `agent-client-protocol` to 0.12.1. While implementing I verified that **main 0.12.1 pins `agent-client-protocol-schema = "=0.13.2"`** (an exact-version pin), whereas **main 0.11.1 pins `=0.12.0`**. So bumping the main crate would have forced schema 0.13.2 and introduced 0.11→0.12 API-drift risk across 7 crates — for no benefit, since the dependents only need the elicitation *types reachable*, not any new 0.12 main-crate API.

Per the task's own escape hatch ("If the version bump turns out to be unnecessary … prefer the smaller change and document what you did"), I took **Path A**: keep `agent-client-protocol = "0.11"` and add `agent-client-protocol-schema = { version = "=0.12.0", features = ["unstable_elicitation"] }` to `[workspace.dependencies]`, then reference it from `agent-client-protocol-extras` so Cargo feature-unification turns the types on through the main crate's `agent_client_protocol::schema` re-export (`pub use agent_client_protocol_schema::*`).

Result: minimal change (Cargo.lock gains exactly one line — schema added to extras' dep list; schema stays 0.12.0; no main-crate bump; zero API drift). Every ACP-using crate (`claude-agent`, `llama-agent`, `avp-common`, `acp-conformance`, `swissarmyhammer-agent`) depends on `agent-client-protocol-extras`, so the feature unifies for all of them. `cargo tree -e features` confirms `agent-client-protocol-schema v0.12.0` builds with `feature "unstable_elicitation"`.

### Warnings
- [x] `crates/agent-client-protocol-extras/Cargo.toml:17` (and `Cargo.toml:196`) — The `unstable_elicitation` feature is enabled only on `agent-client-protocol-extras`, and the types are reachable from `claude-agent`/`llama-agent` solely because those crates transitively depend on extras. This is correct today (resolver="2" feature unification over a single `=0.12.0` schema instance), but the feature lives in a crate logically unrelated to "I need elicitation types" — spooky action at a distance. Risk: if a future ACP-consuming crate uses elicitation but does not depend on extras (or `claude-agent`/`llama-agent` later drop their extras dep), `agent_client_protocol::schema::CreateElicitationRequest` would silently fail to resolve for that crate's own build config, while `cargo build --workspace` (which always pulls extras) stays green — so the existing reachability test would NOT catch it. Hardening (low cost, ~2 lines, not a correctness blocker): also declare `agent-client-protocol-schema = { workspace = true }` (it already carries the `unstable_elicitation` feature) directly in the `[dependencies]` of `claude-agent` and `llama-agent` so each crate is self-sufficient and enables the feature in its own build config, independent of extras. Leaving it as-is is acceptable given the current invariant holds; flagging so the choice is deliberate.

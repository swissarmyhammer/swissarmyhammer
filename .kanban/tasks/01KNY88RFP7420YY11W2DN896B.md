## What

In the perspective filter editor, projects display inconsistently. The backend filter DSL `has_project` (in `swissarmyhammer-kanban/src/task_helpers.rs` around line 508) matches a task's stored `project` field (which is the project **id**) case-insensitively against the filter value. But the frontend generates mention text from `slugify(project.name)` instead of from the id.

Where the divergence lives today:

- `kanban-app/ui/src/hooks/use-mention-extensions.ts` `buildAsyncSearch` (around lines 110-132) receives `{id, display_name, color}` from the backend `search_mentions` command but discards `id` and emits `slug: slugify(r.display_name)`.
- Same file, `buildColorMap` and `buildMetaMap` (around lines 80-107) key their CM6 decoration/tooltip maps by `slugify(getStr(e, displayField))` — not by id.
- `kanban-app/ui/src/components/mention-pill.tsx` (around lines 71-77) resolves a slug to an entity by trying `val === slug || slugify(val) === slug` first, then falling back to `e.id === slug`.
- `kanban-app/ui/src/components/fields/displays/badge-display.tsx` `resolveReferenceBadge` (around lines 22-39) looks up the entity by id and renders `mention_display_field` (the name) as the label.

When a project's id is NOT equal to `slugify(project.name)` — which happens whenever a project id was created as free-form text (e.g., id `AUTH-Migration`, name `Auth Migration System`) — the mention inserted by autocomplete is `$auth-migration-system`, which never matches the task's stored `project` = `AUTH-Migration`. The filter silently breaks.

## Design

**Unify everything on `project.id`. Name is presentation-only: tooltip text on pills and secondary detail in the autocomplete dropdown.**

- Pills render `$<project.id>`.
- Autocomplete dropdown shows `$<project.id>` as the primary label with `project.name` as the detail line and a colored-dot + name block in the info panel (already wired — see `cm-mention-autocomplete.ts` `buildResult`).
- CM6 decorations match the id-shaped slug and color it via the project color.
- Reference-field badges on task cards render `project.id` as the label with `project.name` as the hover tooltip.

Scope this to **projects only**. Tag/actor ids are already slug-shaped, so the current behavior is unchanged for them. Task mentions (`^` prefix, ULID ids vs. slugified titles) are a separate concern — out of scope for this card; file a follow-up if needed.

**Mechanism: a new schema signal `mention_slug_field`.** When set on an entity definition, the mention slug is sourced from that field instead of `slugify(mention_display_field)`. Default (not set) preserves existing behavior so tag/actor mentions are untouched.

## Non-goals

- Task mentions (`^` prefix) with ULID ids vs. slugified titles — out of scope; file as a follow-up card if needed.
- Tag and actor mentions — no behavior change; their ids are already slug-shaped so `mention_slug_field` stays unset.
- Changing `has_project` filter semantics in `swissarmyhammer-kanban/src/task_helpers.rs` — it already matches on the stored project id, which is what we are aligning to.
- Data migration for existing saved perspectives that reference projects by slugified name — those filters are already broken (that is the bug). After this change, users must re-pick the project from autocomplete. Call this out in the PR description; do not ship a silent rewrite.

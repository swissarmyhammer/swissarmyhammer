### Nits
- [ ] `crates/swissarmyhammer-tools/src/mcp/tool_registry.rs:829` — The default `schema_full()` returns `self.schema()`, which is a silent footgun for the exact tool class the method exists for: any operation-based tool whose `schema()` returns the slim wire surface but which forgets to override `schema_full()` will feed the in-process CLI generator a schema with no flat `properties`, producing a command tree missing every parameter — with no compile error and no test catching it. Either derive the full schema from `operations()` in the default (so op-based tools get it for free and the default is correct for the whole class), or add a registration-time/debug assertion that any tool returning a non-empty `operations()` whose `schema()` omits `x-operation-schemas` also overrides `schema_full()`.
- [ ] `crates/swissarmyhammer-tools/src/mcp/tools/files/schema.rs:22` — Naming asymmetry between the two public schema generators: the slim model-facing one is the unsuffixed `generate_files_mcp_schema` while its sibling is `generate_files_mcp_schema_full`. An unsuffixed name reads as 'the complete schema', so a caller skimming the API is likely to reach for `generate_files_mcp_schema` expecting the full surface and silently get the slim wire schema. The underlying primitives it wraps are explicitly named `generate_mcp_schema_wire` / `generate_mcp_schema_full`, so the wrapper drops the distinguishing term on one side only. Rename the wire generator to `generate_files_mcp_schema_wire` to mirror both the underlying `generate_mcp_schema_wire` primitive and the `_full` sibling, making the surface each one produces unambiguous at the call site.
- [ ] `crates/swissarmyhammer-tools/tests/integration/file_tools_integrations.rs:805` — The `oneOf` branch of `verify_tool_schema` nests 4 levels deep (else-if-let → for alternative → if-let alt_required → for (idx, prop) → if contains), and the `found_in_any_alternative` vector it laboriously computes is never read or asserted on — dead nesting that makes the helper hard to reason about. The diff just retargeted this helper from `schema()` to `schema_full()`, so the `oneOf` path is now unreachable in practice (full schemas carry a flat `required` array, not `oneOf`). Delete the entire `else if let Some(one_of)` branch. Since `schema_full()` always emits a flat `required` array, the `if let Some(required)` arm is sufficient; removing the dead `oneOf` arm drops the helper to a single nesting level.
- [ ] `crates/swissarmyhammer-tools/tests/integration/web_fetch_specification_compliance.rs:44` — Five parallel `assert!(properties["X"].is_object(), "Should have X property")` statements (url, timeout, follow_redirects, max_content_length, user_agent) differ only in the property-name constant. That is a table walked by one assertion, not five hand-maintained arms — adding/removing a fetch parameter today means editing five lockstep copies, and a typo in any one (mismatched key vs. message) is easy to miss. Drive it from a slice: `for prop in ["url", "timeout", "follow_redirects", "max_content_length", "user_agent"] { assert!(properties[prop].is_object(), "Should have {prop} property"); }`.

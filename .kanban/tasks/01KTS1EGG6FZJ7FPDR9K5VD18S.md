## What

Every real-model verification so far has depended on the ad-hoc ../calcutron checkout. Commit a small, repeatable harness so we can prove a real local-qwen review works from this repo alone.

Components:

- [x] **Seeded sample crate** at `scripts/review-verify/sample/`: a tiny Rust crate (its own `Cargo.toml` containing an empty `[workspace]` table so it detaches from the parent workspace — the root `Cargo.toml` uses an explicit members list, so it must NOT be added there) with 2–3 small `.rs` files containing planted, obvious findings: duplicated function bodies (same logic copy-pasted with renamed identifiers) and magic numbers (repeated bare numeric literals in logic). Keep it small enough that a local-model review completes in minutes.
- [x] **Driver script** committed at `scripts/review-verify/drive.py`, adapted from the working prototype at `/tmp/sah-rev-verify/drive.py`: a minimal MCP stdio client that spawns `sah serve --model qwen --cwd <sample>`, sends `initialize` + `notifications/initialized`, calls `tools/list`, then calls the `review` tool with `{"op": "review file", "path": "<sample>/src/<file>.rs", "backend": "local", "validators": ["duplication", "magic-numbers"]}`, answering any server→client request with a method-not-found error so nothing blocks. Parameterize the sample path relative to the repo root instead of hardcoding /tmp.
- [x] **Machine-checkable assertions** in drive.py (exit nonzero on any failure):
  - review result markdown non-empty AND findings > 0
  - `counts.failed == 0` and `counts.attempted > 0` (the serialized names of the engine's tasks_failed / tasks_attempted tallies, per `ReviewCountsView`)
  - zero "Queue is full" lines in the sample dir's `.sah/mcp.<pid>.log`
  - at least one "AgentMessage (" reply line in that log
- [x] **Docs**: a `scripts/review-verify/README.md` covering how to run: build first (`just sah`), then in the sample dir `sah init && sah model use qwen` (or rely on the `--model qwen` flag), then `python3 scripts/review-verify/drive.py`.

Implementation note: drive.py `git init`s the sample dir on first run — `sah serve` resolves its `.sah/` data dir (logs, code-context index) at the git root of its cwd, so without a nested repo the per-pid log would land in this repository's root `.sah/` instead of `sample/.sah/`. Paths under a `.git` component are invisible to the parent repository.

- [x] `scripts/review-verify/drive.py:282` — McpClient permits an illegal state: two threads writing to the same stdin pipe with no synchronization. The reader thread calls `self.send()` to decline server->client requests while the main thread is also calling `send()` (it spends most of its time in `time.sleep`, during which a server request can arrive). Interleaved writes would corrupt the newline-delimited JSON-RPC stream; today this only works by leaning on CPython's internal TextIOWrapper locking making a single `write()` call effectively atomic, which is an implementation detail, not a contract. Add a `threading.Lock` to McpClient and take it around the `write` + `flush` pair in `send()` so the class cannot emit interleaved frames regardless of which thread sends. RESOLVED: `McpClient.stdin_lock` (threading.Lock) now guards the write+flush pair in `send()`, with a docstring stating why.
- [x] `scripts/review-verify/drive.py:290` — The JSON-RPC request ids 1/2/3 are magic literals duplicated across three places that must stay in lockstep: the sender in run_review (`"id": 1` at line 330, `"id": 2` at line 342, `"id": 3` at line 347), the dispatcher in McpClient._reader (`== 1`, `== 2`, `== 3` at lines 290/292/299), and the self-test fixtures (`"id": 3` at lines 49, 90, 99). If a handshake step is added or an id renumbered on the send side, the reader silently never matches and the harness hangs until the 1800s timeout instead of failing loudly. Name the ids once at module top (e.g. `INITIALIZE_ID = 1`, `TOOLS_LIST_ID = 2`, `REVIEW_CALL_ID = 3`) and use the constants in run_review, _reader, and _response so sender, dispatcher, and fixtures cannot drift. RESOLVED: `INITIALIZE_ID`/`TOOLS_LIST_ID`/`REVIEW_CALL_ID` module constants used by the sender (drive_review), the reader's id-keyed event map, and the self-test fixtures.
- [x] `scripts/review-verify/drive.py:306` — run_review is ~66 lines of actual code (drive.py:306-384, excluding docstring and blanks), exceeding the ~50-line limit. Name the delay once (e.g. `HANDSHAKE_SETTLE_SECONDS = 2`) and use it for all three pauses, mirroring the existing REVIEW_TIMEOUT_SECONDS convention. RESOLVED (superseded by the response-sequenced handshake below): all sleeps are gone; the one named timing knob is `HANDSHAKE_STEP_TIMEOUT_SECONDS = 120` alongside `REVIEW_TIMEOUT_SECONDS`.
- [x] `scripts/review-verify/drive.py:339` — The MCP handshake is sequenced with fixed time.sleep() calls instead of waiting for each response, so protocol-ordering errors are masked rather than handled. If `sah serve --model qwen` takes more than 2s to answer `initialize` (model load makes this likely on a cold start), the client sends `notifications/initialized`, `tools/list`, and `tools/call` before initialization completes — a server that rejects pre-initialization requests makes the harness fail with the misleading 'no review result within 1800s' message instead of a clear handshake error. RESOLVED: `_reader` now records responses and sets a per-id Event for all three request ids; `drive_review` waits on the initialize response before `notifications/initialized`, and on the tools/list response before `tools/call`, each with `HANDSHAKE_STEP_TIMEOUT_SECONDS` and a step-named failure message ("no initialize response within Ns" / "no tools/list response within Ns").

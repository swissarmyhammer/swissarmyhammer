## What
llama-agent owns the loop, so it can deliver diagnostics two ways: (1) the write op's result already feeds straight back into model context for free (`agent.rs` appends `ToolResult.result` JSON to the conversation) — no work needed; and (2) forward the same diagnostics as an ACP `session/update` to the editor's native gutter (e.g. Xcode). This task does (2).

- After a mutating tool call returns inline diagnostics (from the fold-in step), llama-agent broadcasts a diagnostics `SessionNotification` over the existing `broadcast::Sender<SessionNotification>` (`acp/server.rs:59,320`; builders in `acp/translation.rs:611`). Add a diagnostics-carrying `SessionUpdate` path — either a dedicated variant if the ACP schema supports one, or structured diagnostics in the existing notification envelope — so the editor can render gutter markers.
- Mirror the existing plan/mode notification pattern (`send_plan_notification_from_result` extracts `_plan` from a tool result and broadcasts) — extract diagnostics from the tool result similarly and broadcast.

## Depends on
- "Inline-on-edit: mutated_paths + shared diagnostics fold-in step"

## Acceptance Criteria
- [ ] After an instrumented edit, llama-agent emits a diagnostics `session/update` over the broadcast channel carrying the structured report.
- [ ] The model-context path is verified to already carry diagnostics (no regression) — diagnostics in `ToolResult.result` reach the conversation.
- [ ] No second analysis: the ACP forward reuses the report already in the tool result.

## Tests
- [ ] `cargo test -p llama-agent`: a fake tool result containing a diagnostics field drives `generate()`/the broadcast path; assert a diagnostics `SessionNotification` is published AND the tool result is appended to the session messages. Use the scripted fake-model seam (no GPU/weights), <10s.

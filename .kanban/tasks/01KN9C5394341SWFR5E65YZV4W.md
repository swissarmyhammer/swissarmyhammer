## What

When `perspective.save` creates a perspective file in `.kanban/perspectives/`, the `PerspectiveProvider` never sees it because the `entity-created` event either never fires or fires too late. The frontend's `refresh()` is never triggered, so the tab bar stays empty.

### Root cause investigation

The command dispatch pipeline in `kanban-app/src/commands.rs` calls `flush_and_emit_for_handle()` after every undoable command. This function calls `store_context.flush_all()` which scans registered stores for changes and emits events. The question is: **is the perspectives store registered in `StoreContext`?**

Check these files:
- `kanban-app/src/state.rs` ‚Äî where stores are registered in `StoreContext` during `BoardHandle::open()`. Is the perspective store registered here?
- `kanban-app/src/commands.rs` ‚Äî `flush_and_emit_for_handle()` ‚Äî does it handle perspective store events?
- `swissarmyhammer-store/src/context.rs` ‚Äî `flush_all()` ‚Äî does it see the perspective store?

If perspectives are NOT registered as a `TrackedStore`, then changes rely on the filesystem watcher's polling (debounced, async) ‚Äî which explains the broken feedback loop. The fix is to register the perspective store in `StoreContext` alongside entity stores.

### Files to investigate/modify
- `kanban-app/src/state.rs` ‚Äî register perspective store in `StoreContext`
- `kanban-app/src/commands.rs` ‚Äî ensure `flush_and_emit_for_handle` processes perspective change events
- `swissarmyhammer-kanban/src/context.rs` or equivalent ‚Äî wherever `PerspectiveContext` is created

### What success looks like
After `perspective.save` completes, `flush_all()` detects the new file, emits `entity-created` with `entity_type: \"perspective\"`, and `PerspectiveProvider.refresh()` fires ‚Äî the new perspective appears in the tab bar immediately.

## How to test the full feedback loop

**Do NOT test layers in isolation and assume the loop works.** The whole point of this card is proving the loop end-to-end. Specifically:

### Rust integration test (the critical one)
Write a test that:
1. Creates a `StoreContext` with the perspective store registered
2. Calls `perspective.save` (or the underlying `AddPerspective` operation) to create a perspective
3. Calls `store_context.flush_all()` immediately after
4. Asserts that `flush_all()` returns a `ChangeEvent` with `event_name: \"item-created\"` and `store: \"perspectives\"` (or equivalent)
5. This proves the store ‚Üí event pipeline works for perspectives specifically, not just entities

### Rust regression test
Write a test that:
1. Calls `perspective.filter` (or any update command) on an existing perspective
2. Calls `flush_all()`
3. Asserts `item-changed` event with correct store and ID

### Frontend verification
The existing `PerspectiveProvider` tests already verify that mocked `entity-created` events trigger refresh. That layer is covered. The gap is the backend producing those events ‚Äî which is what the Rust tests above prove.

### Manual smoke test (do this yourself, don't ask the user)
After implementing, use the OS log to verify events fire:
```
log show --predicate 'subsystem == \"com.swissarmyhammer.kanban\"' --last 30s
```
Look for perspective-related event emissions after running a perspective command.

## What

`review working` baselines against `HEAD`, but in a `/finish` fix-loop `HEAD` never moves (changes aren't committed between passes), so every re-review re-scopes the **whole** uncommitted change set × every validator — a fresh ~10-min full-tree pass that re-finds nits on files the latest fix never touched. This is the dominant review-churn cost observed in the calcutron qwen eval.

Fix: make `review working` **incremental**, enforced inside the review tool/engine (NOT in the `/finish` skill) so every caller — the loop, the CLI, parallel subagents — benefits and shares state. Track, per reviewed file, a content+rules **context hash** on disk under the `.validators/` directory; when resolving working changes, subtract files whose context hash is unchanged since they were last reviewed.

### Directory layout (`.validators/`)
`.validators/` is the project validators directory and MAY hold **committed** project rule config — so it must NOT be blanket-ignored at the repo root. The ephemeral hash tracking lives in the **`.hashes/`** subdirectory (source tree mirrored under it), and a `.gitignore` *inside* `.validators/` ignores just that subdirectory:

```
.validators/
  .gitignore            # generated by review init: ignores `.hashes/`
  .hashes/              # ephemeral per-file context-hash tracking (gitignored), source tree mirrored
    src/error.rs.yaml
    src/parser.rs.yaml
  <committed project rule files stay tracked>
```

- **`review`'s init generates `.validators/.gitignore`** with a line ignoring `.hashes/`. Committed project validator config under `.validators/` stays tracked. Do NOT add `.validators/` to the repo-root `.gitignore`.
- If init hasn't run, the tracking writer **lazily ensures `.validators/.gitignore` exists** (create-if-missing) before writing entries, so the hash dir is never accidentally committed.

### Tracking entries (`.validators/.hashes/`)
- One **YAML** entry per reviewed source file (NOT JSON, NOT a shared manifest), path-mirrored for debuggability: `src/error.rs` → `.validators/.hashes/src/error.rs.yaml`.
- Entry contents — plaintext + component hashes, so "why did this re-review?" is eyeball-able (confirmed desired):
  ```yaml
  path: src/error.rs
  context_hash: blake3:<combined>
  content_hash: blake3:<file content>
  rules_hash: blake3:<rules>
  reviewed_at: 2026-06-14T18:40:00Z
  ```
- `context_hash = H(relative_path ‖ file_content ‖ rules_hash)`. Changing the path, the content, OR the rules changes the context hash ⇒ the file is re-reviewed.
- **`rules_hash` is global for v1** — a hash of all loaded validator rule bodies. (Per-file applicable-rules hashing is a later refinement; a rule edit forcing a full re-sweep is acceptable.)
- **Atomic writes** (write temp + rename) so concurrent `sah` processes (subagents) never tear each other's state. Concurrent writers compute the same hash and write the same value — idempotent; races only ever cause a redundant re-review, never incorrectness.

### Flow changes
- `crates/swissarmyhammer-validators/src/review/scope.rs` — after `resolve_working` builds the candidate set (all changed + untracked vs HEAD, unchanged), apply a new tracking filter: for each candidate, compute its current `context_hash` and **subtract** it if `.validators/.hashes/<path>.yaml` exists with an equal `context_hash`. Per-file `before` stays HEAD (so the semantic diff still shows the real change, not just the last fix) — the tracking only decides *inclusion*.
- After a review **completes successfully**, upsert the tracking entry for every reviewed file — **regardless of findings**. Open findings persist in the kanban task checklist; an unchanged file with an open finding is correctly NOT re-scanned (it can't have been fixed, and re-scanning only re-derives the same finding). When the implementer edits it, content changes ⇒ hash changes ⇒ re-review.
- **Zero survivors ⇒ short-circuit to a clean/empty report** without spinning up the fleet (no prime/fork cost for a no-op pass).
- **Force-full escape hatch**: `review working` honors a `force`/`all` signal (and/or a cleared `.validators/.hashes/`) to ignore tracking and review the whole set. Document that `review working` is now default-narrow once a baseline exists.

### Cross-file duplication safety
Subtracting unchanged files from the review target set stays safe: the duplicates probe runs over the whole-repo corpus regardless of scope, and a change can only *introduce* a dup in the file it changed — which survives the subtraction. A dup purely between two unchanged files was already caught when they were last changed.

Files to modify/create:
- `crates/swissarmyhammer-validators/src/review/` — new tracking module (e.g. `tracking.rs`): `context_hash`, read/subtract, atomic upsert, path-mirrored YAML under `.validators/.hashes/`, and ensure-`.validators/.gitignore`.
- `crates/swissarmyhammer-validators/src/review/scope.rs` — wire the subtract filter into `resolve_working`; thread a `use_tracking`/`force` flag.
- `crates/swissarmyhammer-tools/src/mcp/tools/review/review_op.rs` — record reviewed files after a successful review; pass the force flag through.
- review init path (where the project `.validators/` is set up) — generate `.validators/.gitignore` ignoring `.hashes/`.

## Acceptance Criteria
- [ ] `review` init generates `.validators/.gitignore` that ignores `.hashes/`; committed config under `.validators/` stays tracked; `.validators/` is NOT added to the repo-root `.gitignore`.
- [ ] If init hasn't run, writing tracking lazily creates `.validators/.gitignore` before the first entry.
- [ ] After a `review working` pass, `.validators/.hashes/<path>.yaml` entries exist for every reviewed file with plaintext path + `context_hash`/`content_hash`/`rules_hash`/`reviewed_at`.
- [ ] A second `review working` with **no file changes** subtracts everything and short-circuits to a clean/empty report without running the fleet.
- [ ] Editing one file changes only that file's `context_hash`; the next `review working` reviews **only** that file (others subtracted).
- [ ] Changing the rules (global `rules_hash`) invalidates all entries ⇒ next `review working` reviews the full set.
- [ ] A file with an open finding that is left unchanged is NOT re-reviewed on the next pass (its entry was written despite the finding).
- [ ] The force/all hatch reviews the full set ignoring tracking.
- [ ] Entries are written atomically (temp + rename); format is YAML.

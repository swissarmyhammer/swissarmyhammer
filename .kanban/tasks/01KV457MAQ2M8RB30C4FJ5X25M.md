## Symptom
`sah` review tool should default to `claude-code-haiku` (spawn `claude --model haiku`) on an unconfigured project, but it actually runs plain `claude` (no `--model`). qwen/llama review works; the Claude haiku tier silently does not.

## Root cause (verified end-to-end)
The spawn seam and the `ModelManager` resolver family are BOTH correct and already deliver `["--model","haiku"]` when handed the `claude-code-haiku` ModelConfig. The bug is upstream, in how the serve wiring picks the review model NAME.

`apps/swissarmyhammer-cli/src/commands/serve/mod.rs::review_model_config` resolves the name from the **template context**, not the config files:
```
let review_model  = cli_context.template_context.get("review.model")...;  // None when unset
let default_model = cli_context.template_context.get("model")...;          // <-- NOT None in prod
let model_name = ModelManager::review_agent_name_from(review_model, default_model);
```
`review_agent_name_from(a,b) = a.or(b).unwrap_or(REVIEW_DEFAULT_AGENT)`, so the `claude-code-haiku` default is only reached when BOTH inputs are `None`.

But in production the top-level `model` template var is **never None**: `TemplateContext::set_default_variables()` → `set_model_variable()` injects the literal `"claude"` (`resolve_model_name()` returns `"claude"`, template_context.rs ~666/671) whenever `model` is unset. Production builds this context via `main.rs` → `load_cli_configuration()` → `TemplateContext::load_for_cli()` (which calls `set_default_variables()`), then hands it to `CliContext`.

So for an unconfigured project the serve path computes:
- `default_model = Some("claude")` → `model_name = "claude"` (NOT `claude-code-haiku`).
- `find_agent_by_name("claude")` → NotFound (only `claude-code` and `claude-code-haiku` builtins exist) → `review_model_config` returns `None` (logs "review model 'claude' could not be resolved").
- `wire_review_factories` then falls back to the global `tool_context.agent_config` = `ModelConfig::claude_code()` = plain `claude-code`, **no `--model`**.

Net: review runs full Claude, never haiku. The `{{ model }}` Liquid prompt default ("claude") is leaking into agent/model SELECTION.

## Why the existing test didn't catch it (test masks prod)
`commands::serve::tests::test_review_model_config_defaults_to_haiku_when_unset` is GREEN but builds its CliContext from a raw `TemplateContext::new()` (helper `cli_context_with_models`) and **never calls `set_default_variables()`**. So in-test `get("model")` is `None` → resolves to haiku. The fixture diverges from production, which always runs `set_default_variables()`. Classic fixture-only masking (see project memory `real-path-tests-not-mocks`, `fixture-only-anti-pattern`).

## Fix (minimal, correct)
Make review model selection read CONFIG FILES via the canonical resolver, not the prompt-rendering template vars:
- In `review_model_config` (serve/mod.rs), resolve the name via `ModelManager::resolve_review_agent_name(&ModelPaths::sah())` and the config via `ModelManager::resolve_review_agent_config(&ModelPaths::sah())` (these read `review.model` / top-level `model:` from the `.sah` config through `get_review_agent`/`get_agent`, which correctly return `None` when unset → fall through to `claude-code-haiku`). Stop reading `template_context.get("model")` / `get("review.model")` for selection entirely.
- Keep the existing decision-point `tracing::info!("review scope → …")` log.
- Document `resolve_model_name`/`set_model_variable` as PROMPT-RENDERING ONLY — never agent selection.

This is the correct subset of sibling task `f7wcj2r`'s item #2. NOTE: `f7wcj2r` mislabels its item #1 (the `server.rs:610` `template_context.get_agent_config(None)` error-branch fallback for the GLOBAL agent) as "the actual bug" — it is NOT: that branch only fires on a resolver ERROR and feeds the global default, and fixing it alone leaves review still non-haiku. The real trigger is `review_model_config` consuming the defaulted `model="claude"` template var. The `server.rs:610` cleanup and the broader `get_agent_config` removal are reasonable hygiene but are NOT the review-haiku root cause and should not be conflated with it.

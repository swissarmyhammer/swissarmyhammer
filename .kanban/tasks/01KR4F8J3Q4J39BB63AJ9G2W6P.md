## What

The TS port of `generate_sneak_codes` in `kanban-app/ui/src/spatial-nav-jump-to.spatial.test.tsx` (`generateSneakCodesPort`, around the `SNEAK_ALPHABET` constant) is a hand-written re-implementation of `swissarmyhammer-focus/src/sneak.rs`. The two implementations have no compile-time link, so a future change to either side (alphabet ordering, `pick_two_letter_prefix_count` formula, single -> two-letter fill order) can drift silently â€” the overlay would dispatch focus to the wrong scope at runtime without any test catching it.

This task replaces the hand-port with one of:

(a) **Fixture file approach** â€” emit a `sneak-fixture.json` from a Rust `#[test]` in `swissarmyhammer-focus` (e.g. `tests/sneak_fixture.rs`) that runs `generate_sneak_codes(n)` over a representative count set and writes the result. The TS test reads the fixture and asserts the TS port produces the same output. Either keep both sides and assert equality, or drop the TS port and have the test consume the fixture directly.

(b) **Property-style test** â€” exercise the TS port across `[1, 5, 23, 24, 50, 200, 529]` and assert shape invariants:
- length matches `count`
- all codes distinct
- prefix-free (no code is a prefix of another)
- alphabet only contains the 23 letters in `SNEAK_ALPHABET`
- single-letter codes appear before two-letter codes
- single-letter codes are in alphabet order
Add a Rust counterpart that asserts the same invariants, so any future divergence in either side surfaces as a test failure.

## Review Findings (2026-05-08 15:38)

### Warnings
- [x] `swissarmyhammer-focus/tests/sneak_fixture.rs:97-127` â€” A bare `#[test]` always writes to `kanban-app/ui/src/test/fixtures/sneak-fixture.json` whenever `cargo test` is run. That means a developer running the workspace test suite locally with a clean working tree can end up with a dirty tree that they didn't intend to modify (e.g. cross-platform line-ending differences, an in-flight `git stash`, or a partially staged commit). Why it matters: tests should be observers, not mutators of source-controlled files. Suggested fix: gate the write behind an env var (`SAH_BLESS_FIXTURE=1` or `BLESS=1`) and have the un-blessed path read the file and assert its contents match what `generate_sneak_codes` would produce â€” that turns the same test into a verifier in normal runs and a regenerator on demand. Bonus: it removes the dependency on a separate CI `git diff --exit-code` step, since the un-blessed test fails directly when the fixture is stale.

  Resolved: Adopted the `BLESS=1`-gated pattern. The default test path (`cargo test -p swissarmyhammer-focus --test sneak_fixture`) is now read-only â€” it loads the fixture from disk, computes the expected JSON via `generate_sneak_codes` over the same count set, and asserts byte-for-byte equality. On mismatch the failure message names the file and gives the exact `BLESS=1 cargo test ...` regen command. The write path runs only when `std::env::var("BLESS").is_ok()`.

- [x] `swissarmyhammer-focus/tests/sneak_fixture.rs:18-23` â€” Drift detection is documented as "CI runs the test and then checks the working tree is clean (`git diff --exit-code`)". This PR does not include or modify any CI workflow that enforces that gate. Why it matters: without the gate the fixture can land out-of-sync with the algorithm and no test catches it â€” the whole drift-prevention story rests on a check that may not exist. Suggested fix: either (a) add a CI step that runs `cargo test -p swissarmyhammer-focus --test sneak_fixture` followed by `git diff --exit-code -- kanban-app/ui/src/test/fixtures/sneak-fixture.json`, or (b) adopt the BLESS-gated pattern from the previous finding so the test itself fails on drift without needing CI plumbing.

  Resolved: Took option (b). The same code change above replaces the "CI runs `git diff --exit-code`" story with a self-contained drift check â€” the un-blessed test asserts the fixture matches the algorithm. The module-level doc comment in `sneak_fixture.rs` and the `SOURCE OF TRUTH` comment block in `spatial-nav-jump-to.spatial.test.tsx` were both rewritten to describe the new flow. Verified by corrupting the fixture: the test fails with "fixture ... is stale; regenerate with: BLESS=1 cargo test ...".

### Nits
- [x] `kanban-app/ui/src/spatial-nav-jump-to.spatial.test.tsx:300-303` â€” `lookupSneakCodes((a.count as number) ?? 0)` falls back to count=0 when `args.count` is missing. The fixture has `"0": []` so this returns an empty array silently. If a future refactor accidentally drops the `count` arg from the production `generate_jump_codes` call, the test would still pass against an empty code list rather than failing loudly. Suggested fix: throw if `count` is missing/non-numeric â€” the IPC contract requires it, and an explicit failure here is more useful than a silent empty.

  Resolved: Replaced the `?? 0` fallback with an explicit `typeof a.count !== "number" || !Number.isFinite(a.count)` guard that throws with a message naming the offending type/value. A future refactor that drops the `count` arg now surfaces as a loud test failure rather than a silent empty-list match.

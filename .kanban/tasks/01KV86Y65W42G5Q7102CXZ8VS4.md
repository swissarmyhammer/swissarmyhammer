## What
Two pieces that keep the shared view consistent across processes:

1. **Cross-process diagnostics fan-out** — extend the in-process fan-out so the leader re-publishes `publishDiagnostics` to followers over the existing ZMQ pub/sub bus (`Publisher`/`Subscriber`/`BusMessage` in `swissarmyhammer-leader-election`). Implement `BusMessage` for a `{uri, Vec<DiagnosticRecord>}` payload; followers subscribe and receive updates.
2. **Leader watcher** — a continuous file-change task in the leader process (one watcher per workdir): file change → debounce → `LspSession::change` (`didChange`) → diagnostics. This catches a follower's direct `files edit` write (the watcher issues `didChange` so followers need not sync the analyzer themselves) and the leaks the closed write surface cannot cover (subprocesses, formatters, `git checkout`).

## Depends on
- "Capture publishDiagnostics and add in-process fan-out in swissarmyhammer-lsp"
- "Leader-per-workdir: only the leader spawns the LSP session"

## Acceptance Criteria
- [ ] Leader re-publishes diagnostics over the ZMQ bus via a `BusMessage` impl; a follower subscriber receives per-uri updates.
- [ ] A single leader-owned watcher debounces file changes and issues `didChange` into the session; one watcher per workdir.
- [ ] A file changed by a follower (or external tool) results in fresh diagnostics without the follower syncing the analyzer.

## Tests
- [ ] `cargo test -p swissarmyhammer-leader-election` (or lsp): `BusMessage` round-trip for the diagnostics payload; publish on leader → assert subscriber receives (model-free).
- [ ] Integration (gated on rust-analyzer): touch a fixture file on disk, assert the watcher debounces and the session re-reports diagnostics for it.

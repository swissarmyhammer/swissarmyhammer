## What

Virtual tag colors are hardcoded in TWO frontend files instead of being served from the backend. The Rust `VirtualTagRegistry` already knows the colors — the frontend should read them from there, not maintain duplicate maps.

### Current duplication

1. **`virtual-tag-display.tsx`** — `VIRTUAL_TAG_META` map with per-tag colors (green/orange/red)
2. **`use-mention-extensions.ts`** — `VIRTUAL_TAG_COLOR` single purple for all virtual tags

Both are wrong approaches. The backend's `VirtualTagRegistry` is the source of truth.

### Fix approach

The backend already enriches task entities with `virtual_tags` (slug list) and `filter_tags` (union of body tags + virtual tags). What's missing: the backend doesn't expose the virtual tag **color** and **description** metadata to the frontend.

**Option A** (quick, pragmatic): Add a `virtual_tag_meta` field to the schema or board data response that maps slug → `{ color, description }`. The `VirtualTagRegistry` already has this data via its strategies. Expose it through `get_board_data` or a new `list_virtual_tags` command.

**Option B** (simpler interim): The `VirtualTagRegistry` evaluates strategies and returns slugs. The colors live in `DEFAULT_REGISTRY` entries. Add a method `VirtualTagRegistry::metadata()` → `Vec<{ slug, color, description }>` and expose it via a Tauri command or as part of `get_board_data`.

Either way, the frontend deletes both hardcoded maps and reads colors from the backend response.

This is larger than a single card — it spans Rust backend changes + frontend consumption. Recommend `/plan` to split into:
1. Backend: expose virtual tag metadata via `get_board_data` or new command
2. Frontend: consume metadata, delete hardcoded maps from both files

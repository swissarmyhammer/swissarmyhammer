## What

Move a plugin's `name` and `version` out of `plugin.json` and into the `Plugin` base class as plugin-author-set TypeScript properties.

- `crates/swissarmyhammer-plugin/src/sdk/plugin.ts` — add `name` and `version` to the `Plugin` abstract base class. They are author-facing metadata a subclass sets (overridable `readonly` fields with sensible defaults, e.g. `name = "unnamed plugin"`, `version = "0.0.0"`, which a subclass overrides). Document in the class doc comment that they are **descriptive metadata only** — the platform does not use them for identity or discovery (a plugin's identity is its bundle directory name) and the SDK never sends them to the host. Keep the existing `Plugin` surface (`load`/`unload`/`register`/`unregister`/`track`/`log`) intact.
- `crates/swissarmyhammer-plugin/src/codegen.rs` — if `TypesEmitter` / the generated `app.d.ts` describes the `Plugin` class surface, regenerate it so `name`/`version` appear. If codegen does not touch the `Plugin` class, no change — confirm and note it.

This task is host-Rust-free and SDK-only; it can land independently of the discovery work.

## Acceptance Criteria
- [ ] `Plugin` exposes `name` and `version`; a subclass can set them; defaults exist so a subclass that omits them still compiles.
- [ ] The class doc comment states `name`/`version` are metadata only and are not used for identity/discovery.
- [ ] `codegen.rs` / `app.d.ts` reflects the new props, or it is confirmed codegen does not cover the `Plugin` class.

## Tests
- [ ] Extend `crates/swissarmyhammer-plugin/tests/sdk.rs` (the SDK transpile/dispatch test): a `Plugin` subclass that sets `name`/`version` transpiles cleanly and the properties are readable on the constructed instance.
- [ ] If `codegen.rs` has a test asserting `app.d.ts` content, update it to expect `name`/`version`.
- [ ] Run `cargo nextest run -p swissarmyhammer-plugin --test sdk` and any `codegen` test — pass.
- [ ] `cargo clippy -p swissarmyhammer-plugin --all-targets -- -D warnings` — clean.

**File:** `swissarmyhammer-kanban-app/src/commands.rs`, lines 405-427 (the `update_entity_field` function)\n\n**What:** The `update_entity_field` Tauri command accepts an arbitrary `field_name` string from the frontend and writes it directly to the entity via `entity.set(&field_name, value)`. There is no validation that `field_name` is a declared field in the entity's FieldDef schema.\n\n**Why:** This means the UI (or a crafted IPC call) can write any field name to any entity -- including fields that don't exist in the schema, system-internal fields, or fields belonging to other entity types. While the `EntityContext::write` path does validate against the EntityDef, an unknown field would be silently set in the HashMap and then either ignored or cause unexpected behavior during serialization. At minimum, the command should verify the field_name is declared in the entity's schema before allowing the write.\n\n**Suggestion:** After resolving the FieldsContext, check that the field_name exists in the entity's field list:\n```rust\nlet fields_ctx = ectx.fields();\nlet entity_def = fields_ctx.get_entity(&entity_type)\n    .ok_or_else(|| format!(\"Unknown entity type: {}\", entity_type))?;\nif !entity_def.fields.contains(&field_name) {\n    return Err(format!(\"Field '{}' is not defined for entity type '{}'\", field_name, entity_type));\n}\n```\n\n- [ ] Add field-name validation against the EntityDef before writing\n- [ ] Return a clear error message when an invalid field name is provided\n- [ ] Add a test (or integration test) that verifies unknown field names are rejected" #warning

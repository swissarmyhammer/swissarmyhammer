## What

Clean break: all undo/redo logic moves to the store crate via text diffs. Strip old field-level changelog, undo/redo, and transaction indexing from EntityContext. Migrate old per-entity `.jsonl` changelogs to new format (single synthetic create entry). Wire StoreContext into the Tauri dispatch layer. **Stop here and manually test do/undo/redo before proceeding.**

### Key principles
- Data format (YAML / MD+YAML) stays the same — only changelogs change
- Undo/redo works via text diffs on the serialized representation
- On undo/redo, the store returns the full `Item` so the UI can swap it in
- Old changelogs get migrated: detect old format, replace with a single synthetic "create" entry using the oldest log date + current file content
- Transactions wire through StoreHandle

## Changes

### 1. Migrate old changelogs on detection
When a `StoreHandle` first accesses a per-item changelog (`{id}.jsonl`):
- Read the first line. If it has the old format (contains `"changes"` field with `FieldChange` variants like `Set`, `Removed`, `Changed`, `TextDiff`) → it's old format
- Migration: read the current item file, compute a unified diff from "" → current content, write a single synthetic `ChangelogEntry` with `op: Create`, `timestamp` from the oldest entry in the old file, forward/reverse patches from "" ↔ current content
- Replace the old `.jsonl` with just this one entry
- This preserves the creation timestamp while giving us a clean new-format changelog

### 3. Wire transactions through StoreHandle
EntityContext write path currently groups writes under a transaction. Replace with StoreHandle transactions:
- `EntityContext` exposes `begin_transaction()` / `commit_transaction()` that delegate to StoreHandle
- Or: the kanban `KanbanOperationProcessor` calls StoreContext transaction methods directly
- Each operation (task.add, task.move, etc.) that writes multiple entities wraps them in a store transaction
- Undo of a transaction reverses all writes in reverse order (StoreHandle already supports this)

### 4. Wire UndoCmd/RedoCmd through StoreContext
- `swissarmyhammer-entity/src/undo_commands.rs`: `UndoCmd`/`RedoCmd` call `store_context.undo()`/`redo()` instead of `EntityContext::undo()`
- `StoreContext` returns the full restored `Item` (via `undo_erased` returning `Result<()>` — but flush_changes produces the events for the UI)
- Actually: `StoreContext::undo()` applies the reverse patch, writes the file, then `flush_all()` detects the change and emits events with the full item payload

### 6. EntityContext::write() simplification
After cleanup:
```rust
pub async fn write(&self, entity: &Entity) -> Result<Option<UndoEntryId>> {
    // Validate, strip computed fields, apply defaults
    let entity = self.validate_for_write(entity).await?;

    // Delegate to StoreHandle
    let store_handle = self.store_handle_for(&entity.entity_type)?;
    let entry_id = store_handle.write(&entity).await?;

    Ok(entry_id)
}
```

### 7. EntityContext::delete() simplification
```rust
pub async fn delete(&self, entity_type: &str, id: &str) -> Result<UndoEntryId> {
    let store_handle = self.store_handle_for(entity_type)?;
    let entity_id = EntityId::from(id);
    let entry_id = store_handle.delete(&entity_id).await?;
    Ok(entry_id)
}
```

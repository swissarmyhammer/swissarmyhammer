#elicitation

## Context / Why
`apps/kanban-app/ui/src/ai/conversation.ts` (`useConversation`) already owns the analogous **permission** seam: it stores a pending `RequestPermissionRequest`, exposes `permissionRequest` + `respondPermission`, and provides `onRequestPermission` to the ACP client via a promise that `respondPermission` resolves (see `handleRequestPermission` / `permissionResolverRef` / `respondPermission`). This task adds the exact same plumbing for elicitation, consuming the `onElicitation` / `onCompleteElicitation` seams added to the ACP client.

## What
In `apps/kanban-app/ui/src/ai/conversation.ts`:
- [ ] Add `elicitationRequest: CreateElicitationRequest | null` to `ConversationApi` and `respondElicitation(response: CreateElicitationResponse): void`.
- [ ] Add `onElicitation` (returns a `Promise<CreateElicitationResponse>` resolved by `respondElicitation`) and `onCompleteElicitation` (clears the pending request) to `ConversationHandlers`; thread both through the `connect` factory call (so `aiPanelConnectFactory` passes them to `createKanbanClient`).
- [ ] Mirror the permission impl exactly: `elicitationResolverRef`, `setElicitationRequest`, a memoized `handleElicitation`, and inclusion in the `handlers` `useMemo` + the returned `ConversationApi` `useMemo` deps.
- [ ] `newConversation()` and `cancel()` must clear any pending elicitation + resolver (treat an abandoned elicitation as cancelled), matching how permission state is reset.
- [ ] Update `ConversationConnect`/`UseConversationOptions` docs to mention the elicitation handlers.

## Acceptance Criteria
- [ ] `onElicitation` stores the request and returns a promise; `respondElicitation` resolves it and clears `elicitationRequest`.
- [ ] `onCompleteElicitation` clears the pending request (agent-side cancellation dismisses the UI).
- [ ] `newConversation` clears pending elicitation without leaking the resolver.
- [ ] Permission behavior is unchanged.

**Untested critical path.** When a validator with `severity: error` returns `{"status": "failed", ...}`, avp is supposed to:

1. Mark the rule as a blocking failure in `ExecutedRuleSet::blocking_failures()`.
2. `validator_executor.rs::handle_ruleset_results` (around line 142) returns `ChainResult::stop(LinkOutput::from_validator_block(...))`.
3. The chain stops processing.
4. avp's main exits with code `VALIDATOR_BLOCK_EXIT_CODE` (2).
5. claude-code interprets exit code 2 as "block this Stop and surface the message to the user."

This whole path has never been exercised end-to-end. Until today every Stop-hook validation passed (because of the missing-content bug). Once `01KQ7EQFSMMYW80HC8PQZ2YATV`, `01KQ35MHFJQPMEKQ08PZKBKFY0`, and `01KQ7KYNBEHQGEMGND4AEG9EV6` land and validators can actually see and read code, the *first time* a real finding lands the block-on-error path will run for the first time. Easy to break and we'd find out the hard way — at the worst possible moment (user trying to ship code, gets a confusing hook failure).

## What was tested

### 1. Unit: `handle_ruleset_results` produces the right `ChainResult` for an error-severity failure ✅

In `avp-common/src/chain/links/validator_executor.rs::tests`:
- Single failed error-severity rule on a Stop hook → `ChainResult::Stop` with qualified rule name + message.
- Multiple failed rules across rulesets → first ruleset's first blocking failure wins.
- Failed rule alongside passing rules in the same RuleSet → the failure is still surfaced.
- Warn-severity failure → does NOT block.
- Empty / all-pass results → `ChainResult::Continue`.
- Stderr-only hooks (`SessionStart`, `ConfigChange`, etc.) → `ctx.exit_code()` set to `VALIDATOR_BLOCK_EXIT_CODE`.
- Stdout-style hooks (`Stop`, `PreToolUse`, etc.) → `ctx.exit_code()` left at default 0.
- Round-trip coverage: every hook in `hook_uses_stderr_only_failure_surface` flips the code; every stdout-style hook leaves it alone.

### 2. Integration: full hook → validator → chain → exit code ✅

`avp-common/tests/validator_block_e2e_integration.rs::stop_hook_error_severity_failure_blocks_chain_with_exit_code_2`:
- Drives a `PlaybackAgent` (loaded from `tests/fixtures/recordings/rule_magic_number_fail.json`) through the production Stop chain.
- Asserts `validator_block` is set with `validator_name == "<ruleset>:<rule>"`, non-empty failure message, `hook_type == Stop`.
- Asserts `continue_execution == false` and `stop_reason` mirrors the validator message.
- Asserts the chain's reported exit code is `VALIDATOR_BLOCK_EXIT_CODE` (2). This is the load-bearing exit-code contract that makes the Stop hook block claude-code in production.

The test does NOT assert on the literal recorded message text round-tripping — `PlaybackAgent`'s notification bridge is async and can race the runner's response collector, so the runner sometimes sees an empty response and reports "Validator returned empty response". Either outcome is a blocking failure with a non-empty message, which is the contract this test pins down. The PlaybackAgent notification timing is a separate concern from the block path.

### 3. Strategy-render unit test ✅

`avp-common/src/strategy/claude/strategy.rs::tests::block_stop_renders_claude_parseable_json`:
- Builds a `ChainOutput` in memory with `validator_block` set, runs it through `transform_to_claude_output` directly.
- Asserts the rendered `HookOutput` matches claude-code's Stop-block protocol: `continue: true`, `decision: "block"`, `reason` and `stopReason` set to the same string containing both the rule name and the finding, strategy exit code 0.
- Round-trips to JSON to confirm camelCase keys (`continue`, `decision`, `reason`, `stopReason`) match what claude-code parses on stdout.

This rendering is a pure function of `ChainOutput`, so a hand-built input gives a deterministic test without `PlaybackAgent` flakiness or builtin-validator interference. The chain → validator_block half lives in the integration test above.

### Manual smoke test — INTENTIONALLY SKIPPED

Per project policy (commit b4141da8b: "docs(task-standards): forbid manual/smoke tests"), the originally-described "Manual end-to-end" check (run real qwen against `sample_avp_test.rs` and watch claude-code surface the block) has been omitted. The two automated tests above (chain integration + strategy render) cover the same contract end-to-end without depending on a live model run.

## Why this is a separate card

It's a different layer than the validator-execution work. The validator runs through the runner and produces verdicts. This card validates that downstream of the runner — the chain decision, the exit code, the hook output format — actually causes claude-code to block. That's chain-link and main-loop territory, distinct from the per-rule session machinery the other tasks touch. #avp

## Review Findings (2026-04-27 15:30)

### Warnings
- [x] `avp-common/tests/validator_block_e2e_integration.rs:60-140` — Substantial duplication of test setup between this file and `avp-common/tests/recording_replay_integration.rs` (`playback_context`, ruleset-on-disk write helpers, `StopInput` builder, CLAUDE_ACP env guarding).

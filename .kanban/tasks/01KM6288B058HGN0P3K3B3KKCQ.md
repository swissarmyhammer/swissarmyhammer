## What

Test and define the behavior when archiving a task that blocks other tasks. This is a design decision that needs a test to lock it in.

### Scenario
- Task A exists (blocker)
- Task B depends on Task A
- Archive Task A
- What is Task B's `ready` status?

### Options (pick one and test it)

**Option A — Archive is silent removal:** Task B becomes ready because its dep is no longer in the live task list. This is the natural behavior of directory-based archive — `task_is_ready()` checks `all_tasks` from `list()`, and archived tasks aren't in `list()`. Simple, but surprising.

**Option B — Archive cleans deps:** Archiving Task A also removes it from Task B's `depends_on` list (same as `DeleteTask` does). Task B becomes ready explicitly.

**Option C — Archive is blocked:** Archiving a task that has dependents fails with an error. User must resolve deps first.

Recommend **Option B** — mirror `DeleteTask` behavior. Archive should clean up deps the same way delete does. The difference is just where the files go.

### Files
- `swissarmyhammer-kanban/src/task/` or `swissarmyhammer-entity/src/context.rs` — depending on where dep cleanup happens
- Test file for the chosen approach

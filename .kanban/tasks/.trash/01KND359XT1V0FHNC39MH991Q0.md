## What

`TurnStateManager.content_cache` is a `Mutex<HashMap>` that lives in-memory. PreToolUse stashes file content here, PostToolUse reads it to compute diffs. But AVP runs as a **separate process** for each hook invocation — the cache is empty in the PostToolUse process because the PreToolUse process already exited.

Result: `take_content()` always returns `None` in production. `compute_diff()` receives `old_content = None` and produces `--- /dev/null` diffs (full file as new) instead of proper unified diffs showing what changed.

### Evidence from self-test:
1. Created `scratch_test.rs` via Write (new file) → diff correctly shows `--- /dev/null`
2. Edited same file via Edit (changed `x=42` to `x=99`) → diff still shows `--- /dev/null` with full file content instead of a `-/+` diff of the changed line

### Hash-based detection works fine:
The hashes ARE persisted to disk (`turn_state.yaml`), so change detection (did the file change?) works. Only the diff TEXT is degraded.

### Fix options:
1. **Persist pre-content to disk** — write pre-content to a sidecar file in PreToolUse (e.g. `.avp/turn_pre/<session_id>/<encoded_path>.pre`), read it back in PostToolUse. Same pattern as sidecar diffs. Clean up at SessionStart.
2. **Re-read from git** — at PostToolUse time, use `git show HEAD:<path>` to get the pre-edit content. Simpler but only works for tracked files, not new files.
3. **Compute diff from first-seen content** — store the first-seen content (from the first PreToolUse Write that creates the file), never overwrite. Then the diff at Stop time is always \"from original to current\".

Option 1 is most consistent with the existing sidecar pattern.

### Files to modify:
- `avp-common/src/turn/state.rs` — Add `write_pre_content(session_id, tool_use_id, path, content)` and `take_pre_content(session_id, tool_use_id)` methods using sidecar files
- `avp-common/src/chain/links/file_tracker.rs` — PreToolUse writes pre-content to disk, PostToolUse reads from disk instead of in-memory cache

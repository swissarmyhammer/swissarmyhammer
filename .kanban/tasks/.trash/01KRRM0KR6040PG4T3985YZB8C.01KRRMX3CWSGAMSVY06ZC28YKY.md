## What
Enable local llama models behind a Cargo feature so the standard `Kanban.app` bundle stays light (Claude Code only) and does not pay the llama.cpp compile/bundle cost unless opted in.

- Add a `ai-local-models` Cargo feature to `apps/kanban-app/Cargo.toml`, off by default. Gate the `llama-agent` dependency on it.
- Behind the feature: `apps/kanban-app/src/ai/models.rs` enumerates configured llama models (`ModelExecutorType::LlamaAgent`) as selectable entries; `apps/kanban-app/src/ai/session.rs` lets `create_agent` produce a llama-backed `AiSession`. Same MCP URL, same panel, same transport — only the agent backend differs.
- When the feature is off, `ai_list_models` returns Claude Code only and llama models are absent.

Spec: `ideas/kanban/ai_panel.md` — Phase 3 "Build-weight note", Phase 6.

## Acceptance Criteria
- [ ] `apps/kanban-app` builds cleanly both with and without `--features ai-local-models`.
- [ ] With the feature off, `ai_list_models` returns no llama entries.
- [ ] With the feature on, a configured llama model is selectable and a session can be created against `llama-agent` over ACP.
- [ ] The standard release bundle does not enable the feature.

## Tests
- [ ] `cargo build -p kanban-app` and `cargo build -p kanban-app --features ai-local-models` both succeed (add a CI/build-artifacts test asserting both).
- [ ] Unit test: feature off -> enumeration has no llama entries; feature on -> configured llama models appear.
- [ ] Integration test gated on `ai-local-models`: create an `AiSession` for a llama model and assert it runs the ACP `initialize` + `new_session` handshake.
- [ ] `cargo test -p kanban-app` (and `--features ai-local-models`) is green.

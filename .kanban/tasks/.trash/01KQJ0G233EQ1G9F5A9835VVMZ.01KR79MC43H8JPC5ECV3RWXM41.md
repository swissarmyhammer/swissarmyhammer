## What

Pressing **Enter** on a focused card is currently a no-op. The user expects the same drill-in behavior the Inspector provides: card → fields → field edit. Today only the second hop (field → edit) and Inspector's drill-in work. Cards never drill into their own fields, so the user can never reach field-edit from a card.

### Root cause

The drill mechanism is genuinely generic — `buildDrillCommands` in `kanban-app/ui/src/components/app-shell.tsx:322-360` installs a single root-level `nav.drillIn` CommandDef that calls `actions.drillIn(focusedFq, focusedFq)`. Cards fail because of a **registration-shape mismatch**:

| | Inspector (works) | Card (broken) |
|---|---|---|
| Wrapper | `<FocusZone moniker=…>` (`inspectors-container.tsx:332-352`) | `<FocusScope moniker=…>` (`entity-card.tsx:100-120`) |
| Kernel call | `spatial_register_zone` (container) | `spatial_register_scope` (leaf) |
| Field children visible to kernel | yes — FieldRows nest under zone | no — leaf has no descendants; field zones reattach to the column |
| `spatial_drill_in(self)` returns | first field's FQM | `null` (echoes self) — Enter is a no-op |

The card-as-leaf shape was deliberate (see comment at `entity-card.tsx:58-70`) so cross-column kernel navigation treats columns as the only zones. **That decision is the bug.** Making the card a leaf to simplify column-level nav broke per-card drill-in and is exactly the kind of "special case logic" that violates the generic-mechanism contract: Card and Inspector should behave the same because they ARE the same shape (an entity surface with field children).

### Fix direction — make Card a FocusZone

Change `entity-card.tsx:100-120` so the card body wraps in `<FocusZone moniker={asSegment(entity.moniker)}>` instead of `<FocusScope>`. This makes the card a kernel-visible container whose field children are descendants — `spatial_drill_in(card_fq)` then resolves to the first field FQM, identical to Inspector. **No new scope-level shadow command is added.**

### Cross-column navigation implications (must be re-validated, not avoided)

The reason the card was originally a leaf was to keep cross-column navigation behaving correctly when the user presses Left/Right to move between columns. With cards as zones, the kernel sees a nested tree (board > column-zone > card-zone > field-zone). Verify:

1. **Left/Right between columns from a focused card lands on the symmetric position in the sibling column** (e.g. focused on column-2 / card-3 → Right → column-3 / card-3 or column-3 / last-focused). The kernel's existing "remembered position per zone" logic (see `swissarmyhammer-focus/tests/drill.rs::drill_in_after_remembered_position_returns_remembered_fq`) should handle this; if it doesn't when there's an intermediate card-zone layer, the kernel's directional-nav algorithm needs to be adjusted to descend through nested zones for sibling navigation.
2. **Up/Down within a column** still moves between cards. Card-as-zone shouldn't change this — the column zone still contains the cards as direct children.
3. **The kernel test that currently pins card-as-leaf** (`swissarmyhammer-focus/tests/navigate.rs` — search for the test that registers cards as scopes/leaves) must be **rewritten** to register them as zones and assert the same external nav behavior. If the test was protecting an algorithmic shortcut that no longer holds, the kernel may need a corresponding fix.

If the kernel already supports directional nav across nested zones, this is purely a UI change. If not, this task expands to a kernel change and **should be split via `/plan`** before implementing.

### Non-goals

- Do **not** add a `card.drillIn` scope-level shadow command. The fix is structural, not a special case.
- Do **not** touch `nav.drillIn` in `app-shell.tsx` — the generic handler stays generic.
- Do **not** alter `field.edit` — field-level drilling is unchanged.

title: QuickCapture window has no command scope chain -- all actions are direct invoke() calls

**File:** `kanban-app/ui/src/components/quick-capture.tsx`\n\n**Anti-pattern:** The QuickCapture component (rendered in its own Tauri window via QuickCaptureApp in App.tsx) has no CommandScopeProvider, no AppShell, and no command scope chain at all. All user actions are handled via direct `invoke()` calls:\n\n- **Submit** (line 134-139): `invoke(\"dispatch_command\", { cmd: \"task.add\", ... })` -- correct IPC target but no scope chain\n- **Cancel/Escape** (line 185-198): `getCurrentWindow().hide()` -- direct window API call\n- **Board switch** (line 143-149): `invoke(\"dispatch_command\", { cmd: \"file.switchBoard\", ... })` -- direct invoke\n- **Add button** (line 257-265): calls `handleSubmit(draft)` directly\n\n**Provider tree (App.tsx lines 796-813):**\n```tsx\n<SchemaProvider>\n  <EntityStoreProvider entities={{}}>\n    <FieldUpdateProvider>\n      <UIStateProvider>\n        <QuickCapture />\n      </UIStateProvider>\n    </FieldUpdateProvider>\n  </EntityStoreProvider>\n</SchemaProvider>\n```\n\nNotice: no AppShell, no AppModeProvider, no EntityFocusProvider, no InspectProvider, no CommandScopeProvider.\n\n**Correct pattern:** QuickCapture is a secondary window with a minimal UI, so having a full command scope chain may be over-engineering. However, it should at minimum have:\n1. An AppShell (or minimal CommandScopeProvider) so Escape can be handled via `app.dismiss`\n2. Submit should go through a command like `quick-capture.submit`\n\n**Severity:** Warning. The quick capture window is intentionally minimal, but the complete absence of the command architecture means none of its actions are logged, discoverable in a palette, or interceptable. The raw `getCurrentWindow().hide()` calls are particularly problematic -- they bypass any cleanup or state persistence that the command system would handle. #review-finding

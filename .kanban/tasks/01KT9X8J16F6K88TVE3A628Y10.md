title: 'Matcher fidelity: exact-string vs regex with full-match anchoring'

`HookConfig::build_registrations` (crates/agent-client-protocol-extras/src/hook_config.rs) compiles every non-empty, non-`*` matcher as a raw `regex::Regex` and uses `is_match` (unanchored substring). This diverges from Claude Code's documented matcher rules:

- `"*"`, `""`, or omitted → match all.
- Matcher containing only `[A-Za-z0-9_|]` → treated as an exact string, or `|`-separated list of exact strings (e.g. `Edit|Write`). Must be a FULL match, not substring.
- Matcher containing any other character → JavaScript-style regex, anchored to a full match.

Today `matcher: "Bash"` incorrectly matches a tool named `Bash2` or `xBash` (substring). `Edit|Write` happens to work as regex alternation but is unanchored.

## Scope
- Update matcher construction/`HookRegistration::matches` so plain identifier matchers compare for full-string equality (split on `|` for alternation) and regex matchers are anchored (wrap as `^(?:...)$`).
- Keep the `Option<regex::Regex>` storage or introduce a small `Matcher` enum (Exact set vs Regex vs All) — implementer's call; keep it internal to the crate.
- Preserve existing event-without-matcher behavior (UserPromptSubmit/Stop always fire).

## Acceptance criteria + tests
- `Bash` matches only `Bash`, not `Bash2`/`xBash`/`run-Bash`.
- `Edit|Write` matches `Edit` and `Write` exactly, not `Editor`.
- `mcp__memory__.*` (regex) matches `mcp__memory__create_entities`; `^Notebook` matches `NotebookEdit`.
- `*` and empty match everything.
- All existing hook_config tests still pass.

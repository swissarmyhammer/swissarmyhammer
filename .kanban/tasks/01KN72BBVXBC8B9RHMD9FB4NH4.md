kanban-app/src/commands.rs:1409-1470\n\nflush_and_emit_for_handle() calls both store_context.flush_all() (step 1) and watcher::flush_and_emit() (step 2). The StoreHandle cache and the watcher's EntityCache are independent data structures that both track file contents -- StoreHandle uses a HashMap<String, String> of raw text, while EntityCache uses SHA-256 content hashes.\n\nThe problem: after step 1, the StoreHandle cache is updated to the current disk state. But in step 2, the watcher's EntityCache also scans the same files. Since the watcher cache was last updated on the previous flush, it will detect the same changes the StoreHandle already detected -- producing duplicate entity events that are only partially filtered out (only attachments pass through, but the watcher cache is still scanning entity directories unnecessarily).\n\nMore critically, if a command writes through EntityContext (which delegates to StoreHandle), the StoreHandle cache is updated by the write itself. When flush_all() runs, StoreHandle sees no changes (cache matches disk). But the watcher EntityCache was never updated by the write, so it would detect a change if not filtered. The filtering in step 4 correctly discards entity events from the watcher, but the watcher cache itself is not updated with the new content -- it will detect the change on the NEXT flush cycle and silently discard it again. This means the watcher cache is perpetually one cycle behind for entity files.\n\nSuggestion: The watcher EntityCache should be explicitly updated (via update_cache or equivalent) after StoreHandle writes, or the watcher should stop scanning entity directories entirely now that StoreHandle owns that responsibility. The current architecture has two systems tracking the same files independently.",

## What

The review fleet's prompts flow ACP prompt turn (`crates/llama-agent/src/acp/server.rs`, `generate_stream` call ~line 2696) → `AgentServer::generate_stream` (`crates/llama-agent/src/agent.rs` ~1254) → `RequestQueue::submit_streaming_request` (`crates/llama-agent/src/queue.rs`, the `try_send` at ~line 1198), which rejects with `QueueError::Full` when the single-worker queue is saturated — silently dropping fleet tasks. The earlier backpressure fix (commit 470bc0df8) added cancellation-aware bounded `send().await` in `RequestQueue::enqueue_request` (~line 1122) but only routed the NON-streaming `submit_request` through it. Latest real run: 315× "Queue is full", 45/45 fleet tasks failed, empty review.

Fix, all in `crates/llama-agent/src/queue.rs`:

- [x] Make `submit_streaming_request` route its queue submission through `enqueue_request` (the `tokio::select!` on `sender.send()` vs the request's `CancellationToken`) instead of `try_send`.
- [x] On enqueue failure (cancelled or shutdown), remove the session's entry from `active_requests` — the streaming path inserts into `active_requests` (~line 1170) before enqueue and has no `ActiveRequestGuard`, so it would otherwise leak. Prefer reusing `ActiveRequestGuard` the way `submit_request` does.
- [x] Update the now-stale doc comments: `enqueue_request`'s "`Full` is still returned for the unbounded streaming path's non-blocking submit" paragraph, `submit_streaming_request`'s docs, and `try_enqueue_for_test`'s "Mirrors the non-blocking `try_send` the streaming submit path uses" sentence.
- [x] Replace the test `streaming_submit_returns_full_at_capacity` (~line 4170, comment "The streaming submit path has its own try_send + Full branch") — that behavior is the bug.

Do NOT touch the `mpsc::unbounded_channel` for StreamChunks inside `submit_streaming_request` — it is unbounded by design (documented producer-wedge rationale at ~line 1157). Only the queue-submission `try_send` changes. Keep `try_enqueue_for_test` intact for capacity tests (`QueueError::Full` variant stays).

### Warnings
- [x] `crates/llama-agent/src/queue.rs:865` — ActiveRequestGuard::drop removes the session's active_requests entry via a detached spawned task, which races with the next submit on the same session: if turn N's guard drops, then turn N+1 calls track_cancellation_token (queue.rs:1123-1124 inserts unconditionally) before the spawned removal acquires the lock, the stale removal deletes turn N+1's fresh token. cancel_session then returns false and the in-flight turn is uncancellable. Sequential turns on one session — the agentic loop's normal shape — hit exactly this back-to-back pattern. Make the removal conditional on identity: store a monotonically increasing generation id alongside the token (map value `(u64, CancellationToken)`), record the generation in the guard, and have the spawned task remove the entry only if the stored generation still matches the guard's. → FIXED: `active_requests` now maps to `TrackedToken { generation, token }`; `track_cancellation_token` assigns a monotonic generation (`request_generation: AtomicU64` on `RequestQueue`) and returns it; the guard records it and its spawned removal only deletes the entry if the stored generation still matches. Regression test `stale_guard_drop_does_not_clobber_newer_turns_token`.
- [x] `crates/llama-agent/src/queue.rs:1166` — Cancellation — an expected, client-initiated outcome — and shutdown are both surfaced as QueueError::WorkerError(String), and the new doc comments (queue.rs:1071-1073, 1178-1181) codify that conflation. - [x] `crates/llama-agent/src/queue.rs:1234` — After guard.disarm(), nothing in RequestQueue ever removes a streaming session's active_requests entry when the turn completes normally: removal happens only in the guard's Drop (disarmed here) and in cancel_session. Entries for completed streaming turns persist until overwritten by a later submit on the same session, so on the long-lived shared server the map grows with every session that ends without cancellation — the same stale-entry accumulation the guard was introduced to fix — and cancel_session spuriously returns true (and fires a dead token) for already-finished sessions. Hand ownership of the entry's lifetime to the turn itself: pass the (disarmed) guard or an equivalent removal handle into the QueuedRequest so the worker clears the entry when the streaming turn finishes, or have the stream-relay completion path remove the session's entry once the final chunk is sent. → FIXED: `disarm()` is gone; the streaming submit moves its (armed) guard into `QueuedRequest::active_request_guard`, so the entry lives exactly as long as the turn — the worker drops the request (and guard) when `execute_streaming` finishes, and the enqueue-failure paths drop it on unwind. Mid-stream cancellation still works because the entry persists while the worker holds the request. Regression test `streaming_turn_clears_active_requests_on_completion`.

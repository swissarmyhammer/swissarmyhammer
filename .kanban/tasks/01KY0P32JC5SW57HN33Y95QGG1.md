## What

The review engine's scope stage has no ignore mechanism: `resolve_working` in `crates/swissarmyhammer-validators/src/review/scope.rs` sweeps every tracked change plus untracked code files, and `resolve_sha`/`resolve_glob` take whatever git/the glob returns. Non-source artifacts — most painfully the `.kanban/` board files that every finish-loop comment touches — land in review scope and generate churn. Add a `.reviewignore` file (gitignore syntax), auto-generated at the repo root with `.kanban/` in it by default, and additionally exclude anything the repo's `.gitignore` chain ignores (gitignored files aren't source and must never be reviewed, e.g. a `Scope::Sha`/`Scope::Glob` that touches committed-but-ignored artifacts).

Implementation shape — one choke point, gitignore semantics via the `ignore` crate (workspace dep `ignore = "0.4"` already exists; add `ignore = { workspace = true }` to `crates/swissarmyhammer-validators/Cargo.toml` next to `glob`):

1. **New module `crates/swissarmyhammer-validators/src/review/ignore.rs`** (registered in `src/review/mod.rs`):
   - `ensure_reviewignore(repo_path) -> PathBuf`: if `<repo>/.reviewignore` is missing, write the default template — a short comment header explaining the file (gitignore syntax, consumed by `review` ops) plus `.kanban/`. Never overwrite or append to an existing file; user edits are authoritative.
   - `load_review_ignore_matcher(repo_path)`: build an `ignore::gitignore::Gitignore` via `GitignoreBuilder::new(repo_path)`, adding `<repo>/.reviewignore` and `<repo>/.gitignore` (builder's `add()` — missing files are fine). Full gitignore semantics including `!` negation. Match directories correctly (`matched_path_or_any_parents` so `.kanban/` excludes files under it).
2. **Apply in `resolve_scope_files` (`scope.rs`)**: after the per-scope resolver returns its `ResolvedScope`, filter the file set through the matcher so all four scopes (`Working`/`Sha`/`File`/`Glob`) are covered uniformly. An explicitly named `Scope::File` that is ignored resolves to an empty scope (consistent, not an error). Log each excluded path at debug with the pattern source — full paths, never truncated.
3. Call `ensure_reviewignore` from the same entry (`scope_review`'s scope-resolution path) so any review op auto-generates the file on first run. The generated `.reviewignore` itself is a new untracked non-code file, so it does not enter the working scope.

## Acceptance Criteria
- [ ] Running any review op in a repo without `.reviewignore` creates it at the repo root containing `.kanban/`; a pre-existing `.reviewignore` is never modified (byte-identical after the run)
- [ ] A tracked, edited file under `.kanban/` is excluded from `Scope::Working`; an untracked `.kanban/` file is excluded too
- [ ] A file matched by the repo's `.gitignore` is excluded from `Scope::Sha` and `Scope::Glob` results even when committed/tracked
- [ ] Gitignore semantics hold: a `!pattern` negation in `.reviewignore` re-includes a file that a broader pattern excluded
- [ ] `Scope::File` naming an ignored path yields an empty scope (no findings, no error), and non-ignored review behavior is unchanged (existing scope tests stay green)

## Tests
- [ ] Unit tests in the existing `scope.rs` test module (`TestRepo` harness): (1) tracked+edited `.kanban/board.md` excluded from working scope, (2) committed file matching `.gitignore` excluded from sha scope, (3) glob scope excludes `.reviewignore`d files, (4) `!` negation re-includes, (5) auto-generation creates the default file and a second run preserves a user-edited one
- [ ] Unit tests in `ignore.rs` for template content and matcher construction with missing `.gitignore`/`.reviewignore`
- [ ] Run: `cargo nextest run -p swissarmyhammer-validators review::scope` and `cargo nextest run -p swissarmyhammer-validators review::ignore` — green, all under 10s

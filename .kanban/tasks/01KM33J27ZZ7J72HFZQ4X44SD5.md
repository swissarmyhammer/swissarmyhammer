## What
Regression: dragging tasks between columns on the same board and reordering within a column are both broken. Cross-board dragging still works.

**Root cause analysis:**

The issue is in `board-view.tsx` `handleTaskDragEnd` (lines 250-258):
```typescript
const handleTaskDragEnd = useCallback(
    (_entity: Entity, _dropEffect: string) => {
      setTaskDrag(null);
      emit("drag-ended", {});
      setTimeout(() => cancelSession(), 300);
    }, ...
```

This fires on the **source card** after every drag ends. It unconditionally cancels the backend drag session after 300ms. But `handleTaskDrop` (the drop handler on the target column) also calls `cancelSession()` for same-board drops (line 306) and then calls `persistMove()` which is async.

The race: `handleTaskDrop` fires тЖТ calls `cancelSession()` + async `persistMove()` тЖТ 300ms later `handleTaskDragEnd`'s timeout fires another `cancelSession()`. The second cancel may interfere with the backend state while `persistMove` is still in flight.

Additionally, `handleTaskDragEnd` calls `setTaskDrag(null)` which clears the drag state **before** the drop handler has processed. Since `handleDragEnd` fires on the source and `handleDrop` fires on the target, the ordering depends on the browser. If `dragEnd` fires before or simultaneously with `drop`, the drop handler may not have the drag context it needs.

**Likely fix approach:**
- `handleTaskDragEnd` should NOT cancel the session тАФ that's the drop handler's job. The dragEnd handler should only clean up visual state (setTaskDrag(null)).
- For drops that don't land on a valid target (drag cancelled by user), the dragEnd handler should check `dropEffect` тАФ if it's "none", then cancel the session.
- Remove the 300ms setTimeout cancel тАФ it's a race condition waiting to happen.

**Files:**
- `kanban-app/ui/src/components/board-view.tsx` тАФ `handleTaskDragEnd` (lines 250-258), `handleTaskDrop` (lines 284-339)

**Also verify:**
- Within-column reordering: when source and target column are the same, the before/after placement logic (lines 309-323) should still compute correct ordinals
- The `persistMove` call sends the right `before_id`/`after_id` to `task.move`

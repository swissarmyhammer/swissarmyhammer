## What

Two date displays currently roll their own formatting:

1. **`DateDisplay`** at `kanban-app/ui/src/components/fields/displays/date-display.tsx` — renders the raw stored string. System-derived timestamps come through as RFC 3339 UTC strings like `2026-04-12T15:30:00.123456Z` — ugly.
2. **`StatusDateDisplay`** at `kanban-app/ui/src/components/fields/displays/status-date-display.tsx` (added by the now-shipped smart `status_date` card `01KP24H4ADRYEC1T6KC6SY6K02`) — has its own hand-rolled `formatDurationMagnitude` with `MINUTE`/`HOUR`/`DAY`/`WEEK`/`MONTH`/`YEAR` constants (lines 107–148) producing magnitudes like `"3 days"`, `"2 weeks"` for sentences like `"Completed 2 days ago"` and `"Overdue by 5 days"`.

Build a single luxon-backed helper module that both displays consume. The smart status date card landed first; this card refactors its hand-rolled formatter onto the shared module so there is one source of truth for relative-time phrasing.

### Add luxon

Install **luxon** (and its types) in the UI workspace:

```
cd kanban-app/ui && bun add luxon && bun add -D @types/luxon
```

Luxon's `toRelativeCalendar()` produces calendar-aware English ("yesterday", not "1 day ago") that native `Intl.RelativeTimeFormat` cannot match. ~22kb minified — trivial in a Tauri desktop app.

### Create the format helper

**New file:** `/Users/wballard/github/swissarmyhammer/swissarmyhammer-kanban/kanban-app/ui/src/lib/format-date.ts`

Two exported entry points so both call sites can use the right shape:

```ts
import { DateTime } from "luxon";

/**
 * Full sentence for a date string. Used by DateDisplay where the field
 * value is the only thing rendered (no surrounding prefix).
 *
 * Returns "yesterday", "3 hours ago", "Apr 12, 2026" — calendar-aware.
 * Falls back to the raw string on parse failure.
 */
export function formatDateForDisplay(value: string, now?: DateTime): string;

/**
 * Magnitude-only relative phrase for an instant relative to `now`.
 * Used by StatusDateDisplay to compose sentences like
 * "Completed 2 days ago" / "Scheduled in 3 weeks" / "Overdue by 5 days"
 * where the caller owns the prefix and direction wording.
 *
 * Returns "just now", "5 minutes", "2 days", "3 weeks", "Apr 12, 2026".
 * The caller decides whether to prepend "in" or append "ago".
 */
export function formatRelativeMagnitude(
  timestamp: string,
  now?: DateTime,
): string;
```

Both functions parse via `DateTime.fromISO`, treat bare `YYYY-MM-DD` as midnight UTC, and return the input string unchanged on parse failure (fail-safe).

**`formatDateForDisplay` format ladder** (sentence form):
- **Sub-day** (|Δh| < 24): `dt.toRelative({ base: now })` → "3 hours ago", "45 minutes ago", "in 2 hours"
- **Calendar bucket** (24h ≤ |Δ| ≤ ~30 days): `dt.toRelativeCalendar({ base: now })` → "yesterday", "today", "tomorrow", "2 days ago", "last week", "last month", "in 3 days"
- **Far past/future** (> ~30 days): `dt.toLocaleString(DateTime.DATE_MED)` → "Apr 12, 2026"
- **Same year**: when falling through to the localized string and `dt.year === now.year`, use `{ month: "short", day: "numeric" }` → "Apr 12"
- **Date-only inputs** (no `T`): always use the calendar-bucket or localized-date branches — never the sub-day relative form. "Apr 30" / "tomorrow", never "in 3 hours".

**`formatRelativeMagnitude` format ladder** (magnitude only — match the existing buckets in `status-date-display.tsx`):
- < 1 minute → `"just now"`
- 1–60 minutes → `"N minute(s)"`
- 1–24 hours → `"N hour(s)"`
- 1–7 days → `"N day(s)"`
- 1–4 weeks → `"N week(s)"`
- 1–12 months → `"N month(s)"`
- > 1 year → `"N year(s)"`
- Unparseable → return raw string

Use `DateTime.fromISO` + `Duration` math (e.g. `dt.diff(now, ["years","months","weeks","days","hours","minutes","seconds"])`). Pluralization rules match what's currently in `status-date-display.tsx` so no behavior change in existing snapshots.

Both functions accept `now` so tests can pin the clock.

## What

Two date displays currently roll their own formatting:

1. **`DateDisplay`** at `kanban-app/ui/src/components/fields/displays/date-display.tsx` — renders the raw stored string. System-derived timestamps come through as RFC 3339 UTC strings like `2026-04-12T15:30:00.123456Z` — ugly.
2. **`StatusDateDisplay`** at `kanban-app/ui/src/components/fields/displays/status-date-display.tsx` (added by the now-shipped smart `status_date` card `01KP24H4ADRYEC1T6KC6SY6K02`) — has its own hand-rolled `formatDurationMagnitude` with `MINUTE`/`HOUR`/`DAY`/`WEEK`/`MONTH`/`YEAR` constants producing magnitudes like `"3 days"`, `"2 weeks"` for sentences like `"Completed 2 days ago"` and `"Overdue by 5 days"`.

Build a single luxon-backed helper module that both displays consume. The smart status date card landed first; this card refactors its hand-rolled formatter onto the shared module so there is one source of truth for relative-time phrasing.

### Add luxon

Install **luxon** (and its types) in the UI workspace. (The workspace uses `npm`; `bun` is not available locally — the sibling card 01KP236XG87W8WVB42T2CQ85FD notes the same deviation.)

```
cd kanban-app/ui && npm install luxon && npm install -D @types/luxon
```

Luxon's `toRelativeCalendar()` produces calendar-aware English ("yesterday", not "1 day ago") that native `Intl.RelativeTimeFormat` cannot match. ~22kb minified — trivial in a Tauri desktop app.

### Create the format helper

**New file:** `kanban-app/ui/src/lib/format-date.ts`

Two exported entry points so both call sites can use the right shape:

```ts
import { DateTime } from "luxon";

export function formatDateForDisplay(value: string, now?: DateTime): string;
export function formatRelativeMagnitude(timestamp: string, now?: DateTime): string;
```

Both functions parse via `DateTime.fromISO`, treat bare `YYYY-MM-DD` as midnight UTC, and return the input string unchanged on parse failure (fail-safe).

**`formatDateForDisplay` format ladder** (sentence form):
- **Sub-day** (|Δh| < 24, datetime inputs only): `dt.toRelative({ base: now })` → "3 hours ago", "45 minutes ago", "in 2 hours"
- **Calendar bucket** (|Δ| ≤ ~30 days): `dt.toRelativeCalendar({ base: now })` → "yesterday", "today", "tomorrow", "2 days ago", "last week", "last month", "in 3 days"
- **Far past/future** (> ~30 days): `dt.toLocaleString(DateTime.DATE_MED)` → "Apr 12, 2026"
- **Same year**: when falling through to the localized string and `dt.year === now.year`, use `{ month: "short", day: "numeric" }` → "Apr 12"
- **Date-only inputs** (no `T`): always use the calendar-bucket or localized-date branches — never the sub-day relative form. "Apr 30" / "tomorrow", never "in 3 hours".

**`formatRelativeMagnitude` format ladder** (magnitude only — match the existing buckets exactly so the 14 existing status-date tests pass unchanged):
- < 1 minute → `"just now"`
- 1–60 minutes → `"N minute(s)"`
- 1–24 hours → `"N hour(s)"`
- 1–7 days → `"N day(s)"`
- 1–4 weeks → `"N week(s)"`
- 1–12 months → `"N month(s)"`
- > 1 year → `"N year(s)"`
- Unparseable → return raw string

Both functions accept `now` so tests can pin the clock.

## Implementation notes

- `formatRelativeMagnitude` is implemented with the same millisecond-based bucket constants (`SECOND`/`MINUTE`/`HOUR`/`DAY`/`WEEK`/`MONTH`/`YEAR` — 30-day month, 365-day year) and `Math.round` rules as the helper it replaces. This bit-for-bit preservation is what lets the existing 14 status-date-display tests pass unchanged — they assert on kind-specific prefix/suffix text that the magnitude is embedded into; the bucket rounding governs whether "Scheduled in 1 week" wins over "Scheduled in 7 days" for `daysFromNow(7)`.
- `composeStatusPhrase` signature now takes both the raw timestamp string (to pass to `formatRelativeMagnitude`) and the parsed `Date` (used to compute `deltaMs` for direction past/future). The magnitude helper is intentionally direction-agnostic so the caller owns the "ago" vs "in" wording.
- `formatDateForDisplay` uses `dt.diff(now, "days").days` rather than millisecond math for the threshold check because luxon's `Duration.days` yields a fractional value that's more stable than raw-ms near the 1-day boundary. The `diffDays < 1` / `<= 30` thresholds match the card's specified "24h" and "~30 days" windows.
- Bare `YYYY-MM-DD` inputs are explicitly re-parsed with `{ zone: "utc" }` so "midnight UTC" semantics apply regardless of the caller's local zone — matching what `parseDateOrDatetime` already does in `status-date-display.tsx`.

title: Stream actual review findings + verdicts over MCP notifications/message as they resolve
---
## What

Stream the review engine's actual results over MCP as they are produced — not just progress ticks — so an MCP client (e.g. the main agent driving a review) can start resolving findings in parallel while the run is still going. Builds on the ^jn2wjd5 event channel/bridge (depends on it).

Ground truth about finality (verified in code): fan-out validator tasks run independently (`collect_fan_out`, `crates/swissarmyhammer-validators/src/review/fleet.rs`), and verification is per-finding and independent — `verify_findings` (`crates/swissarmyhammer-validators/src/review/verify.rs`) runs each candidate through the deterministic `run_guard` and then its own adversarial verify task, pipelined on the same pool; `synthesize` only exact-dedups, it never revises verdicts. So each finding's data is final at two well-defined moments: its raw content when its validator task completes, and its confirmed/refuted verdict when its verify task resolves. Stream both.

Transport: `notifications/progress` carries only progress/total/message — wrong channel for content. Use MCP logging notifications (`notifications/message`), which carry arbitrary structured `data`; the peer path already exists (`send_mcp_log` in `crates/swissarmyhammer-tools/src/mcp/tool_registry.rs` / `peer.notify_logging_message`). Full payloads always — never truncate a finding.

Implementation shape:

1. **Engine events** — extend `ReviewProgressEvent` (`crates/swissarmyhammer-validators/src/review/fleet.rs`, added by ^jn2wjd5) with two content-carrying variants:
   - `Findings { validator: String, findings: Vec<Finding> }` — emitted in `collect_fan_out` at each validator task's resolution with the task's parsed, validator-tagged findings (empty vec = that validator came back clean; failed task = no Findings event, the existing PairDone/failure accounting covers it).
   - `Verdict { finding: Finding, confirmed: bool, reason: String }` — emitted as each candidate's verdict resolves: guard refutations in/after `run_guard`, agent verdicts in `collect_verify`/`verify_findings` (`verify.rs`). Requires threading the existing optional progress sender from `run_review_over_agent` (`drive.rs`) into the verify stage (it already reaches the fleet stage from ^jn2wjd5).
2. **MCP bridge** — in `spawn_review_progress_bridge` (`crates/swissarmyhammer-tools/src/mcp/tools/review/review_op.rs`, from ^jn2wjd5), map the two new variants to `LoggingMessageNotificationParam` with structured `data` (e.g. `{"kind": "review.findings", "validator": …, "findings": [full Finding JSON…]}` / `{"kind": "review.verdict", "finding": …, "confirmed": …, "reason": …}`, logger `"review"`, level Info) sent via the peer; keep progress-tick variants flowing to `notifications/progress` as today. When only an in-process `progress_sink` exists (no peer), skip logging notifications — the sink contract is progress params.
3. Document the streamed message shapes in `crates/swissarmyhammer-tools/src/mcp/tools/review/description.md` — note events are per-validator granular while the final report exact-dedups across validators (dedup, never retraction).

Subtasks:

## Acceptance Criteria
- [ ] During a review with a live peer, each validator task's completion produces one `notifications/message` carrying every parsed finding for that validator, before the final tool result arrives
- [ ] Each finding's verdict (confirmed or refuted, with reason — both guard- and agent-decided) produces its own `notifications/message` as it resolves
- [ ] Streamed finding payloads are complete Finding JSON (file, rule, validator, claim/description fields) — no truncation, no summaries
- [ ] The final `ReviewReport` is unchanged, and the union of streamed confirmed verdicts matches the report's findings modulo synthesize's exact dedup
- [ ] With no peer (in-process sink only, or neither), zero logging notifications are emitted and existing behavior is unchanged

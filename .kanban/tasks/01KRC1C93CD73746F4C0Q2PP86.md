## What

Foundational change: add an optional `view_id` field alongside the existing `view` (kind) field on perspectives so that perspectives can be scoped to a specific view instance, while existing perspectives that only have `view: board`/`view: grid` continue to load and behave as before (shared by kind).

### Files to modify

- `swissarmyhammer-kanban/src/perspective/add.rs` ŌĆö `AddPerspective` command struct: add `pub view_id: Option<String>` with `#[serde(default, skip_serializing_if = "Option::is_none")]`. Doc-comment it as "id of the view this perspective is scoped to; None means legacy shared-by-kind."
- `swissarmyhammer-kanban/src/perspective/mod.rs` (or wherever `Perspective` is defined ŌĆö confirm via `code_context get symbol Perspective`): add the same `view_id: Option<String>` field with the same serde attrs. Update all constructors that take `view` to accept `view_id`. Keep `view: String` (kind) for backwards-compat.
- `swissarmyhammer-kanban/builtin/commands/perspective.yaml` ŌĆö declare `view_id` argument on relevant verbs (at minimum `add`, `update`). Use the existing `view_kind` argument pattern in the same file as the model.
- `kanban-app/ui/src/types/kanban.ts` ŌĆö extend `PerspectiveDef`:
  ```ts
  export interface PerspectiveDef {
    readonly id: string;
    readonly name: string;
    /** Legacy: view kind ("board", "grid"). Retained for backwards compat. */
    readonly view: string;
    /** New: id of the specific view this perspective is scoped to.
     *  Optional ŌĆö when absent, perspective is shared across all views of `view` kind. */
    readonly view_id?: string;
    // ... rest unchanged
  }
  ```

### Compatibility rule

Document this rule in the `Perspective` struct doc-comment AND the `PerspectiveDef` TS docstring:

> When `view_id` is `Some(id)`, the perspective belongs to exactly that view instance. When `view_id` is `None`, the perspective is **legacy-shared**: it appears in every view whose kind matches `view`. Newly created perspectives always set `view_id`; legacy `view_id`-less perspectives keep working unchanged until re-saved.

No behavior changes in this task. The filter call sites still ignore `view_id`. Subsequent tasks switch the filters.

## Implementation Notes

The three new round-trip unit tests live in `swissarmyhammer-perspectives/src/types.rs` (the actual home of the `Perspective` struct ŌĆö `swissarmyhammer-kanban/src/perspective/mod.rs` re-exports from there). The TS type test was added as a new file `kanban-app/ui/src/types/kanban.test.ts` (no pre-existing `kanban.types.test.ts`).

The kanban command-yaml file (`swissarmyhammer-kanban/builtin/commands/perspective.yaml`) declares slash/keybinding commands rather than the `add`/`update` operation verbs directly, so `view_id` was added as an arg on:
- `perspective.save` (the user-facing "save current view as a new perspective" path, which dispatches `AddPerspective`)
- `perspective.goto` (which already declared `view_kind` ŌĆö `view_id` pairs with it for per-view-id resolution)

Both `add.rs` and `update.rs` thread `view_id` through `Execute::execute`. `dispatch.rs` (`dispatch_add_perspective` and `dispatch_update_perspective`) now reads `view_id` from operation params; `UpdatePerspective.view_id` uses the same `Option<Option<String>>` tri-state shape as `filter` and `group`, so `Some(None)` clears the field back to legacy shared-by-kind.

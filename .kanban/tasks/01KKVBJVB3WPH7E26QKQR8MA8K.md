**Full stack, not just playback**\n\nA true e2e test for validator tools needs the entire stack running in isolation:\n\n1. **Setup**: Isolated temp directory with `.git`, source files to validate against\n2. **MCP server**: Start HTTP MCP server with `/mcp/validator` endpoint serving `code_context` + `files` (read-only)\n3. **AVP install**: Initialize AVP context with a validator that requires tool use (e.g. \"read src/main.rs and check for unsafe blocks\")\n4. **Agent spawn**: Validator agent spawned with `--tools \"\" --mcp-config <validator-endpoint> --strict-mcp-config`\n5. **Agent writes code**: Trigger a code change (or simulate PostToolUse hook with file context)\n6. **Validator fires**: Validator receives the hook event, calls `files` tool to read the file, calls `code_context` to check symbols\n7. **Result flows back**: Validator returns pass/fail JSON, runner parses it, decision propagates\n\nThe key assertion: the validator's decision was **informed by actual tool results**, not just the hook event JSON. A validator that says \"read the file and check for X\" must actually read the file via MCP.\n\n**Recording for CI**: Once the live e2e works, record the agent interaction as a PlaybackAgent fixture so CI can replay without a live LLM. The fixture captures the tool_use requests the agent made and the tool results it received.\n\n**What this catches that unit tests don't**:\n- Route ordering bugs (axum nesting `/mcp/validator` vs `/mcp`)\n- MCP session management for the validator endpoint\n- `--tools \"\"` actually disabling built-ins (not just passing the flag)\n- `--strict-mcp-config` preventing tool escape\n- Tool context sharing (the blocker finding — registry leak)\n- Prompt template rendering with tool descriptions\n- End-to-end JSON parsing of validator results after tool calls\n\n**Verification**: `cargo test -p avp-common --test e2e_validator_tools` passes both live and playback modes.

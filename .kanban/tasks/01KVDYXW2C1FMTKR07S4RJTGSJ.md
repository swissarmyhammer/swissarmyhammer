## What
Follow-on to ^ref6nj4 (Request/response multiplexer over the election socket). That card built and tested the multiplexer MECHANISM end-to-end (RequestServer/RequestClient in `swissarmyhammer-leader-election/src/request_ipc.rs`; `serve_session_requests` + `SessionRequestClient` in `swissarmyhammer-diagnostics/src/request_api.rs`; gated integration test with a real rust-analyzer leader + follower over a real socket). It deliberately DEFERRED the live production rewiring as separate cross-crate scope. This card captures that deferred wiring so the follower diagnostics path actually rides the multiplexer in `sah serve`.

## Why
Today the live MCP path does NOT use the multiplexer:
- `crates/swissarmyhammer-tools/src/mcp/server.rs` does not call `serve_session_requests` anywhere — the leader never binds/serves the request socket at startup.
- `crates/swissarmyhammer-tools/src/mcp/tools/diagnostics/mod.rs` `produce_outcome()` still returns `settled_empty()` for followers instead of routing through `SessionRequestClient`.

So a follower process running the diagnostics MCP tool silently returns empty rather than round-tripping to the leader's single rust-analyzer. The mechanism exists but is unplugged.

## Acceptance Criteria
- [x] Leader spawns `serve_session_requests(...)` at startup in `server.rs` `do_initialize_code_context`'s Leader branch, binding the `RequestServer` at the election `socket_path()`. (server.rs `spawn_request_server_for_leader`, invoked from `spawn_lsp_workers_for_clients` — the chokepoint both initial-leader and post-promotion startup pass through, where the live sessions first become available.)
- [x] Expose `socket_path()` (and the lock path) on `CodeContextWorkspace` (the election already knows it) so the tool layer can construct a `SessionRequestClient`. (workspace.rs pub `socket_path()`/`lock_path()`.)
- [x] `produce_outcome()` falls back to `SessionRequestClient::connect(socket, lock)` when there is no in-process session, instead of `settled_empty()`; connect failure surfaces the typed not-leader/leader-pid error. (diagnostics/mod.rs `diagnose_via_leader`, returns `Result<DiagnoseOutcome, IpcError>`; execute_check surfaces the error.)
- [x] Integration: a follower `sah` process running the diagnostics tool gets a real DiagnosticsReport from the leader over the socket, with only the leader's single rust-analyzer running (gated on rust-analyzer). (tools `tests/integration/diagnostics_request_socket_e2e.rs` — follower routes diagnose through `ws.socket_path()`/`ws.lock_path()` to a real rust-analyzer leader, asserts a real clean report; serial + kill-on-drop; rust-analyzer exercised, no orphan.)
- [x] Address the ^ref6nj4 double-check item: blocking `session.request` inside an async task. (request_api.rs `lsp_request_blocking` runs the synchronous blocking `session.request` on `spawn_blocking`; unit `dispatch_lsp_request_does_not_block_the_runtime_thread`.)

## Notes

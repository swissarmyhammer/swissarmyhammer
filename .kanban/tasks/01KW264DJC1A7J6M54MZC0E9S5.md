## What
Stand up the ACP delegation seam by reusing `review`'s machinery, so `expect` can borrow an agent for reasoning while keeping the verdict deterministic and inside `expect`. Per `ideas/expect.md` §"Delegation over ACP". Preserve the engine boundary: the engine is agent-construction-free and receives a `DynConnectTo<Client>` + notifier.

- Reuse from `swissarmyhammer-validators` (do NOT re-derive):
  - `AgentPool` (`src/validators/pool.rs`) — `submit`/`submit_forked`/`submit_primed`/`SessionPinGuard`/`PoolError`, idle_timeout→abandon_turn→session/cancel.
  - The `run_review_over_agent` ACP wiring (`src/review/drive.rs`): `Client.builder().with_handler(TolerantResponseRouter)`, the SINGLE notifier feed (double-feeding corrupts JSON), once-per-connection `initialize`, `answer_agent_request` for `session/request_permission` (auto-allow) + `fs/read_text_file` (confined under repo root).
- New `crates/swissarmyhammer-expect/src/drive.rs`: `run_expect_over_agent(agent: DynConnectTo<Client>, notification_rx, scope, repo_root, config, …)` mirroring review's flow. **Tamper-resistance**: `answer_agent_request` must DENY writes under `.expect/` (specs/goldens/fixtures) — extend review's `WriteTextFileRequest` handling so the driving agent cannot edit the ledger.
- Tool/engine glue in `tools/expect/`: mirror `review_op.rs` — `AgentHandle { agent, notification_rx }`, an `AgentFactory` seam, a process-global `EXPECT_PIPELINE_GATE` semaphore, and the spawn-blocking-on-a-current-thread-runtime pattern. Resolve the driving agent from `ToolContext::get_agent_for_use_case(AgentUseCase::Expectations)`.
- Decision to record in the task per Open Question 4: start by copying `drive.rs`/`pool.rs` patterns (or depend on validators' re-exported `AgentPool`/`run_review_over_agent`); converge into a shared crate later. `extract_json_value` is `pub(crate)` in validators — re-export it (or copy the tolerant extractor) for `expect`'s structured capture.

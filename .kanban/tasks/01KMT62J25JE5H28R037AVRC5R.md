title: NavBar board inspect button uses callback prop instead of command dispatch
---
**File:** `kanban-app/ui/src/components/nav-bar.tsx`, lines 40-49 and `App.tsx` line 549-551\n\n**Anti-pattern:** The NavBar receives an `onBoardInspect` callback prop, and the board inspect button calls it directly. In App.tsx, this is wired as `() => inspectEntity(\"board\", \"board\")`. This bypasses the command scope chain entirely.\n\n**Current code (nav-bar.tsx):**\n```tsx\n<button onClick={onBoardInspect} title=\"Inspect board\">\n  <Info className=\"h-4 w-4\" />\n</button>\n```\n\n**Current code (App.tsx):**\n```tsx\nonBoardInspect={() => inspectEntity(\"board\", \"board\")}\n```\n\n**Correct pattern:** The NavBar should use `useExecuteCommand()` to dispatch a command like `board.inspect` or `entity.inspect` through the scope chain. The board entity already has commands registered via `useEntityCommands(\"board\", \"board\")` in BoardView.\n\nHowever, NavBar sits OUTSIDE the BoardView's FocusScope, so the board's `entity.inspect` command is not in the NavBar's scope chain. The fix would require either:\n1. Moving the board-level command scope higher in the tree (above NavBar), or\n2. Dispatching `entity.inspect` directly to Rust with `target: \"board:board\"` (matching the pattern in `entity-commands.ts`)\n\n**Severity:** Warning. Same functional result today but skips command logging and scope chain resolution.\n\n**Scope chain impact:** The `onBoardInspect` callback prop is a callback-passing anti-pattern. It forces App.tsx to wire the action imperatively instead of letting the command system handle it. #review-finding

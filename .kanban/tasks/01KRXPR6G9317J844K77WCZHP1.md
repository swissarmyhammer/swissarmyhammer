title: ACP capability advertise-vs-enforce consistency
---
An advertised ACP capability must match what the agent actually enforces or delivers. The two agents diverge.

## Current inconsistencies
- `loadSession`: claude enforces it before `load_session` acts (`check_load_session_requirements`); llama advertises it (from `config.capabilities.supports_session_loading`) but `load_session` never checks it.
- `supports_slash_commands`: llama advertises `true` in the capabilities `meta` map, but its `CommandRegistry` (`crates/llama-agent/src/acp/commands.rs`) is dead code — not wired into the server, no `AvailableCommandsUpdate` ever emitted. It advertises a capability it does not deliver.
- prompt content: claude advertises `image/audio/embedded_context: true` and validates prompt content against it (`ContentCapabilityValidator`); llama advertises all three `false` but does no content validation — it will not reject content types it claims not to support.
- Capability `meta` shape differs: claude puts `tools`; llama puts `supports_modes/plans/slash_commands`. Real capabilities living as ad-hoc `meta` keys (rather than first-class fields) is itself inconsistent.

## Target
Every advertised capability is either enforced (gated) or genuinely delivered, symmetrically on both agents:
- `loadSession`: if advertised, both enforce; if either cannot, advertise accordingly.
- slash commands: either wire llama's `CommandRegistry` and emit `AvailableCommandsUpdate`, or stop advertising `supports_slash_commands`.
- prompt content: both reject content types advertised as unsupported — run the content validator in llama too.
- Audit the capability `meta` maps: anything that is a real ACP capability should be a first-class field, not a `meta` key.

## Verify
- For each advertised capability, a check that the agent enforces or delivers it; both agents behave identically.

Ties to the notification-parity card (slash commands / AvailableCommandsUpdate).

## Resolution (implemented)

### loadSession enforcement (llama)
`AcpServer::load_session` now gates on `config.capabilities.supports_session_loading` before doing any work. When the capability is not advertised it returns ACP error `-32601` with `data.requiredCapability = "loadSession"` / `data.method = "session/load"` — identical to claude's `LoadSessionNotSupported` mapping. claude already enforced this via `check_load_session_requirements`; the two agents are now symmetric.

### Slash commands decision: STOP ADVERTISING (not wire)
Decision: stop advertising slash-command support rather than wire the `CommandRegistry`.

Rationale: advertising a capability the agent does not deliver is the bug. Wiring `CommandRegistry` into the server is a substantial cross-cutting feature — it needs the registry constructed with an MCP client + skill library at session creation, `AvailableCommandsUpdate` notifications emitted onto the broadcast channel, and change-detection wired to MCP server changes. That work is explicitly tracked by the separate notification-parity card. The task instructs preferring the smaller honest change when wiring is large, so:
- Removed the `supports_slash_commands` field from `AcpCapabilities` (it was a dead config knob).
- `initialize` no longer emits `supports_slash_commands` in the agent-capability `_meta` map.
- The `CommandRegistry` code and its unit tests are kept intact, ready for the notification-parity card to wire.

### Prompt content validation (llama)
New module `crates/llama-agent/src/acp/content_validation.rs` — a minimal equivalent of claude's `ContentCapabilityValidator`. `AcpServer::prompt` now rejects content types the agent advertised as unsupported (image/audio/embedded resource) with ACP error `-32602` and the same structured `data` shape as claude (`contentType`, `declaredCapability`, `required`, `supportedTypes`). The advertised `PromptCapabilities` are produced by a single `advertised_prompt_capabilities()` helper used by both `initialize` and `prompt`, so advertise and enforce can never drift.

### Capability `meta` audit
ACP 0.11 `AgentCapabilities` / `PromptCapabilities` have NO first-class fields for `streaming`, `tools`, `supports_modes`, or `supports_plans` — only `load_session`, `prompt_capabilities`, `mcp_capabilities`, `session_capabilities`. So none of the existing `_meta` keys is a real first-class ACP capability; they are genuine agent-specific extras and correctly stay in `_meta`. The only `_meta` fix required was removing the dishonest `supports_slash_commands` key. Both agents already use the proper first-class fields.

### Tests
3 new regression tests in `acp/server.rs` (loadSession rejected when disabled / passes gate when advertised / prompt rejects unsupported content) plus 10 unit tests in `content_validation.rs`. Conformance fixtures under `.fixtures/llama/` updated to drop the stale `supports_slash_commands` key. `cargo nextest run -p llama-agent -p claude-agent`: 1432 passed, 0 failed. Clippy clean for both crates.

### Nits
- [x] `crates/llama-agent/src/acp/content_validation.rs:119` — `validate_content_blocks` returns only the *first* violation, whereas claude's `ContentCapabilityValidator::validate_content_blocks` collects all violations and returns a `MultipleViolations` variant when a prompt has more than one bad block. Single-violation error `data` is byte-identical between the agents (so the common case is symmetric), but a multi-violation prompt observes a different error shape — claude returns `MultipleViolations` with a `violations` array, llama returns the first single violation. Either document this as an accepted simplification or mirror claude's multi-violation collection for full shape symmetry.
- [x] `crates/acp-conformance/.fixtures/llama/test_content_validation.json` — the fixture named `test_content_validation` only sends text content blocks, so it does not exercise the new rejection path at all (rejection is covered by the unit tests in `content_validation.rs` and `test_prompt_rejects_unsupported_content_type` in `server.rs`). The fixture name is now slightly misleading; consider renaming it or adding a fixture that actually sends an unsupported (image/audio/resource) block and asserts the `-32602` rejection.

## Nit Resolution (2026-05-18)

### Nit 1 — multi-violation shape symmetry (FIXED)
`ContentCapabilityError` in `content_validation.rs` is now an enum mirroring claude's: `UnsupportedContentType { content_type, required_capability, supported_types }` and `MultipleViolations { violation_count, violations }`. `validate_content_blocks` collects *all* violations — 0 → `Ok`, exactly 1 → the single `UnsupportedContentType` (byte-identical `data` to before), >1 → `MultipleViolations` carrying every violation. `to_acp_error` matches claude's shapes: single → `data` with `contentType`/`declaredCapability`/`required`/`supportedTypes`; multiple → `data` with `violationCount` and a `violations` array of nested JSON-RPC error objects. `AcpServer::validate_prompt_content` updated to log `acp_error.message` (the struct field it referenced is gone). Added two unit tests (`validate_blocks_multiple_violations_collected`, `multiple_violations_acp_error_carries_every_violation`); 11 unit tests in the module now pass.

### Nit 2 — conformance fixture genuinely exercises rejection (FIXED)
`test_content_validation` driver now sends an image content block as its primary probe: agents advertising `image: false` MUST reject with `-32602` (asserted on `e.code`), agents advertising `image: true` skip the probe (their accept path is covered by `test_image_content_with_capability`). The llama `test_content_validation.json` fixture was rebuilt to record `initialize` → `new_session` → `prompt(image)` with a recorded `-32602` error response carrying the full structured `data` — so playback genuinely replays the rejection. `ContentMockAgent` in the module unit tests was updated to enforce the same contract (reject image/audio/resource with `-32602`) so it remains a faithful stand-in. The claude fixture is unchanged (claude advertises `image: true`, so the driver skips the probe for claude).

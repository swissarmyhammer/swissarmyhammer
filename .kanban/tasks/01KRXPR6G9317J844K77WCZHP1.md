title: ACP capability advertise-vs-enforce consistency
---
An advertised ACP capability must match what the agent actually enforces or delivers. The two agents diverge.

## Current inconsistencies
- `loadSession`: claude enforces it before `load_session` acts (`check_load_session_requirements`); llama advertises it (from `config.capabilities.supports_session_loading`) but `load_session` never checks it.
- `supports_slash_commands`: llama advertises `true` in the capabilities `meta` map, but its `CommandRegistry` (`crates/llama-agent/src/acp/commands.rs`) is dead code — not wired into the server, no `AvailableCommandsUpdate` ever emitted. It advertises a capability it does not deliver.
- prompt content: claude advertises `image/audio/embedded_context: true` and validates prompt content against it (`ContentCapabilityValidator`); llama advertises all three `false` but does no content validation — it will not reject content types it claims not to support.
- Capability `meta` shape differs: claude puts `tools`; llama puts `supports_modes/plans/slash_commands`. Real capabilities living as ad-hoc `meta` keys (rather than first-class fields) is itself inconsistent.

## Target
Every advertised capability is either enforced (gated) or genuinely delivered, symmetrically on both agents:
- `loadSession`: if advertised, both enforce; if either cannot, advertise accordingly.
- slash commands: either wire llama's `CommandRegistry` and emit `AvailableCommandsUpdate`, or stop advertising `supports_slash_commands`.
- prompt content: both reject content types advertised as unsupported — run the content validator in llama too.
- Audit the capability `meta` maps: anything that is a real ACP capability should be a first-class field, not a `meta` key.

## Verify
- For each advertised capability, a check that the agent enforces or delivers it; both agents behave identically.

Ties to the notification-parity card (slash commands / AvailableCommandsUpdate).

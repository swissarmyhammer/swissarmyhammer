## Symptom
A `review sha` over a large/multi-commit range fails ALL fan-out tasks (e.g. 15/15), tripping the completeness guard (`check_review_completeness`, >50% failed) and returning an error. Small/single-commit reviews (what `/finish` runs as `HEAD~1..HEAD`) are fine. The engine's error text blames "saturated/unavailable agent queue" — that is a guess and is WRONG.

## Root cause (verified by code + reproduction)
The fan-out grain is the VALIDATOR, not (validator × file). All changed files are inlined ONCE into a single shared "primed prefix" (`fleet.rs::render_run_prime`), and each of ~M validators forks that prime and runs its whole rule-set over ALL files at once. So it is **M invocations, each crammed with every diff** — NOT N×M.

There is **no aggregate size budget on the prime** (only a per-file cap, `MAX_INLINE_SOURCE_BYTES = 32 KB`), and file-batching was deliberately removed ("no longer a file-batching dimension"). The prime inlines the FULL current content of each changed file (≤32 KB), not just the diff. When the combined content exceeds the review model's context window the prime turn fails; the per-task fallback (`render_fleet_prompt`) ALSO carries everything and also fails → every validator fails uniformly and immediately (not a slow timeout). Hard cliff, all-or-nothing.

## Reproduction (this repo, post-rebuild engine)
`attempted` is constant 15 regardless of file count — proof the grain is per-validator (M), not N×M:
| Range | Files | Added code | Result |
|---|---|---|---|
| `ad7ffced1` | 1 | ~6 | 15 attempted, 0 failed |
| `eccd288dc` | 12 | ~158 | 15 attempted, 0 failed |
| `725ab50cd` | 75 | ~54 (mostly deletions) | 15 attempted, 0 failed |
| `eccd288dc^..HEAD` | 114 | ~974 | 15 attempted, **15 FAILED** |
75 files PASSED but 114-with-974-insertions FAILED → it is aggregate inlined **content volume**, not file count.

## Decided design — content-budgeted batching at file granularity
Rationale: cramming every diff into one context reviews POORLY even when it fits (attention dilution → misses instances, fights "report every instance"). So batches are kept deliberately small and is a tunable, NOT derived from the model context window.

- **`batch_size` is a NEW parameter on the review tool**, default **32 KB (32768 bytes)**. Applies to every scope (`working`/`sha`/`file`).
- **Whole-file granularity.** Greedily pack whole files into a batch until adding the next file's inlined content would exceed `batch_size`; then start a new batch. **NEVER split a file across batches** — a file is atomic.
- **A single file whose inlined content exceeds `batch_size` is a HARD ERROR.** Do NOT degrade it to a bounded slice and do NOT spill to `read_file`. Error must name the file, its size, and the limit, and instruct: raise `batch_size` or narrow scope.
- Each batch runs the full fan-out: its own shared prime over that batch's files, fork per validator. **Findings merged + deduped by file:line across batches** (synthesis already dedups by file:line).
- Batches are independent → parallelize across the pool (respecting the pipeline gate / worker count). Total invocations = M validators × B batches, each bounded by `batch_size`.

## Reconcile with existing per-file cap
Today an oversized file silently degrades to a bounded slice via `MAX_INLINE_SOURCE_BYTES = 32 KB` (+ `OVERSIZED_FILE_READ_NOTE`). The new rule REPLACES that silent degrade with an explicit error (fail loud, consistent with the rest of the engine). Default `batch_size` == that 32 KB cap, so default behavior: files ≤32 KB inline whole and pack into batches; the first file over 32 KB errors instead of being quietly trimmed. Remove/retarget the slice-degrade path accordingly.

## Wiring
- Review tool schema: add `batch_size` (bytes, default 32768) to the `review` op input; thread through `ReviewRequest` → scope/fleet.
- Batching dimension lives where the work-list / prime is built (`scope.rs` work-list → `fleet.rs` prime), restoring a batching loop around the per-run prime.
- Replace the misleading "saturated/unavailable agent queue" wording in `check_review_completeness` (review_op.rs) — distinguish real fan-out failure from "diff too large / file over batch_size".

## Acceptance
- `review sha eccd288dc^..HEAD`-class range (114 files) completes with findings (batched), instead of 15/15 failing.
- Small reviews unchanged (still 1 batch).
- A single file > `batch_size` returns a clear error naming the file + size + limit; no file is ever split.
- `batch_size` is settable per call and defaults to 32 KB.

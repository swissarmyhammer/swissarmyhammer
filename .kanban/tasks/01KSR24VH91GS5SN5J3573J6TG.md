## Core principle (from user)
ALL agent-specific config knowledge lives in **mirdan**, behind a **Rust strategy per supported agent**. A tool declares intent through the EXISTING `Initializable` trait and **delegates to mirdan** — it never writes agent config itself. No new trait (`Initializable` stays; it was just used poorly). The tool/common layers must contain ZERO Claude-specific paths or logic.

## 1. mirdan: per-agent strategy (Rust) — mirdan is in charge of agent-specific config
- New trait `AgentConfigStrategy` (e.g. `crates/mirdan/src/strategy/mod.rs`):
  - `fn register_mcp(&self, agent: &AgentDef, scope: InitScope, server_name: &str, entry: &McpServerEntry) -> Result<bool, RegistryError>`
  - `fn unregister_mcp(&self, agent: &AgentDef, scope: InitScope, server_name: &str) -> Result<bool, RegistryError>`
  - `fn deny_tool(&self, agent: &AgentDef, scope: InitScope, tool: &str) -> Result<bool, RegistryError>`
  - `fn allow_tool(&self, agent: &AgentDef, scope: InitScope, tool: &str) -> Result<bool, RegistryError>`
  - (returns `bool` = changed; emits nothing — callers emit reporter events)
- Per-agent impls: `ClaudeCodeStrategy` (owns: MCP per scope = `.mcp.json` / `~/.claude.json` `projects.<project_key>.mcpServers` / global `~/.claude.json`; deny = `permissions.deny` array in `.claude/settings.json` | `.claude/settings.local.json` | `~/.claude/settings.json`), plus a `GenericMcpJsonStrategy` for agents that just use the YAML `mcp_config` fields (servers_key/entry_extras/project_path/global_path) and have no special local/permission mechanism (e.g. Zed via entry_extras source:custom).
- Dispatch: `fn strategy_for(agent: &AgentDef) -> Box<dyn AgentConfigStrategy>` matching on `agent.id` ("claude-code" → ClaudeCodeStrategy; else Generic). The strategy reads existing `AgentDef` data (mcp_config, settings_path, etc.) and adds the structural behavior data can't express.

## 2. mirdan: thin appliers the tools call (iterate detected agents → dispatch to strategy)
In a mirdan module (e.g. `mirdan::install`):
- `pub fn register_mcp_server(scope: InitScope, server_name: &str, entry: &McpServerEntry, reporter: &dyn InitReporter) -> Vec<InitResult>`
- `pub fn unregister_mcp_server(scope, server_name, reporter) -> Vec<InitResult>`
- `pub fn deny_tool(scope, tool, reporter) -> Vec<InitResult>`
- `pub fn allow_tool(scope, tool, reporter) -> Vec<InitResult>`
Each loads detected agents, gets `strategy_for(agent)`, applies, emits Action/Skipped/Warning events, aggregates results. This is the SINGLE place per-agent register/deny/skill iteration + reporting lives (kills the tool↔sah triplication the review flagged).
(mirdan already depends on swissarmyhammer-common, so it can take `InitScope` + `&dyn InitReporter`.)

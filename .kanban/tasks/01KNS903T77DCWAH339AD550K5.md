## What

Add the infrastructure artifacts that support the MCP server runtime. These are independent of the command implementations and can land early.

### 1. File-based tracing to `.kanban/mcp.log`

Set up tracing in `main.rs` using the `FileWriterGuard` pattern from `shelltool-cli/src/main.rs`. Log dir is `".kanban"`, log file is `mcp.log`: `PathBuf::from(".kanban").join("mcp.log")`. Fall back to stderr if the log file can't be created.

The `FileWriterGuard` struct wraps an `Arc<Mutex<File>>` and flushes + syncs on every write — copy it directly from `shelltool-cli/src/main.rs`.

### 2. `.kanban/.gitignore` entry for `mcp.log`

The `.kanban/` directory is committed board data, unlike `.shell/` which is entirely gitignored. Add a `mcp.log` entry to `.kanban/.gitignore` so the runtime log doesn't get tracked. Must handle the case where `.kanban/.gitignore` may already exist — append `mcp.log` only if not already present.

### 3. `kanban-cli/.mcp.json` for local dev

Create `kanban-cli/.mcp.json` for development use:
```json
{
  "mcpServers": {
    "kanban": {
      "command": "kanban",
      "args": ["serve"]
    }
  }
}
```

This is a static file committed to the repo, mirroring `shelltool-cli/.mcp.json`.

## Acceptance Criteria
- [ ] `kanban serve` writes tracing output to `.kanban/mcp.log`, not stderr
- [ ] `.kanban/.gitignore` contains `mcp.log`
- [ ] `kanban-cli/.mcp.json` exists and is valid JSON with `kanban serve` entry
- [ ] If `.kanban/` doesn't exist yet, logging falls back to stderr gracefully

swissarmyhammer-tools/src/mcp/tool_config.rs:28-43\n\nBoth `ToolEntry { pub enabled: bool }` and `ToolConfig(pub HashMap<String, ToolEntry>)` use public fields. Per Rust API guidelines, public fields on non-`Copy` structs are a permanent API commitment — once published, you can never change the representation without a breaking change.\n\nCallers in `commands/tools/mod.rs` directly construct `ToolEntry { enabled: true }` and access `config.0[name]`, coupling the CLI tightly to the internal representation. If `ToolEntry` ever needs a second field (e.g., `reason: Option<String>`) or `ToolConfig` switches from `HashMap` to `BTreeMap`, all callers break.\n\nSuggestion:\n- `ToolEntry`: Make `enabled` private, add `pub fn new(enabled: bool) -> Self` and `pub fn is_enabled(&self) -> bool`.\n- `ToolConfig`: Make the inner `HashMap` private, expose `get`, `insert`, `iter`, and `clear` methods instead of `.0` direct access.\n\nVerification: Ensure all callers in `commands/tools/mod.rs` and tests compile after refactoring.\n\n#review-finding #review-finding

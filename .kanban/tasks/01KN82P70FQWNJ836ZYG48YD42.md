## What

Remove the static board window from `tauri.conf.json` so ALL board windows are created dynamically via `create_window_impl`. This eliminates the primary/secondary asymmetry that caused `dragDropEnabled`, `disable_drag_drop_handler`, and inspector scoping bugs.

The quick-capture window stays static (it's a different kind of window with different config — transparent, no decorations, etc.).

### Current state

- `kanban-app/tauri.conf.json` defines two static windows: the main board window and quick-capture
- `kanban-app/src/main.rs:123-137` — special-cases the \"main\" window for geometry restore and show
- `kanban-app/src/main.rs:139-175` — restores secondary windows via `create_window_impl`, skipping \"main\" and \"quick-capture\"
- Dynamic windows get `disable_drag_drop_handler()` in `create_window_impl` (`commands.rs:689`)

### Changes

1. **`kanban-app/tauri.conf.json`** — Remove the first window entry (the board window). Keep only quick-capture. Tauri requires at least one window OR the app must create one in setup.

2. **`kanban-app/src/main.rs` setup()** — Replace the two-path logic (special main + restore secondaries) with one path:
   - Read ALL saved windows from `UIState::all_windows()` (no skip for \"main\")
   - For each saved window with a board_path, call `create_window_impl` with saved label + geometry
   - If no saved windows have board paths, create one window for: CLI arg path > cwd `.kanban` > new board dialog
   - Remove the `if label == \"main\"` skip at line 147

3. **`kanban-app/src/commands.rs` create_window_impl** — Already handles everything. The `disable_drag_drop_handler()` call ensures `dragDropEnabled` is false on dynamic windows, matching the pre-attachment-feature behavior.

## What
Build the bidirectional message stream the TypeScript ACP SDK consumes, backed by the **browser-native `WebSocket`**. This is the webview's connection to the ACP agent — pure WebSocket, no Tauri in the data path.

- Add `@agentclientprotocol/sdk` to `apps/kanban-app/ui/package.json`.
- New `apps/kanban-app/ui/src/ai/acp-stream.ts`. Open a browser `WebSocket` to the `ws://127.0.0.1:<port>` URL returned by `ai_start_agent` (task "ACP agent process manager"). Wrap it as a `ReadableStream<Uint8Array>` / `WritableStream<Uint8Array>` byte pair.
- Feed that through the SDK's `ndJsonStream` to produce the bidirectional `Stream` of ACP JSON-RPC messages that `ClientSideConnection` expects.
- Handle lifecycle: the stream ends when the socket closes (cleanly or on error); surface errors.

### ndJsonStream wiring (verified against the actual SDK `.d.ts`/source, not docs)
- `ndJsonStream(output: WritableStream<Uint8Array>, input: ReadableStream<Uint8Array>): Stream` — note the argument order: the writable byte **sink** first, the readable byte **source** second.
- `Stream = { writable: WritableStream<AnyMessage>; readable: ReadableStream<AnyMessage> }` — the message-level stream `ClientSideConnection`'s constructor `(toClient, stream)` consumes.
- `acp-stream.ts` opens a browser `WebSocket`, then on `open` builds two byte streams and passes them through `ndJsonStream`:
  - **Outbound** `WritableStream<Uint8Array>`: each chunk (one newline-terminated JSON-RPC message produced by `ndJsonStream`) is decoded back to text and sent as a single `ws.send()` text frame. This matches `agent_ws.rs`'s contract of exactly one JSON-RPC message per WebSocket text frame; the trailing `\n` is inert because the Rust side parses each frame as one JSON value.
  - **Inbound** `ReadableStream<Uint8Array>`: each `message` event is converted to bytes **with a `"\n"` appended**. This is the framing adapter — the Rust side sends one message per frame with no delimiter, but `ndJsonStream`'s reader is a newline-delimited parser that only emits a message when it sees `"\n"`. Appending the newline makes each frame surface as a message immediately instead of being buffered until the socket closes.

### Lifecycle & error surfacing
- `connectAcpStream(url, options?)` returns `Promise<AcpConnection>` where `AcpConnection = { stream: Stream; close(): void }`. The promise resolves once the socket fires `open`.
- A WebSocket that fails to connect always fires `error` then `close`; `close`-before-`open` is the single deterministic "did not connect" signal, so the promise rejects there ("ACP WebSocket closed before it opened").
- The readable byte stream ends on a **clean** socket `close` (`controller.close()` → SDK connection sees end-of-input) and **errors** on an `error` event or an unclean `close` (`wasClean === false`), so a transport failure surfaces instead of looking like a silent end-of-stream.
- `AcpConnection.close()` closes the underlying `WebSocket`, which ends the readable. Teardown is **socket-driven**, not writable-driven: `ndJsonStream`'s own writable never forwards its `close` to the wrapped byte sink, so `outboundByteStream` deliberately has no `close`/`abort` handler and the explicit `close()` handle is the teardown path.
- Pure WebSocket — no Tauri `invoke` in the data path. The `AcpSocket` interface is the narrow subset of the browser `WebSocket` the module uses, declared so tests can inject an in-memory fake.

### Nits
- [x] `apps/kanban-app/ui/src/ai/acp-stream.ts:171-173` and `:252-257` — Event listeners are added (`message`/`close`/`error` in `inboundByteStream`, `open`/`close` in `connectAcpStream`) but `removeEventListener` is never called. The `AcpSocket` interface declares `removeEventListener` and `FakeWebSocket` implements it, yet nothing invokes it — the declared capability signals an intent the code does not fulfill. Minor — non-blocking.

### Nit resolution (2026-05-17)
Fixed. `inboundByteStream` now pairs every `addEventListener` with a `removeEventListener`: a `detach` closure removes the `message`/`close`/`error` listeners and is invoked on every finish path (clean close, unclean close, error) and from the `ReadableStream`'s new `cancel` handler. `connectAcpStream` adds a `detachConnectListeners` closure that removes the `open`/`close` connect listeners as soon as the promise settles via either `onOpen` or `onCloseBeforeOpen`. Note: `ndJsonStream` (`dist/stream.js`) wraps the inbound byte stream by holding a reader in a loop with no `cancel` propagation, so cancelling the message-level `stream.readable` does not reach the byte stream's `cancel` — the `cancel` handler remains correct for a direct byte-stream consumer but is not reachable through the SDK wrapping; listener removal is therefore verified through the finish paths instead.

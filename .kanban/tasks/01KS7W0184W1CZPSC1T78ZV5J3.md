#elicitation

## Context / Why
MCP/ACP elicitation requests carry a JSON-Schema-like `ElicitationSchema` describing form fields. The panel must render these "as documented in MCP" and coerce the user's input back into a typed `CreateElicitationResponse`. Keeping the schema→form logic as a pure, React-free module makes every field/type/validation rule exhaustively unit-testable (same split as `conversation.ts`'s pure reducer).

Read the property-schema types in `apps/kanban-app/ui/node_modules/@agentclientprotocol/sdk/dist/schema/types.gen.d.ts`:
- `ElicitationPropertySchema` variants by `type`: `string` (`StringPropertySchema` — may carry `enum`/`oneOf` for single-select, `format`, `minLength`/`maxLength`), `number`, `integer`, `boolean`, `array` (`MultiSelectPropertySchema` — multi-select).
- `ElicitationSchema`: `{ properties?: {[k]: ElicitationPropertySchema}, required?: string[], title?, description? }`.
- `ElicitationContentValue` = `string | number | boolean | string[]`.

## What
Create `apps/kanban-app/ui/src/ai/elicitation.ts`:
- [ ] `type ElicitationField` descriptor: `{ key, label, kind: "text" | "textarea" | "select" | "boolean" | "number" | "integer" | "multiselect", required, options?, description?, format? }` and a discriminator for url-mode vs form-mode requests.
- [ ] `parseElicitation(request: CreateElicitationRequest)`: returns either `{ mode: "form", message, fields: ElicitationField[] }` or `{ mode: "url", message, url, elicitationId }`. Map each property schema to a field kind (string+enum/oneOf → select; string (long/format) → textarea vs text; array → multiselect; boolean → boolean; number/integer → number/integer).
- [ ] `initialFormState(fields)`: default values per field.
- [ ] `validateForm(fields, values)`: returns missing-required / type errors.
- [ ] `toAcceptResponse(fields, values): CreateElicitationResponse` with `action:"accept"` and a `content` map of `ElicitationContentValue` (coerce numbers, booleans, string[]).
- [ ] `declineResponse()` / `cancelResponse()` helpers returning the respective `CreateElicitationResponse`.

## Acceptance Criteria
- [ ] Every `ElicitationPropertySchema` variant maps to a field kind.
- [ ] `toAcceptResponse` produces values matching the requested types (number not string, boolean not "true", array for multiselect).
- [ ] Required-field validation flags empty required fields; optional empties are omitted from `content`.
- [ ] URL-mode requests parse to the url branch without inventing form fields.

## Tests (`apps/kanban-app/ui/src/ai/elicitation.test.ts`)
- [ ] One test per field kind: schema → expected `ElicitationField`.
- [ ] `toAcceptResponse` coercion tests (number/integer/boolean/array/string).
- [ ] `validateForm` flags missing required, passes when satisfied.
- [ ] URL-mode parse test.
- [ ] The single-string `{answer: string}` schema used by the SAH `ask question` tool parses to one required text field and round-trips to `{answer: "..."}`.
- [ ] Run the vitest suite for this file — all green.

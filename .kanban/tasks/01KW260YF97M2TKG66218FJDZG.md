## What
`expect` needs a *driving* agent distinct from the *grading* model. Per `ideas/rule_agent.md` and `ideas/expect.md:93`, introduce a minimal use-case-based agent selection and slot in `Expectations`. Keep scope tight: only what `expect` needs, reusing the existing injected-factory pattern rather than rebuilding agent construction.

- Add `AgentUseCase` enum to `crates/swissarmyhammer-config` (variants `Root`, `Rules`, `Workflows`, `Expectations`; `#[serde(rename_all="lowercase")]`). This is the design's chosen placement (rule_agent.md §"Design Decisions").
- Extend `ToolContext` (`crates/swissarmyhammer-tools/src/mcp/tool_registry.rs:259`) with `use_case_agents: Arc<HashMap<AgentUseCase, ModelConfig>>` and a resolver `get_agent_for_use_case(&self, AgentUseCase) -> &ModelConfig` that falls back to the existing `agent_config` (root) when unset. Default the map to empty so existing tools are unaffected.
- Wire population from config at server init (`server.rs`) — read the `[agent]`/`agents` mapping; silent fallback to root when a use case is unconfigured (rule_agent.md decision #3).
- This task does NOT build the ACP driver (that's the agent-seam task); it only provides resolution so `expect` can later ask `ToolContext` for its `Expectations` agent.

## Acceptance Criteria
- [ ] `AgentUseCase` exists in `swissarmyhammer-config`, serializes to `"root"|"rules"|"workflows"|"expectations"`.
- [ ] `ToolContext::get_agent_for_use_case(AgentUseCase::Expectations)` returns the configured agent or falls back to root `agent_config`.
- [ ] Existing tools/tests unaffected (empty map ⇒ everything resolves to root).
- [ ] `cargo build` of config + tools crates succeeds.

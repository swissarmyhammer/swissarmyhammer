## What
Implement the single callback primitive the SDK uses wherever a plugin hands the host a function (command `available`/`execute`, view `render`, event handlers, elicitation responses). Functions can't cross the host/plugin boundary directly.

SDK side (`crates/swissarmyhammer-plugin/src/sdk/`) + host side (`crates/swissarmyhammer-plugin/src/`):
- When the SDK dispatches a call whose input contains function values, it assigns each a callback id, stores `{id → fn}` locally in the isolate, and replaces the function in the outgoing payload with `{ "$callback": "cb_xxxx" }`.
- The host treats `$callback` markers as opaque handles. To invoke one, the host sends `notifications/callbacks/invoke { id, args }` into the isolate.
- The SDK receives the notification, looks up the stored function, runs it, and — if a return value is expected — sends `notifications/callbacks/result`.
- This is the ONE primitive behind event subscriptions, command handlers, view renderers, async streaming, and elicitation.
- Explicitly NOT used for MCP tool calls themselves — those cross via the transports' real MCP JSON-RPC. The primitive is purely plugin→host function references.

## Acceptance Criteria
- [x] Functions in a dispatched payload are replaced with `{$callback: id}` markers; the `{id→fn}` table lives in the isolate.
- [x] `notifications/callbacks/invoke` runs the stored function; a return value flows back via `notifications/callbacks/result`.
- [x] Tool-call payloads are unaffected — no `$callback` machinery on the transport path.

### Warnings
- [x] `crates/swissarmyhammer-plugin/src/sdk/plugin.ts:182` — `marshalCallbacks` recurses unconditionally into plain objects and arrays with no visited-set or depth guard. A callback-bearing payload containing a cyclic plain object (e.g. `const a = {}; a.self = a; callbackDispatch({ a })`) recurses until the V8 stack overflows. This is a legitimately-shaped plugin input, and the function's own JSDoc claims it walks "to any depth" — implying safety it does not provide. The blast is contained to the offending isolate (it is plugin-trusted code), but it is still a crash where the doc implies a clean walk. Add a `WeakSet` of already-visited objects/arrays and skip (or error on) a node already seen, then update the JSDoc to state the cycle behavior.
- [x] `crates/swissarmyhammer-plugin/src/host.rs:873` — `collect_callback_ids` has the same unbounded recursion as `marshalCallbacks`: a host-side `callbackDispatch` payload with a JSON cycle is impossible to construct from `serde_json` parsing alone, but a deeply nested payload (legal JSON, no cycle) still recurses one stack frame per level and can overflow the host worker thread's stack. Lower priority than the SDK side since `serde_json::Value` cannot be cyclic, but a depth bound (or an explicit work-stack instead of recursion) would make the host walk robust against a hostile-depth payload from a compromised isolate.

## What

Board management is spread across AppState fields (`boards`, `active_board`) and AppConfig (`open_boards`). Four Tauri commands bypass dispatch_command: `open_board`, `close_board`, `set_active_board`, `switch_board`.

YAML commands already exist: `file.newBoard`, `file.openBoard`, `file.closeBoard`.

### Changes
- Add board management state to UIState: `open_boards`, `active_board_path`, per-window board assignments
- Add UIState methods: `open_board()`, `close_board()`, `set_active_board()`, `switch_board(window, path)`
- Wire `file.openBoard`, `file.closeBoard` command impls to UIState
- Add `file.switchBoard` command if missing
- Remove `open_board`, `close_board`, `set_active_board`, `switch_board` Tauri commands
- Frontend: all board management goes through `dispatch_command`
- Remove `open_boards` and related fields from AppConfig
- Remove `active_board` from AppState
- Note: BoardHandle storage may need to remain in a Tauri-level map since handles hold KanbanContext — UIState tracks paths, Tauri layer maps paths to handles

## Acceptance Criteria
- [ ] No direct `invoke(\"open_board\")`, `invoke(\"close_board\")`, `invoke(\"set_active_board\")`, `invoke(\"switch_board\")` in frontend
- [ ] Board management goes through `dispatch_command`
- [ ] Open boards persist across app restarts
- [ ] All four Tauri commands removed

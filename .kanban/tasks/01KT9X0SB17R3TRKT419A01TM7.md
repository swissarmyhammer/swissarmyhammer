Make board lifecycle observable on the bridge (the "window change" a kanban plugin actually cares about — each board ≈ a window).

Current state: board lifecycle emits direct Tauri only — `board-opened` (apps/kanban-app/src/menu.rs:838/840, payload {path}), `board-changed` (commands.rs:1571 switch, :1598 close, empty payload), plus `ui-state-changed` kinds board_switch/board_close. No bridge notifications.

## Work
- Add `notifications/board/opened|switched|closed` (NEW methods) carrying the board path, and publish on the bridge at the board open/switch/close sites.
- Swap the frontend `board-opened`/`board-changed` listeners to the bridge-forwarded methods; remove the direct emits once swapped.
- Declare on the window (or app) service tool. Decide owner: board lifecycle is window-service territory (board files ↔ windows). Use #[notification] struct=payload.
- Coverage guard.

## Acceptance
A plugin can `this.window.on("board.opened"/...)` (final names TBD); the frontend reads board lifecycle from the bridge; no direct `board-opened`/`board-changed` emit remains; declared == published.

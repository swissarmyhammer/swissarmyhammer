## What

The tag editor and task-dependency editor render as a multi-line CM6 editor. Pressing Enter in vim insert mode or pasting multi-line text expands the editor vertically. The user expects both to be single-line inputs — type, pick from autocomplete, commit.

Both fields use the same shared editor:

- `swissarmyhammer-kanban/builtin/definitions/tags.yaml` → `editor: multi-select`
- `swissarmyhammer-kanban/builtin/definitions/depends_on.yaml` → `editor: multi-select`
- `swissarmyhammer-kanban/builtin/definitions/assignees.yaml` → `editor: multi-select` (also benefits from this fix)

All three route to `kanban-app/ui/src/components/fields/editors/multi-select-editor.tsx` (`MultiSelectEditor`).

### Root cause

`MultiSelectEditor` calls `buildSubmitCancelExtensions({ ..., singleLine: true })` (see the `extensions` useMemo in `multi-select-editor.tsx`) but **does not** set `alwaysSubmitOnEnter: true`. Per `kanban-app/ui/src/lib/cm-submit-cancel.ts` (`buildVimEnterExtension`), when `singleLine=true` + `alwaysSubmitOnEnter=false`, vim **insert-mode** Enter falls through to CM6 and inserts a newline. Compare with `kanban-app/ui/src/components/command-palette.tsx` (~line 302) which uses `alwaysSubmitOnEnter: true` for exactly this reason.

There is also no transaction filter that strips `\n`/`\r` from inserted text, so pasting multi-line content (e.g. a list copied from elsewhere) inserts real newlines and grows the editor vertically.

### Approach

In `kanban-app/ui/src/components/fields/editors/multi-select-editor.tsx`:

1. **Block Enter newlines in vim insert mode** — change the `buildSubmitCancelExtensions` call inside the `extensions` useMemo to pass `alwaysSubmitOnEnter: true`. With `singleLine: true` already set, this makes Enter commit in every mode (cua, emacs, vim-normal, vim-insert). Autocomplete interception is preserved by the existing `completionStatus` guard inside `buildVimEnterExtension` and `buildCuaExtensions`.

2. **Strip newlines from all document changes** — add an `EditorState.transactionFilter.of(...)` extension that inspects each transaction's inserted text and rewrites any `\n`/`\r`/`\r\n` to a single space (or strips them). This defeats paste-of-multi-line-text, programmatic `dispatch({changes: ...})` with newlines, and any residual path that could otherwise insert a newline. Add it to the `extensions` useMemo alongside the existing ones. Import `EditorState` from `@codemirror/state`.

3. **Keep existing UX intact** — do not change the CM6 `basicSetup` (already has `lineNumbers: false`, `foldGutter: false` etc.), do not touch `shadcnTheme`, do not change the mention decorations or completion sources, and do not touch the `singleLine: true` flag elsewhere in the file.

### Why fix MultiSelectEditor, not the YAML

The YAML already declares intent (`editor: multi-select`). The bug is in the shared editor implementation. Fixing it once in `multi-select-editor.tsx` enforces single-line for **every** field that uses `multi-select` (tags, depends_on, assignees) instead of stamping the same config onto three YAML files.

### Why `transactionFilter` instead of CSS `white-space: nowrap`

CSS would visually clip wrapped content but the document would still contain newlines, which round-trip to storage and confuse `parseDocTokens` in `multi-select-editor.tsx` (it splits on `prefix`, not on newline — a pasted `#bug\n#feature` would not parse as two tokens). `transactionFilter` is the CM6-idiomatic way to keep the document structurally single-line. The existing `parseDocTokens` logic then Just Works.

### Scope boundaries

- Do **not** touch `TextEditor` (`kanban-app/ui/src/components/fields/text-editor.tsx`). It already accepts a `singleLine` prop — different callers, different policy, out of scope here even though it shares the same underlying submit/cancel helper.
- Do **not** modify `buildSubmitCancelExtensions` in `cm-submit-cancel.ts`. Its contract is correct; `MultiSelectEditor` is the misuser.
- Do **not** touch the YAML definitions. The `editor: multi-select` declaration is already correct.

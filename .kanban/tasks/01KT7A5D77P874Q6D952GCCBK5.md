avp-common reimplements agent path resolution + settings IO that mirdan already owns, and does it less safely.

## Problem (`crates/avp-common/src/install.rs`)
- `settings_path` (15) + inline JSON read/parse/write in `install` (159) / `uninstall` (220) **hardcode** `.claude/settings.json`, `.claude/settings.local.json`, `~/.claude/settings.json`.
- Uses strict `serde_json::from_str` — so an AVP install against a JSONC settings file (Zed/VS Code, trailing commas/comments) ERRORS where mirdan's JSONC-tolerant reader succeeds.
- `create_avp_project_structure` (202) creates `.avp/validators/` separately though mirdan has `validators_dir`.

## Change
- Resolve settings paths via `mirdan::agents::agent_project_settings_file` / `agent_global_settings_file` (+ the strategy's local-settings sibling) instead of hardcoded strings.
- Read/write via `mirdan::settings::read_json` (JSONC-tolerant) / `write_json` / `set_object` / `remove_key`.
- Use `mirdan::install::validators_dir` for the validators location.
- The AVP-specific `hooks.*` merge is a genuine AVP concern with no mirdan primitive — keep that logic, but layer it on mirdan's path + IO primitives, not hand-rolled ones.

## Done when
- No hardcoded `.claude/settings*.json` paths in avp-common; all via `mirdan::agents::*`.
- Settings read/write goes through `mirdan::settings::*` (JSONC-tolerant) — an AVP install against a JSONC settings file no longer errors.
- AVP hooks behavior unchanged; install/uninstall round-trips clean.
- `cargo build --workspace` green; clippy clean; avp tests pass.

Independent of the Profile cards (uses mirdan APIs that already exist); can run any time.
